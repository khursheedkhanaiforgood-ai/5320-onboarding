Step 1 — SW1 Post-Factory-Reset CLI

Run once manually over console or SSH immediately after factory reset. These commands are not idempotent — do NOT put enable ipforwarding or enable dhcp ports into an EP1 Supplemental CLI template.

Critical gotcha — two different DHCP commands

enable dhcp vlan <name> = DHCP client mode (SW1 asks for an IP). EXOS will throw an error if the VLAN already has a static IP — this is correct behaviour, not a bug.

enable dhcp ports <port> vlan <name> = DHCP server mode (SW1 serves IPs to clients on that port). This is what you need.

# ── One-time post-factory-reset commands ──────────────────────────────────

# Enable all ports and IP forwarding on Default VLAN
enable ports all
enable ipforwarding vlan Default

# Create VLANs
create vlan Employee tag 5
create vlan Guest tag 10

# SVI IP addresses — SW1 is the default gateway for VLAN 5 and 10
configure vlan Employee ipaddress 10.5.0.1 255.255.255.0
configure vlan Guest ipaddress 10.10.0.1 255.255.255.0

# Port 3 trunk toward AP1
# VLAN 1 is already native/untagged from factory reset — no change needed
configure vlan Employee add ports 3 tagged
configure vlan Guest add ports 3 tagged

# DHCP server — address pools
configure vlan Employee dhcp-address-range 10.5.0.10 - 10.5.0.254
configure vlan Employee dhcp-options default-gateway 10.5.0.1
configure vlan Employee dhcp-options dns-server primary 8.8.8.8
configure vlan Employee dhcp-options dns-server secondary 1.1.1.1

configure vlan Guest dhcp-address-range 10.10.0.10 - 10.10.0.254
configure vlan Guest dhcp-options default-gateway 10.10.0.1
configure vlan Guest dhcp-options dns-server primary 8.8.8.8
configure vlan Guest dhcp-options dns-server secondary 1.1.1.1

# Enable IP forwarding on both VLANs
enable ipforwarding vlan Employee
enable ipforwarding vlan Guest

# Enable DHCP server on port 3 for each VLAN
enable dhcp ports 3 vlan Employee
enable dhcp ports 3 vlan Guest

save config

Step 3 — AP1 VIQ Migration (Karl's Lab → KhKLab)

Critical order — do not factory reset before releasing from old VIQ

An AP bound to a VIQ reconnects to that VIQ after every factory reset — the cloud owns it by serial number, not local config. Factory resetting first and hoping it appears in your VIQ will not work.

Correct sequence

  1. In Karl's VIQ (Khursheed had access): EP1 → Manage → Devices → AP1 → Delete/Remove from organization
  2. In your VIQ: EP1 → Manage → Devices → Onboard → enter serial number HA012519Y-10623
  3. Factory reset AP1 — from EP1 (Manage → Devices → AP1 → Actions → Reset to Factory Default) or physical button
  4. AP1 boots → CAPWAP → connects to your VIQ → appears as Connected (green)

Lessons Learned

1. DHCP server vs client — two different EXOS commands

enable dhcp vlan <name> enables DHCP client mode on the VLAN interface. EXOS throws an error if the VLAN already has a static IP — this is correct. The DHCP server requires a separate command:

enable dhcp ports <port> vlan <name>   ← server mode, enables on specific port
# Confirmed by: "Ports DHCP Enabled: No ports enabled" in show dhcp-server
# Fix: enable dhcp ports 3 vlan Employee / enable dhcp ports 3 vlan Guest

2. VIQ migration order is non-negotiable

Delete AP from old VIQ before factory reset. The cloud owns the AP by serial number. A factory reset without the cloud-side release just reconnects the AP to the old VIQ. Sequence: delete from old org → onboard SN in new org → reset AP.

3. VLAN assignment in EP1 = User Profile, not SSID name

The SSID name and VLAN name can be anything — what controls which VLAN tag AP1 applies to client frames is the User Profile assigned to the SSID. Always verify: SSID → Default User Profile → VLAN ID.

4. AP3000 SSH password after EP1 onboarding

Default SSH credentials not found during this session. Password is set via Device Admin Profile in EP1. Location: Configure → Common Objects → Management → Device Admin Profiles. Configure this in the network policy before needing SSH access.

5. IQAgent heartbeat is normal

proxy device-connector unknown POST /health-check/[serial] every ~60 s = normal IQAgent keepalive. Not an error or connection problem.
