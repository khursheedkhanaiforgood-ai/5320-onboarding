Phase 0 — EXOS SW1 Pre-flight
Pre-flight SW1 EXOS — VLANs 5 & 10, Port 3, DHCP, ipforwarding

Run on SW1 console before first AP1 test. Port 3 = uplink to AP1. Do NOT put enable ipforwarding in Supplemental CLI (XIQ hangs at 15%).

create vlan Employee tag 5
create vlan Guest tag 10
configure vlan Employee add ports 3 tagged
configure vlan Guest add ports 3 tagged

enable ipforwarding vlan Employee
enable ipforwarding vlan Guest

create dhcp-address-range Employee-Pool vr VR-Default 10.5.0.100-10.5.0.200
configure dhcp-address-range Employee-Pool vlan Employee default-gateway 10.5.0.1 dns-server 8.8.8.8
configure vlan Employee ipaddress 10.5.0.1 255.255.255.0
enable dhcp ports 3 vlan Employee

create dhcp-address-range Guest-Pool vr VR-Default 10.10.0.100-10.10.0.200
configure dhcp-address-range Guest-Pool vlan Guest default-gateway 10.10.0.1 dns-server 8.8.8.8
configure vlan Guest ipaddress 10.10.0.1 255.255.255.0
enable dhcp ports 3 vlan Guest

show vlan Employee detail
show dhcp-server vlan Employee

Verify

show fdb ports 3 should show AP1 MAC in VLAN 1 (native). After client connects to Employee SSID, AP tags frames with VLAN 5 — client MAC should appear in VLAN 5 FDB within 30s.

Phase 0b — SW1 Onboarding into EP1 (Switch Engine)
Pre-flight B EP1 Switch Onboarding — Template + VLAN + Supplemental CLI

Source: Universal Switch Deployment Guide v25.7.0, pp.14–60. Perform AFTER Phase 0 CLI pre-flight and BEFORE deploying network policy to AP1.

Step 1 — Onboard SW1 (Monitoring > Network Devices)

  1. Go to Monitoring > Network Devices
  2. Click Onboard Devices (or + icon)
  3. Select Manual Entry (for ≤10 devices; use CSV bulk for more)
  4. Enter SW1 serial number (format: 12-char alphanumeric on chassis label)
  5. Click ADD DEVICE → DONE
  6. Wait up to 10 min — status shows No Connection icon initially; resolves to Connected after IQAgent establishes cloud tunnel

IQAgent Prerequisite

SW1 must have IQAgent running and cloud connectivity (TCP 443 + HTTPS). IQAgent is pre-installed on Switch Engine 31.6+. Verify: show idm on SW1 CLI → RUN state = Connected, RadSec cert = Valid, port 2083 open.

Step 2 — Network Policy (Switching type)

  1. Go to Configuration > Network (left nav)
  2. Open or create a policy: tab 1 Policy Details
  3. Check both Wireless (for AP1) and Switching/Routing (for SW1); or create a separate switch-only policy
  4. For Management Servers — Switch Engine/EXOS only: set routing instance to VR-Default or VR-mgmt so DNS, NTP, SNMP, and Syslog route correctly
  5. Click SAVE → NEXT

Step 3 — Create Switch Template

  1. From the policy, click 3 Switching/Routing tab → Switch Settings → Switch Template
  2. Select the default template for your model: e.g., Switch Engine 5320-24T-8XE
  3. Click Copy and give it a name (e.g., KhKLab-SW1). Never edit the default template directly.
  4. Open the new template and toggle Enable Override Policy Common Settings to ON
  5. Under Management Interface Settings (Switch Engine only): select VLAN Interface → Management VLAN = 1 → Management IP Settings = DHCP

Step 4 — VLAN Attributes

  1. In the switch template: Switch Settings → VLAN Attributes → +
  2. For each VLAN, select Manual and set VLAN ID + Name:
VLAN NameVLAN IDNotes
Mgmt1Native / management (AP1 CAPWAP, SW1 IQAgent)
Employee5Staff PPSK VLAN
Guest10Guest CWP VLAN (or Student VLAN)
GuestWireless15Walk-in guest VLAN (if separate from VLAN 10)

IGMP Snooping override order (highest priority first): Device-specific VLAN → VLAN Attributes → Device Template → Common Settings.

Step 5 — Port Types (Trunk + Access)

  1. In the policy: Switch Settings → Port Types → +
  2. Create a Trunk Port for the AP uplink (Port 3):
    • Port Usage: Trunk Port (802.1Q VLAN Tagging)
    • Native VLAN: Mgmt (VLAN 1)
    • Allowed VLANs: all (Switch Engine keyword — auto-tags all VLANs assigned in any port type in this policy)
    • Enable LLDP Transmit + LLDP Receive
    • STP: Edge Port OFF (it’s a switch-to-AP uplink, not an edge/user port)
  3. Create an Access Port for workstation ports:
    • Port Usage: Access Port
    • VLAN: assign Employee (VLAN 5) or Guest (VLAN 10) as needed
    • STP: Edge Port ON + BPDU Guard = Guard
  4. In the switch template Port/VLAN Configuration view, assign the Trunk port type to Port 3 and the Access port type to remaining ports

Step 6 — Supplemental CLI (EXOS features not in EP1)

Enable First

Go to Administration & Settings > Backup & Restore > VIQ Management → toggle Supplemental CLI to ON. This is a one-time org-level setting.

  1. In the switch template: Configuration > Advanced Settings > Supplemental CLI > +
  2. Add EXOS commands that EP1 does NOT manage (example: DHCP server, ipforwarding, ACL): 
    configure vlan Employee ipaddress 10.5.0.1 255.255.255.0
enable ipforwarding vlan Employee
create dhcp-address-range Employee-Pool vr VR-Default 10.5.0.100-10.5.0.200
configure dhcp-address-range Employee-Pool vlan Employee default-gateway 10.5.0.1 dns-server 8.8.8.8
  3. Max 8192 characters; one command per line
  4. Use ${vlan:Employee} to reference a VLAN object by name (resolves to VLAN ID); use ${ip:server_ip} for IP objects
  5. After adding Supplemental CLI: perform Complete Configuration Update (not Delta) to SW1

Hard Constraint — No Overlap Rule

Never put commands in Supplemental CLI that EP1 already generates from the Network Policy (VLANs, STP, IGMP, port types). Overlapping CLI = Device Update Failure. Also: never put enable dhcp or enable ipforwarding in Supplemental CLI — these hang EP1/XIQ at 15%. Run them manually via console after first deploy.

Step 7 — Deploy to SW1

  1. Go to Configuration > Network Policies → select your policy → tab 4 Deploy Policy
  2. Check the box next to SW1 in the device table
  3. Click UPLOAD → select Complete Configuration Update (first time or after Supplemental CLI change) or Delta (subsequent changes)
  4. Click PERFORM UPDATE
  5. Monitor: Monitoring > Network Devices > SW1 → Config Audit icon should change to Match

Verify

On SW1 CLI: show vlan detail — VLANs 5/10/15 should appear with correct port assignments. show fdb ports 3 — AP1 MAC in VLAN 1. After policy push Config Audit icon = Match (green checkmark).

Phase 1 — Create Network Policy (p.235)
Phase 1 Add Network Policy — Configuration > Network
  1. Go to Configuration > Network (left nav pane in EP1 Networking portal)
  2. Select Network Policies from the sub-menu
  3. Click + (Add Network Policy button in list view, or select thumbnail Add Network Policy)
  4. In New Policy window, select policy type: check Wireless (APs). Optionally also check Switching/Routing if managing SW1 via EP1.
  5. Type a Policy Name: e.g., KhKLab-Policy
  6. (Optional) Type a Description
  7. Leave Presence Analytics disabled (lab — not needed)
  8. Click SAVE
  9. Click NEXT → tab advances to 2 Wireless

AP Prerequisite

EP1 assumes APs have already established CAPWAP connections (UDP 12222 or HTTPS TCP 80/443). AP1 must be in your VIQ org (not Karl’s) before the policy push will land on it. Move AP1: unclaim from Karl’s org → claim in your org → wait for CAPWAP reconnect.

Phase 2 — Policy Settings (pp.236–253)
Phase 2 Policy Settings — Time Zone, SNMP, Hive (Roaming Cache)

Accessed from the left sidebar inside the network policy workflow (tab 1 still active or via sidebar). Each setting has its own expandable sub-panel.

  1. In the policy left sidebar, click Policy Settings
  2. Device Time Zone (p.245): toggle ON → select America/New_York (EST, UTC−5)
  3. SNMP Server (p.240): toggle ON → + Add server → IP or hostname → Version: v2c → Operation: Get and Trap → Community: hivecommunity → ADD SNMP SERVER → SAVE SNMP SERVER
  4. NTP Server (p.238): leave at default DHCP-provided (device auto-discovers 0.aerohive.pool.ntp.org if DHCP does not advertise NTP). Optionally add explicit NTP IP.
  5. Hive (p.246): expand Client Roaming > Share connected client information (Roaming cache):
    • Devices send client information every: default 60 seconds ✓ (leave as-is — lower = faster roaming)
    • Remove cached client information when absent after: default 5 missed updates (5 × 60s = 5 min)
  6. Management and Native VLAN (p.249): set MGT Interface VLAN if managing AP via a specific VLAN (optional for lab)
  7. Click SAVE → Policy → NEXT to advance to tab 2 Wireless
Phase 3 — Employee SSID (Private Pre-Shared Key / VLAN5)
Phase 3 Employee SSID — PPSK Authentication + VLAN 5

On tab 2 Wireless of the network policy. Guide ref: pp.268–279, 292.

  1. Select 2 Wireless tab, then click + to add a new SSID (or select existing → edit icon)
  2. Name: Employee (internal EP1 name)
  3. Broadcast Name: Employee (what clients see in beacon; auto-derived from Name)
  4. Radio bands: check both Wi-Fi 0 Radio (2.4 GHz or 5 GHz) and Wi-Fi 1 Radio (5 GHz only) for dual-band
  5. Under SSID Usage > SSID Authentication, select Private Pre-Shared Key (p.292)
  6. User Group: click + to add new group
    • Name: Staff
    • Password DB Location: Cloud (for cloud-resident PPSK database)
    • Password Type: PPSK
    • PSK Generation Method: Password Only
    • Password Length: 12 characters
    • Generate Password Using: Letters + Numbers
    • Account Expiration: Never Expire
    • Click SAVE USER GROUP
  7. Default User Profile: click + to add new profile
    • User Profile Name: Employee-Profile
    • Connect to: VLAN → + New VLAN object: Name Employee, VLAN ID 5 → SAVE VLAN
    • Click SAVE USER PROFILE
  8. Toggle Client Monitor ON (default — leave enabled)
  9. Click SAVE to save the Employee SSID

PPSK Users

Individual PPSK passphrases are added separately: Configuration > User Management (p.525) → Add User → assign to the Staff user group → system generates passphrase. Each user gets a unique passphrase for the same SSID.

Phase 4 — Guest SSID (Open / CWP / VLAN10)
Phase 4 Guest SSID — Open Auth + Captive Web Portal + VLAN 10

Guide ref: pp.268–271, 295 (Device-based CWP), 302 (Cloud-based CWP). For lab: use Device-based CWP (UPA style) unless GuestEssentials is licensed.

  1. On tab 2 Wireless, click + again to add a second SSID
  2. Name: Guest
  3. Broadcast Name: Guest_Wireless
  4. Radio bands: check both Wi-Fi 0 and Wi-Fi 1 (2.4 + 5 GHz). Note: Open auth is not available for 6 GHz (Wi-Fi 2).
  5. Under SSID Authentication, select Open
  6. For Captive Web Portal (step 12 of SSID workflow, p.271):
    • Select Customize and Preview Device-based Captive Web Portal Settings (p.295)
    • CWP Type: User Acceptance Policy (UPA) — simplest T&C acceptance
    • Configure the T&C text and optional redirect URL
    • Click SAVE CWP
  7. Default User Profile: click + to add new profile
    • User Profile Name: Guest-Profile
    • Connect to: VLAN → + New VLAN object: Name Guest, VLAN ID 10 → SAVE VLAN
    • Click SAVE USER PROFILE
  8. Under Optional Settings > Traffic Filters (Additional Settings), add RFC 1918 firewall rules:
    • DENY 10.0.0.0/8 → ANY (block access to corporate LAN)
    • DENY 172.16.0.0/12 → ANY
    • DENY 192.168.0.0/16 → ANY
    • PERMIT ANY → ANY (allow internet)
  9. Toggle Client Monitor ON
  10. Click SAVE to save the Guest SSID

GuestEssentials vs. Built-in CWP

Device-based CWP (configured here) = simple T&C page, no identity capture, no audit trail. GuestEssentials (separate EP1 app) = full branded portal, email/SMS self-reg, sponsorship workflow, HIPAA-grade audit trail. For this lab: Device-based CWP is sufficient.

Phase 5 — Deploy to AP1 (p.267)
Phase 5 Deploy Policy — Tab 4 → UPLOAD → PERFORM UPDATE
  1. Navigate to tab 4 Deploy Policy (or Configuration > Network Policies → select policy → Deploy icon)
  2. The device table shows all devices assigned to this policy. Use Assigned / Eligible / Filtered controls to find AP1.
  3. Check the box next to AP1
  4. Click UPLOAD
  5. In Device Update window:
    • Update type: Delta (only changed configs) for incremental updates; Complete for full repush
    • (Optional) Enable Distributed Image Upgrade if WAN is slow
  6. Click PERFORM UPDATE
  7. Monitor: Monitoring > Network Devices > AP1 → status changes to Connected + Updated
  8. The AP reboots to activate new config. Expect 60–90 sec downtime.

Verify SSIDs

After AP reboots: iPhone WiFi scan should show both Employee and Guest_Wireless. Connect iPhone to Employee with a PPSK passphrase → check show fdb ports 3 on SW1 → iPhone MAC should appear in VLAN 5. Connect to Guest_Wireless → browser should redirect to CWP T&C page.

Transition Timeline Correction — Source: CustomerFAQ May 6, 2026

Earlier notes said “July 1 classic UI retirement.” This is wrong. Official timeline:

  • July 2026: New UI becomes the default on login — but admins can still toggle back to classic via the 9-dot menu
  • Q4 CY2026: Classic UI permanently retired — new UI is sole interface for all customers
  • EP1 customers: classic remains accessible via 9-dot until the Q4 final cutover

Also critical (Feature Comparison, April 9, 2026): Guest Essentials, AirDefense Essentials, Location Essentials, CoPilot, and Packet Capture are all “Planned” — NOT yet in either New UI or EP1 as of April 2026. Targeted for July 2026 delivery. These still run as separate Essentials apps accessed via the 9-dot launcher.

Retired in new UI (will NOT migrate): WiNG Controller (Connected Mode), A3, Aerohive SR Switches, Dell Switches, XR Routing Support (AP150W/AP302W router mode), Legacy Entitlement Keys (LEK).
