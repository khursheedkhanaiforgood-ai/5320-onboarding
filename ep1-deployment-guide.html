Stack Hard Rule — Same Level + Sufficient Count for All Slots

All subscriptions in a stack must be the same level (all Standard OR all Pilot — cannot mix levels; can mix tiers within a level, e.g. Tier A + Tier B in one Standard stack). If any slot lacks a sufficient subscription, the entire stack is Unmanaged — not just that slot. Verify: Subscriptions & Services > Subscriptions & Licensing.

Entitlement Violations — What Happens When Subs Run Out

EP1/XIQ auto-unmanages most recently onboarded devices first until compliant. Unmanaged APs stop CAPWAP telemetry, traps, and AVC reports. AP LED alternates amber/white (searching for new CAPWAP server). Banner: “LICENSE VIOLATION – Unlicensed devices. Check affected devices here.”

Resolve: purchase/renew or unmanage overflow devices. Sync stale pool: Global Settings > Administration > License Management > Synchronize (Classic) or Extreme Portal > Assets > Cloud Licenses Home. CUID must match between Portal and XIQ.

CoPilot note: CoPilot violations are independent of Pilot violations — both can be active simultaneously. Standard CoPilot features are not affected unless a Pilot violation is also active. Disable CoPilot per VIQ: Global Settings > Administration > VIQ Management > Disable CoPilot feature for this VIQ.

Unlock Full Features — Pilot → EP1 Standard (NON-REVERSIBLE)

Mixed-subscription deployments show a blue icon on limited-feature buildings/sites. To unlock EP1 full features: Subscriptions & Services > Inventory > Unlock Full Features (or Monitoring > Visualize > Quick Location Nav > 3-dot > Unlock Full Features). Non-reversible. If Fabric Engine devices present, must select ALL buildings with FE.

XIQ Pilot-only accounts are missing: AI Expert, Agent Workflows, AI Expert Actions, Contracts page, Inventory page, Custom Roles, Fabric Configuration, Visualize layer views. All require EP1 Standard Networking.

Legacy Entitlement Keys (LEK) — Blocking Issue

LEK are no longer available for purchase and are disabled in the new LEM schema. Critical: if your account has LEK present AND Extreme Portal is linked, you cannot add new devices or change Unmanaged to Managed. Contact GTAC to convert LEK to XIQ Pilot subscriptions. Perpetual LEK are incompatible with XIQ entirely.

AI Credits, CUID & Expiration Alerts

Some AI Expert agents activate AI Credits. AI Credits are automatically revoked monthly (not permanently consumed). Monitor: Subscriptions & Services > Subscriptions & Licensing > Credits tab.

CUID (Customer Unique Identifier) = your license pool ID. Always quote CUID to GTAC. Visible on Subscriptions & Licensing page header. Sync max once every 5 minutes.

Proactive expiry alerts (XIQ Classic): Banner + email at 90, 60, 30, 7, 6, 5, 4, 3, 2, 1 days before expiry. Daily from 7-day threshold. Opt-in: Global Settings > Account Details > “Proactive license warning email messages” toggle.

Phase 0b — SW1 Onboarding into EP1 (Switch Engine)
Pre-flight B EP1 Switch Onboarding — Template + VLAN + Supplemental CLI

Source: Universal Switch Deployment Guide v25.7.0, pp.14–60. Perform AFTER Phase 0 CLI pre-flight and BEFORE deploying network policy to AP1.

Step 1 — Onboard SW1 (Monitoring > Network Devices)

  1. Go to Monitoring > Network Devices
  2. Click Onboard Devices (or + icon)
  3. Select Manual Entry (for ≤10 devices; use CSV bulk for more)
  4. Enter SW1 serial number (format: 12-char alphanumeric on chassis label)
  5. Click ADD DEVICE → DONE
  6. Wait up to 10 min — status shows No Connection icon initially; resolves to Connected after IQAgent establishes cloud tunnel

IQAgent + Firmware Prerequisites

SW1 minimum firmware: Switch Engine 33.5 (EP1 v25.9.0-199 requirement for 5320 series). If running lower, upgrade EXOS first via download image-management or drag-and-drop in Monitoring > Network Devices. IQAgent is pre-installed on Switch Engine 31.6+; upgrade IQAgent if EXOS <33.2 (see release notes p.10 for IQAgent version matrix).

AP1 minimum firmware: IQ Engine 10.8.2 (latest: 10.8.6 / AP 10.18.1). Verify: Monitoring > Network Devices > AP1 > Firmware column.

Firewall: allow SW1 and AP1 outbound to EP1 cloud. US East (VA/VA2): 34.202.197.0/26, 44.192.245.0/26, 3.234.248.0/27. Ports: TCP 443 (HTTPS/IQAgent), TCP/UDP 2083 (RadSec). Verify IQAgent: show idm → RUN state = Connected, RadSec cert = Valid.

Step 2 — Network Policy (Switching type)

  1. Go to Configuration > Network (left nav)
  2. Open or create a policy: tab 1 Policy Details
  3. Check both Wireless (for AP1) and Switching/Routing (for SW1); or create a separate switch-only policy
  4. For Management Servers — Switch Engine/EXOS only: set routing instance to VR-Default or VR-mgmt so DNS, NTP, SNMP, and Syslog route correctly
  5. Click SAVE → NEXT

Step 3 — Create Switch Template

  1. From the policy, click 3 Switching/Routing tab → Switch Settings → Switch Template
  2. Select the default template for your model: e.g., Switch Engine 5320-24T-8XE
  3. Click Copy and give it a name (e.g., KhKLab-SW1). Never edit the default template directly.
  4. Open the new template and toggle Enable Override Policy Common Settings to ON
  5. Under Management Interface Settings (Switch Engine only): select VLAN Interface → Management VLAN = 1 → Management IP Settings = DHCP

Step 4 — VLAN Attributes

  1. In the switch template: Switch Settings → VLAN Attributes → +
  2. For each VLAN, select Manual and set VLAN ID + Name:
VLAN NameVLAN IDNotes
Mgmt1Native / management (AP1 CAPWAP, SW1 IQAgent)
Employee5Staff PPSK VLAN
Guest10Guest CWP VLAN (or Student VLAN)
GuestWireless15Walk-in guest VLAN (if separate from VLAN 10)

IGMP Snooping override order (highest priority first): Device-specific VLAN → VLAN Attributes → Device Template → Common Settings.

Step 5 — Port Types (Trunk + Access)

  1. In the policy: Switch Settings → Port Types → +
  2. Create a Trunk Port for the AP uplink (Port 3):
    • Port Usage: Trunk Port (802.1Q VLAN Tagging)
    • Native VLAN: Mgmt (VLAN 1)
    • Allowed VLANs: all (Switch Engine keyword — auto-tags all VLANs assigned in any port type in this policy)
    • Enable LLDP Transmit + LLDP Receive
    • STP: Edge Port OFF (it’s a switch-to-AP uplink, not an edge/user port)
  3. Create an Access Port for workstation ports:
    • Port Usage: Access Port
    • VLAN: assign Employee (VLAN 5) or Guest (VLAN 10) as needed
    • STP: Edge Port ON + BPDU Guard = Guard
  4. In the switch template Port/VLAN Configuration view, assign the Trunk port type to Port 3 and the Access port type to remaining ports

Step 6 — Supplemental CLI (EXOS features not in EP1)

Enable First

Go to Administration & Settings > Backup & Restore > VIQ Management → toggle Supplemental CLI to ON. This is a one-time org-level setting.

  1. In the switch template: Configuration > Advanced Settings > Supplemental CLI > +
  2. Add EXOS commands that EP1 does NOT manage (example: DHCP server, ipforwarding, ACL): 
    configure vlan Employee ipaddress 10.5.0.1 255.255.255.0
enable ipforwarding vlan Employee
create dhcp-address-range Employee-Pool vr VR-Default 10.5.0.100-10.5.0.200
configure dhcp-address-range Employee-Pool vlan Employee default-gateway 10.5.0.1 dns-server 8.8.8.8
  3. Max 8192 characters; one command per line
  4. Use ${vlan:Employee} to reference a VLAN object by name (resolves to VLAN ID); use ${ip:server_ip} for IP objects
  5. After adding Supplemental CLI: perform Complete Configuration Update (not Delta) to SW1

Hard Constraint — No Overlap Rule

Never put commands in Supplemental CLI that EP1 already generates from the Network Policy (VLANs, STP, IGMP, port types). Overlapping CLI = Device Update Failure. Also: never put enable dhcp or enable ipforwarding in Supplemental CLI — these hang EP1/XIQ at 15%. Run them manually via console after first deploy.

Step 7 — Deploy to SW1

  1. Go to Configuration > Network Policies → select your policy → tab 4 Deploy Policy
  2. Check the box next to SW1 in the device table
  3. Click UPLOAD → select Complete Configuration Update (first time or after Supplemental CLI change) or Delta (subsequent changes)
  4. Click PERFORM UPDATE
  5. Monitor: Monitoring > Network Devices > SW1 → Config Audit icon should change to Match

Verify

On SW1 CLI: show vlan detail — VLANs 5/10/15 should appear with correct port assignments. show fdb ports 3 — AP1 MAC in VLAN 1. After policy push Config Audit icon = Match (green checkmark).

Known Issue XIQ-46735 — Do Not Toggle Auto Config

In Administration & Settings > Backup & Restore > VIQ Management, do not toggle the Auto Config setting after devices reach audit-match state. This is a known bug in v25.9.0-199: toggling flips audit-match (green) devices to audit-mismatch (yellow) incorrectly. Source: Release Notes p.16 + p.18.

Phase 2 — Policy Settings (pp.236–253)
Phase 2 Policy Settings — Time Zone, SNMP, Hive (Roaming Cache)

Accessed from the left sidebar inside the network policy workflow (tab 1 still active or via sidebar). Each setting has its own expandable sub-panel.

  2. Device Time Zone (p.245): toggle ON → select America/New_York (EST, UTC−5)
  3. SNMP Server (p.240): toggle ON → + Add server → IP or hostname → Version: v2c → Operation: Get and Trap → Community: hivecommunity → ADD SNMP SERVER → SAVE SNMP SERVER

  6. Management and Native VLAN (p.249): set MGT Interface VLAN if managing AP via a specific VLAN (optional for lab)

Phase 3 — Employee SSID (Private Pre-Shared Key / VLAN5)
Phase 3 Employee SSID — PPSK Authentication + VLAN 5

On tab 2 Wireless of the network policy. Guide ref: pp.268–279, 292.

  1. Select 2 Wireless tab, then click + to add a new SSID (or select existing → edit icon)
  2. Name: Employee (internal EP1 name)
  3. Broadcast Name: Employee (what clients see in beacon; auto-derived from Name)
  4. Radio bands: check both Wi-Fi 0 Radio (2.4 GHz or 5 GHz) and Wi-Fi 1 Radio (5 GHz only) for dual-band
  5. Under SSID Usage > SSID Authentication, select Private Pre-Shared Key (p.292)
  6. User Group: click + to add new group
    • Name: Staff
    • Password DB Location: Cloud (for cloud-resident PPSK database)
    • Password Type: PPSK
    • PSK Generation Method: Password Only
    • Password Length: 12 characters
    • Generate Password Using: Letters + Numbers
    • Account Expiration: Never Expire
    • Click SAVE USER GROUP
  7. Default User Profile: click + to add new profile
    • User Profile Name: Employee-Profile
    • Connect to: VLAN → + New VLAN object: Name Employee, VLAN ID 5 → SAVE VLAN
    • Click SAVE USER PROFILE
  8. Toggle Client Monitor ON (default — leave enabled)
  9. Click SAVE to save the Employee SSID

PPSK Users

Individual PPSK passphrases are added separately: Configuration > User Management (p.525) → Add User → assign to the Staff user group → system generates passphrase. Each user gets a unique passphrase for the same SSID.

Phase 4 — Guest SSID (Open / CWP / VLAN10)
Phase 4 Guest SSID — Open Auth + Captive Web Portal + VLAN 10

Guide ref: pp.268–271, 295 (Device-based CWP), 302 (Cloud-based CWP). For lab: use Device-based CWP (UPA style) unless GuestEssentials is licensed.

  1. On tab 2 Wireless, click + again to add a second SSID
  2. Name: Guest
  3. Broadcast Name: Guest_Wireless
  4. Radio bands: check both Wi-Fi 0 and Wi-Fi 1 (2.4 + 5 GHz). Note: Open auth is not available for 6 GHz (Wi-Fi 2).
  5. Under SSID Authentication, select Open
  6. For Captive Web Portal (step 12 of SSID workflow, p.271):
    • Select Customize and Preview Device-based Captive Web Portal Settings (p.295)
    • CWP Type: User Acceptance Policy (UPA) — simplest T&C acceptance
    • Configure the T&C text and optional redirect URL
    • Click SAVE CWP
  7. Default User Profile: click + to add new profile
    • User Profile Name: Guest-Profile
    • Connect to: VLAN → + New VLAN object: Name Guest, VLAN ID 10 → SAVE VLAN
    • Click SAVE USER PROFILE
  8. Under Optional Settings > Traffic Filters (Additional Settings), add RFC 1918 firewall rules:
    • DENY 10.0.0.0/8 → ANY (block access to corporate LAN)
    • DENY 172.16.0.0/12 → ANY
    • DENY 192.168.0.0/16 → ANY
    • PERMIT ANY → ANY (allow internet)
  9. Toggle Client Monitor ON
  10. Click SAVE to save the Guest SSID

GuestEssentials vs. Built-in CWP

Device-based CWP (configured here) = simple T&C page, no identity capture, no audit trail. GuestEssentials (separate EP1 app) = full branded portal, email/SMS self-reg, sponsorship workflow, HIPAA-grade audit trail. For this lab: Device-based CWP is sufficient.

Use Case 1: Deploy & Create Policies Faster (p.38 — Talk to Knowledge)

AI Workflow Knowledge Mode — Guide-Backed Policy Creation Assistance

Talk to Knowledge searches 29,000+ GTAC KB articles + all product documentation + Extreme Networks website content. Use this when you need verified step-by-step procedures without searching PDFs manually.

  1. Click AI Expert icon → click + (New Conversation)
  2. Select topic: Knowledge → choose or type a query
  3. Example: “How do I create a PPSK SSID with a Cloud User Group and VLAN 5 on a 5320 switch in EP1?”
  4. AI Expert returns the exact procedure from the User Guide, referencing the relevant page
  5. Follow up in the same thread: “What port type should I use on the switch template for the AP uplink?”
  6. Click Add to Canvas to pin the answer as a widget for future reference

Best Practice (p.41)

Always name the product model: “5320-24T-8XE” not just “switch.” For large networks (70,000+ users), include time scope: “clients connected in last 24 hours” instead of open-ended queries — significantly reduces response time.

Use Case 2: Monitor Network in Real Time (p.38 — Talk to Data)

AI Workflow Talk to Data — Live Network Queries via Natural Language

Talk to Data queries live EP1 APIs — same data as the dashboard, delivered as tables / bar charts / time-series graphs in response to natural language. Combines telemetry + config data. Shows reasoning transparency (data sources + API calls used).

  1. Click + → select topic: Devices, Clients, or Alerts
  2. Select an enablement prompt (structured query suggestions provided automatically at conversation start)
  3. Example queries:
    • “Show all APs with channel utilization above 80% in the last 24 hours.”
    • “List all clients that failed to authenticate in the last hour.”
    • “How many unacknowledged critical alerts do I have and what are the top device types?”
  4. AI Expert retrieves, correlates, and visualizes: output is tables, charts, or graphs
  5. Follow up in same thread: “Of those 26 critical alerts, which 3 devices had the most?” → generates a breakdown chart
  6. Click Add to Canvas → the query result becomes a live widget on your AI Canvas dashboard (auto-refreshes hourly when published)

Standard Monitoring (Non-AI Alternative)

Monitoring > Dashboard: Alerts (3-day severity breakdown), Usage & Capacity, Device Status (offline count), Client Health (issues count), Device Health. Filter by Sites.
Monitoring > Visualize: Geographic (default), Physical Layer (all managed + LLDP-discovered), Fabric Layer (IS-IS/SPB), Service Layer (VLANs/I-SIDs), Heat Map, Access view. Split view available for Physical+Service, Fabric+Service etc. Inspector Panel for per-element details.

Use Case 3: Troubleshoot Errors with AI (pp.39–40 — Service Agent Beta)

AI Workflow Service Agent (Beta) — AI-Driven Troubleshooting + Automated GTAC Cases

Service Agent is an intelligent support assistant that operates continuously inside EP1 to diagnose issues, gather evidence, and create GTAC cases. Uses Human-in-the-Loop (HITL) — you explicitly approve every action before it executes. Requires ExtremeWorks support package + Beta Program enabled.

  1. Enable: AI Agents Hub (Beta icon) → accept Terms & Conditions → Enable Beta Program
  2. Click + → select topic: Service Agent
  3. Describe the symptom: “AP3 in Building B shows clients connected but no traffic passes.”
  4. Service Agent asks clarifying questions: affected serial? Which site? When did this start?
  5. Automatically queries device APIs: health metrics, config data, environmental context, error logs
  6. Searches 30,000+ GTAC KB articles → recommends step-by-step resolution based on device context
  7. If issue persists: builds an Evidence Report (device IDs, config data, health indicators, env metrics, conversation history)
  8. Review and approve the Evidence Report → Service Agent creates a GTAC case with ExtremeWorks — no copy/paste, no email, no form-filling

HITL Hard Rule — Nothing Executes Without Your Approval

Service Agent will NOT: submit evidence reports, create GTAC cases, or execute any configuration changes without explicit user approval. This is by design. Current beta limitations: requires ExtremeWorks package; no proactive/autonomous monitoring; no mobile support; no co-delivered support.

Use Case 4: Route Workflows on Performance Issues (p.39 — Think it Through Beta)

AI Workflow Think it Through (Beta) — Multi-Step Reasoning for Complex Correlations

Think it Through combines real-time telemetry with documentation and KB content for multi-step analytical queries. Use when a problem spans multiple data sources and requires sequential reasoning to reach a conclusion.

  1. Enable via AI Agents Hub (same as Service Agent beta)
  2. Click + → select topic: Think it Through
  3. Ask a multi-dimensional question: “Users on Floor 3 are reporting slow WiFi after yesterday’s firmware update. What could be causing this and what should I check first?”
  4. Think it Through performs sequential reasoning: correlates firmware changelog + device health pre/post + client auth fail rates + channel utilization + GTAC KB articles
  5. Returns a structured report with expandable reasoning section showing its data sources and API calls used — full transparency
  6. Identifies root cause hypothesis with confidence level and supporting evidence
  7. Escalation path: if Think it Through identifies a hardware or cloud bug, hand off to Service Agent for automated evidence collection + GTAC case creation

Current Beta Limitations

  1. What database technologies does the XIQ/EP1 base platform use for telemetry and config? The p.3 diagram shows "DB" per region but does not name the technology.
