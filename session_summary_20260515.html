Verification Workflow: FDB + ARP

Two commands on SW1 are sufficient to prove a client landed on the correct VLAN. Run them immediately after the client associates and gets a DHCP lease.

Step 1 — show fdb ports 3 (which VLAN is each MAC in right now?)

5320-16P-2MXT-2X-SwitchEngine.6 # show fdb ports 3
MAC                       VLAN Name( Tag)  Age  Flags   Port
---------------------------------------------------------------------------
00:e6:0e:55:66:80          Default(0001) 0037  d m      3   ← AP1 management ✅
74:a6:cd:8b:19:60           VLAN10(0010) 0000  d mi     3   ← Students client ✅
76:f6:58:7c:18:00           VLAN10(0010) 0000  d mi     3   ← Students client ✅

The FDB is the authoritative live view. A client MAC under Default(0001) means untagged frames — VLAN1, Staff. Under VLAN10(0010) means tagged VID=10, Students. If the AP’s own MAC appears under VLAN10 instead of Default, the AP’s management path is broken (see May 14 Bug #1).

Step 2 — show iparp vlan <VLAN> (confirm the IP)

5320-16P-2MXT-2X-SwitchEngine.7 # show iparp vlan VLAN10
VR-Default    10.10.0.10    76:f6:58:7c:18:00    6   VLAN10   10   3
VR-Default    10.10.0.11    74:a6:cd:8b:19:60   15   VLAN10   10   3

5320-16P-2MXT-2X-SwitchEngine.8 # show iparp vlan Default
VR-Default    192.168.0.1   58:13:d3:9c:c1:ee   20   Default   1   1   ← QF-Modem GW
VR-Default    192.168.0.10  74:a6:cd:8b:19:60   20   Default   1   1   ← MacBook (stale)

IPARP confirms which IP address each MAC holds and which DHCP server assigned it. Age shown in minutes; timeout = 20 min. The 192.168.0.10 entry for the MacBook is stale — explained below.

Dual-MAC Forensic Observation

MAC 74:a6:cd:8b:19:60 (MacBook) appears in both ARP tables simultaneously. This is not a misconfiguration — it is the expected result when the same device is tested with two different passphrases in the same session:

1. MacBook joins with Staff passphrase
   → VLAN1, gets 192.168.0.10 from QF-Modem
   → Default ARP: 192.168.0.10 / 74:a6:cd:8b:19:60  (age timer starts, 20-min TTL)

2. MacBook joins with Students passphrase
   → VLAN10, gets 10.10.0.11 from SW1
   → VLAN10 ARP: 10.10.0.11 / 74:a6:cd:8b:19:60  (new entry)
   → Default ARP: 192.168.0.10 still present (age = 20, about to expire)
   → FDB: MAC now listed only under VLAN10 (current active state)

The Default ARP entry will evict itself when the 20-min timer fires.
FDB always reflects NOW. ARP reflects up to 20 min of history.

Principle

FDB = live state. Trust it when FDB and ARP disagree. A MAC in VLAN10 on FDB but still in Default ARP does not mean the client is dual-homed — it means the ARP entry is stale. The client is on VLAN10.

New Lesson — ipforwarding and Supplemental CLI

The working Supplemental CLI block included enable ipforwarding vlan Default alongside the DHCP commands. This revealed a new risk:

Do Not Add to Supplemental CLI Template

enable ipforwarding vlan Default is not idempotent on EXOS. Running it when ipforwarding is already enabled returns an error — causing the same XIQ 15% hang as enable dhcp ports 3. Treat it exactly like the DHCP commands: manual post-factory-reset step only, never in the permanent template.

CommandIn Template?Why Not
configure vlan VLAN10 dhcp-*Pass 2 only, remove afterErrors on re-run; XIQ hangs at 15%
enable dhcp ports 3 vlan VLAN10Pass 2 only, remove afterSame — not idempotent
enable ipforwarding vlan DefaultNever — manual step onlySame idempotency problem; XIQ won’t push Default VLAN anyway

Post-factory-reset checklist (run before first XIQ deploy):

  1. enable ports all
  2. enable ipforwarding vlan Default
  3. save config
