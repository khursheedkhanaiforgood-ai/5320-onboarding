PPSK Architecture

Five Actors, One Cryptographic Chain

The session derived the full architecture from first principles, not from the Rieben guide's IT-operator walkthrough. The Rieben guide (v6, 95pp) is valuable as a step-by-step IT reference but contains no RFC citations and does not explain why each object exists.

The Five Actors

Admin 👨‍💻Creates SSID, User Groups, User Profiles, Assignment Rules in XIQ. Creates PPSK users and triggers email/SMS delivery.
XIQ ☁️Policy engine + Cloud PPSK database + SMTP relay + RadSec server. Pre-computes PMK at provisioning time. Returns PMK + VLAN at runtime via RadSec Access-Accept.
AP1 📡RadSec proxy — initiates per-auth call to XIQ after M2. Enforces firewall policy (L2–L7). Tags guest frames VLAN 100.
SW1 🔀EXOS switch. VLAN 100 / 10.100.0.1/24. DHCP server for guest pool. Tagged trunk port to AP1. QF-Modem uplink to internet.
Guest 💻Receives PPSK via email. Types passphrase into standard Wi-Fi dialog. Never sees RADIUS, RadSec, or VLAN selection.

PMK Derivation (the cryptographic core)

The same function runs twice — once at XIQ when the user is provisioned, once on the guest device the moment it receives the ANonce (M1). The PPSK never crosses the air. The only proof of possession sent over-the-air is the MIC in M2.

PMK = PBKDF2(PRF=HMAC-SHA1, Password=PPSK, Salt=SSID_name, c=4096, dkLen=32)
          ↑ IEEE 802.11i §8.5.1.3 + RFC 2898 (PKCS#5)

Runs at XIQ provisioning → PMK stored in Cloud DB
Runs on UE at runtime (after M1) → PMK computed locally
PTK = PRF(PMK ‖ ANonce ‖ SNonce ‖ AP_MAC ‖ STA_MAC)
KCK → signs M2 MIC   KEK → encrypts GTK in M3   TK → encrypts data

The 4-Way Handshake in Cloud PPSK Mode

The critical difference from standard WPA2: the AP does not have the PMK locally. After receiving M2 with the MIC, the AP is stuck and must call XIQ before it can verify or proceed.

  1. UE associates. AP port: LOCKED.
  2. AP sends M1 (ANonce). UE computes PMK + PTK locally. Sends M2 (SNonce + MIC).
  3. AP receives M2 but cannot verify MIC — no PMK stored locally.
  4. AP calls XIQ via RadSec (RFC 6614, TCP 2083). Access-Request: User-Name = client MAC.
  5. XIQ returns Access-Accept: PMK (MS-MPPE-Recv-Key, RFC 2548) + VLAN 100 (Attr 81, RFC 2868) + firewall policy.
  6. AP verifies M2 MIC ✅. Sends M3 (GTK encrypted). Receives M4 ACK.
  7. AP port: OPEN. Frames tagged VLAN 100. Guest device gets DHCP from SW1 VLAN 100 pool.

XIQ Object Hierarchy

SSID "KhKLab-Guest"  (Security: PPSK · WPA2/AES only — NOT WPA3/SAE · Cloud DB)
  └── User Group "Guest-24h"  (Cloud · 24h from first_login · 10-char alphanum)
        └── User "John Smith"  (john@company.com · PPSK: xK9mP2qR7n)
  └── User Profile "Guest-VLAN-100"  (VLAN 100 · Firewall: Guest-Internet-Only)
  └── Assignment Rule: Guest-24h → Guest-VLAN-100  (priority 1)

Key trap (Rieben guide p.20 — "Create User Groups Outside SSID Objects"): User Group must be created inside the SSID configuration for email delivery to work. Rieben ProTip verbatim: "If you create User Groups outside of the SSID, those users will not deliver emails even if you choose the setting to send one." This is the single most common PPSK misconfiguration.

MAC Randomization — Why Binding Breaks Guest Auth

iOS 14+ / Android 10+Randomize MAC per SSID. MAC can change on reconnect in stable mode.
MAC binding for guestsWrong approach. When the MAC randomizes, the bound MAC no longer matches → the device appears as a new unknown device → loses binding on every reconnect.
Correct approachNo MAC binding for guests. PPSK validation is passphrase/PMK-based — MAC is irrelevant to auth. MAC binding only for IoT/managed devices with fixed hardware MACs.
Socratic Session

Key Exchanges and Corrections

Q: Five actors in PPSK?

User derived: UE/user, AP, XIQ, RADIUS-type server where PPSKs stored, admin. Confirmed correct. Addition: SMTP relay as a XIQ sub-service, not a separate actor.

Q: PMK derivation — what does XIQ send to AP?

User: "PPSK needs to be from a secure server. AP creates PMK from private key for user stored in DB. XIQ sends back whatever is required to create the PMK at the AP — a function of the user's shared key."

Correction: XIQ pre-computes PMK = PBKDF2(PPSK, SSID) and sends the PMK itself (not the PPSK) to the AP via RadSec Access-Accept using the MS-MPPE-Recv-Key attribute (RFC 2548). The PPSK never leaves XIQ. Same attribute format used in 802.1X MSK delivery.

Q: What does RadSec return? "UserGroup?"

User: "The RadSec returns the UserGroup." Partially correct intuition about containers, wrong payload.

Correction: User Group is an admin provisioning container, not a runtime return value. RadSec returns the User Profile (VLAN + firewall policy). VLAN delivered via Tunnel-Private-Group-ID, Attr 81 (RFC 2868). Same mechanism as 802.1X VLAN assignment — Attr 81 = "154" on May 11, today = "100".

Q: Control plane vs data plane timeline?

User: "Control sets up the E2E pipe and data flows." Confirmed with refinement: Scenes 1–2 (provisioning) are control-plane-only. Scene 3 up to M4 ACK is the hybrid control/data setup phase. Encrypted data begins at Scene 3 Step 13.

Q: Cloud PPSK WAN risk — MAC binding as solution?

User identified the WAN dependency risk from the May 8 DHCP incident and proposed MAC binding as a mitigation.

Correction: MAC binding makes the problem worse for guests with randomized MACs. The correct mitigation for Cloud PPSK WAN dependency is: Local PPSK fallback (not MAC binding), or accept that guest auth fails during WAN loss (acceptable for guests, unacceptable for employees). MAC binding is appropriate only for IoT with fixed hardware addresses.

Q: Full 12-step runtime walkthrough (written by user independently)

User wrote the entire 12-step flow from memory. Three corrections applied:

  1. M3 = GTK delivery from AP, M4 = ACK from UE — not PMK delivery (PMK was already computed at Step 4).
  2. UE computes PMK locally at M1 receipt — it does not wait for the RadSec round trip.
  3. Provisioning (Scenes 1–2) happens before the guest ever arrives. The guest's Scene 3 runtime auth reuses that pre-computed PMK.

Phase 4a — Vision

An Intent-Based XIQ Wizard

The user articulated the Phase 4a vision: "I want to deploy a guest network at five locations with 100 users max at each location, keep them isolated from all other networks, deliver through policies that govern the five locations independently. Tell me what parameters you need, then take me through the XIQ creation flow and a digital-twin-like flow."

The Key Architectural Insight

XIQ config layer100% deterministic. Given a fully resolved intent, the XIQ object creation sequence has no ambiguity. Five locations → five identical SSID+UG+UP+Rule+NetworkPolicy blocks. No confidence scoring needed here.
NLP/parsing layerConfidence calculator lives here. English is ambiguous. "100 users max" might mean concurrent users vs total provisioned users. "Isolated" might mean VLAN-only or VLAN + firewall + no local subnet routing. The confidence score quantifies parsing uncertainty before the intent reaches the deterministic config engine.
AnalogySame architecture as the CISCO EN CLI Mapping Agent (Sprint 14) — but maps intent → XIQ API calls instead of CLI commands. The confidence calculator is the parsing tool, not the output validator.
RAG backendAll XIQ API docs + EN switch docs + PPSK Guide v6 + RFCs (6614, 2898, 5415, 2865, 2868) in one vector store. Wizard queries RAG before generating API calls to resolve exact endpoint/payload.

RFC Reference Index

Eleven Standards Behind PPSK

Every RFC and IEEE standard that was grounded in today's session, with the specific role each plays in the PPSK flow. These are the primary sources; the Rieben guide is the IT-operator derivative.

PPSK Core — Key Derivation

RFC 2898 / PKCS#5 — PBKDF2
PMK = PBKDF2(HMAC-SHA1, PPSK, SSID_name, 4096, 32 bytes). Same derivation at XIQ provisioning and UE runtime. Salt = SSID name per 802.11i §8.5.1.3.
IEEE 802.11i (2004) — RSN / 4-Way Handshake
PTK key hierarchy: KCK (sign MICs) + KEK (encrypt GTK) + TK (encrypt data). PMK derivation basis. Absorbed into 802.11-2020.
IEEE 802.11-2020 — Full Wi-Fi standard
Association frames, RSN Information Element, 4-Way Handshake §12.7.6, CCMP/AES §12.5.3, Beacon frame structure.

PPSK Runtime — RadSec + RADIUS

RFC 6614 — RadSec: RADIUS over TLS
TCP 2083. AP1 per-auth call to XIQ after M2. Returns PMK + VLAN + FW policy. TLS tunnel pre-established via CAPWAP. Replaces UDP RADIUS with encrypted transport.
RFC 2865 — RADIUS
Access-Request (User-Name = client MAC, Attr 1) / Access-Accept payload. Same framing inside RadSec TLS tunnel. Attr 80 = Message-Authenticator.
RFC 2548 — MS-MPPE-Recv-Key (Vendor 311, Type 17)
Carries the PMK from XIQ → AP in Access-Accept. Identical payload to 802.1X NPS MSK delivery. PPSK reuses this RADIUS attribute entirely.
RFC 2868 — Tunnel-Private-Group-ID (Attr 81)
VLAN assignment in Access-Accept. Attr 81 = "154" on May 11 (802.1X), "100" today (PPSK). AP reads this and applies VLAN tag to guest frames.

Provisioning — CAPWAP

RFC 5415 — CAPWAP
AP ↔ XIQ management tunnel. UDP 5246 (control) / 5247 (data). Carries SSID config, User Group definitions, RadSec TLS certificate to AP. Always-on — not a per-auth channel. CAPWAP push does NOT reach SW1.

Network Layer

IEEE 802.1Q — VLAN tagging
4-byte tag in Ethernet header. AP1 adds VLAN 100 tag after M4 ACK. SW1 EXOS forwards on tagged port toward QF-Modem internet path.
IEEE 802.1X — Port-based access control
Controlled port model: LOCKED until auth, OPEN after M4 ACK. PPSK uses the same port-lock mechanism without EAP — only the 4-Way Handshake.
RFC 2131 — DHCP
Discover → Offer → Request → ACK. DHCP OFFER from SW1 VLAN 100 SVI (10.100.0.1) is the smoking-gun for full E2E confirmation. Lease = 86400s (24h), matching PPSK first_login timer.
