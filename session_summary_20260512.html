02 PPSK Architecture — First Principles

The Rieben guide (v6, 95pp) is IT-operator-driven, not engineering-first. No RFC citations. Today's Socratic session derived the full PPSK architecture from cryptographic and protocol first principles.

Five Actors

Admin 👨‍💻Creates SSID, User Groups, User Profiles, Assignment Rules in XIQ. Creates PPSK users + triggers email delivery.
XIQ ☁️Policy engine + Cloud PPSK database + SMTP relay + RadSec server. Pre-computes PMK at provisioning time. Returns PMK+VLAN at runtime.
AP1 📡RadSec proxy — initiates per-auth call to XIQ after receiving M2. Enforces firewall policy (L2–L7) at AP. Tags guest frames VLAN 80.
SW1 🔀EXOS switch. VLAN 80 10.80.0.1/24. DHCP server for guest pool. Tagged trunk port to AP1. QF-Modem uplink.
Guest 💻Receives PPSK via email. Types passphrase into standard Wi-Fi dialog. Never sees RADIUS, RadSec, or VLAN selection.

PMK Derivation

# IEEE 802.11i §8.5.1.3 + RFC 2898 (PKCS#5)
PMK = PBKDF2(PRF=HMAC-SHA1, Password=PPSK, Salt=SSID_name, c=4096, dkLen=32)

# Same derivation performed TWICE:
#   1. XIQ at provisioning → stores PMK in Cloud DB
#   2. Guest device at runtime (M1 received) → computes PMK locally
# The PPSK never crosses the air. MIC in M2 is the only proof of possession.

Key trap (Rieben guide p.42): User Group must be created inside the SSID configuration for email delivery to work. User Groups created as standalone objects (outside SSID) have email suppressed by XIQ.

MAC Randomization — The Guest Decision

iOS 14+ / Android 10+Randomize MAC per SSID. MAC changes on reconnect in some modes.
MAC binding for guestsWRONG — makes it worse. Bound MAC changes → every reconnect appears as a new unknown device → loses the binding.
Correct approachNo MAC binding for guests. MAC binding only for IoT/managed devices with fixed hardware MACs.
PPSK advantagePPSK validation is by passphrase/PMK — not MAC. Each user's unique key is the identity. MAC randomization is irrelevant to PPSK auth.

03 Socratic Session Log — Key Exchanges

  • Q: Five actors in PPSK?
    User derived: UE/user, AP, XIQ, RADIUS-type server where PPSKs stored, admin. Confirmed correct. Added: SMTP relay as XIQ sub-service.
  • Q: PMK derivation — what does XIQ send to AP?
    User derived: "PPSK needs to be from a secure server. AP creates PMK from private key for user stored in DB. XIQ sends back whatever is required to create the PMK at the AP — a function of the user's shared key."
    Correction refined: XIQ pre-computes PMK = PBKDF2(PPSK ‖ SSID). Sends PMK (not PPSK) to AP via RadSec Access-Accept (MS-MPPE-Recv-Key). Same attribute format as 802.1X.
  • Q: What does RadSec return? "UserGroup?"
    User guessed "UserGroup" as return payload. Corrected: User Group is the admin container. RadSec returns User Profile (VLAN + Firewall policy) — same as RADIUS Attr 81 in 802.1X (Attr 81 = "154" from May 11). Today: Attr 81 = "80".
  • Q: Control plane vs data plane timeline?
    User framed: "Control sets up the E2E pipe and data flows." Confirmed with refinement: provisioning (control) is Scene 1–2. Runtime auth (hybrid control/data setup) is Scene 3 up to M4. Encrypted data is Scene 3 Step 13.
  • Q: Cloud PPSK WAN risk? MAC binding solution?
    User identified: "PPSK on cloud has a risk like what we saw in DHCP/triage on May 8." Confirmed: RadSec dependency = WAN dependency. Suggested MAC binding for cloud. Corrected: MAC binding breaks for iOS/Android randomized MACs. Solution: no MAC binding for guests; MAC binding only for IoT.
  • Q: Step-by-step runtime 12-step walkthrough
    User wrote entire 12-step flow independently. Key corrections: (1) M9/M10 — M3 = GTK delivery, M4 = ACK (not PMK delivery). (2) UE already had PMK since Step 4 (computed locally). (3) "Opens SSID, gets key, logs in and then the work in the backend starts" — confirmed but reversed: provisioning + key delivery happen BEFORE guest arrives. Scene 3 is runtime.

08 Architecture Reference Index

All protocols, standards, and design concepts from today's Socratic session. Live links to primary sources. Saved to persistent memory for future sessions.

PPSK Core — Key Derivation

  • RFC 2898 / PKCS#5 — PBKDF2: PMK = PBKDF2(HMAC-SHA1, PPSK, SSID_name, 4096, 256-bit). Same derivation at XIQ (provisioning) and UE (runtime). Salt = SSID name per 802.11i §8.5.1.3.
  • IEEE 802.11i (2004) — RSN framework, 4-Way Handshake key hierarchy: PTK = KCK (sign MICs) + KEK (encrypt GTK) + TK (encrypt data). PMK derivation basis. Now in 802.11-2020.
  • IEEE 802.11-2020 — Full Wi-Fi standard: Association frames, RSN IE, 4-Way Handshake §12.7.6, CCMP/AES §12.5.3, Beacon frame structure.

PPSK Runtime — RadSec + RADIUS

  • RFC 6614 — RadSec: RADIUS over TLS, TCP 2083. AP1 per-auth call to XIQ. Returns PMK + VLAN + FW policy. Replaces UDP RADIUS with encrypted TCP. TLS tunnel pre-established via CAPWAP.
  • RFC 2865 — RADIUS: Access-Request / Access-Accept. Attr 1 (User-Name = client MAC), Attr 80 (Message-Authenticator). Same framing inside RadSec TLS tunnel.
  • RFC 2548 — MS-MPPE-Recv-Key (Vendor 311, Type 17): carries PMK from XIQ → AP in Access-Accept. Identical payload to 802.1X NPS MSK delivery. PPSK reuses this RADIUS attribute.
  • RFC 2868 — Tunnel-Private-Group-ID (Attr 81) = VLAN ID. Same mechanism as 802.1X VLAN assignment (NPS Attr 81 = "154" on May 11, today = "100"). AP reads this from Access-Accept and applies VLAN tag.

PPSK Provisioning — CAPWAP

  • RFC 5415 — CAPWAP: AP↔XIQ management tunnel, UDP 5246 (control) / 5247 (data). Carries SSID config, User Group definitions, RadSec TLS certificate to AP. Always-on — not a per-auth channel. XIQ CAPWAP push does NOT reach SW1.
  • XIQ IQAgent (EXOS) — Separate path for switch management. If SW1 has IQAgent and is onboarded to MyLab_XIQ, XIQ can push VLAN config. If not onboarded, EXOS CLI required (create vlan Guest_100 tag 100).

Network Layer

  • IEEE 802.1Q — VLAN tagging: 4-byte tag in Ethernet header. AP1 adds VLAN 100 tag after M4 ACK. SW1 EXOS forwards on tagged port. Firewall enforced at AP before tag is applied.
  • IEEE 802.1X — Port-based access control: controlled port locked until auth. PPSK uses the same model — port LOCKED after Assoc, OPEN after M4 ACK. No EAP exchange (only 4-Way Handshake).
  • RFC 2131 — DHCP: Discover → Offer → Request → ACK. Smoking-gun: DHCP OFFER from SW1 VLAN 100 SVI = full E2E confirmed. Lease 86400s = 24h, matches PPSK expiry.

XIQ Object Model (Design)

  • XIQ Configure PPSK SSID (doc) — Official XIQ User Guide: Private Pre-Shared Key SSID configuration.
  • XIQ Object Hierarchy — SSID → User Group (container, Cloud/Local, 24h timer) → User (PPSK per person) → User Profile (VLAN + Firewall) → Assignment Rule. UG must be inside SSID for email delivery.
  • PPSK Constraints — WPA2/AES only (not WPA3/SAE). XIQ-native APs only. Cloud: 100k users/UG, 32 UGs/SSID. Local: 1k users/UG, 10k total. TCP 2083 must be open outbound for RadSec.

Phase 4a — XIQ Wizard Architecture

  • Intent Layer — NLP parser deconstructs English intent. Confidence calculator at this layer only. Example: "5 locations, 100 users each, isolated" → {locations:5, users:100, isolation:VLAN+FW}.
  • Config Layer — Deterministic: once intent fully resolved, XIQ API call sequence is 100% deterministic. No confidence scoring needed here. 5 locations → 5×{SSID+UG+UP+Rule+NetworkPolicy}.
  • RAG Backend — Vector store: XIQ API docs + PPSK Guide v6 (Rieben) + EXOS/VOSS CLI refs + all RFCs above + ExtremeGuest Essentials docs. Query before each API call to resolve exact endpoint/payload.
