Question 1
"I know the OSI model has 7 layers. But why does any of this matter? My laptop just sends data — is this theoretical or does it come up in real troubleshooting?"
✓ Model Answer

The OSI model is a diagnostic framework, not a physical thing. It matters because it gives you a structured place to stop guessing. When something breaks, instead of randomly checking cables, IP configs, and firewall rules, you start at Layer 1 and work up — isolating where the failure actually is.

  • L1 broken → port is down, cable fault, no signal
  • L2 broken → device in wrong VLAN, MAC not in FDB, STP blocking port
  • L3 broken → wrong IP, missing route, gateway unreachable

Every experienced network engineer uses this mentally on every ticket. It's not academic — it's the fastest way to narrow a fault.

Question 2
"When a packet from a wireless client arrives at the 5320 — coming through the AP3000 — which OSI layers does that packet interact with before it gets routed out? Which layers light up and in what order?"
✓ Model Answer
  • L1 — Physical: Client transmits over RF (air interface). AP3000 antenna captures the signal as a bitstream.
  • L2 — Data Link: AP converts bitstream into an Ethernet frame with source/destination MAC headers. Frame is tagged with the client's VLAN ID before being sent over the trunk to the switch.
  • L3 — Network: When the frame reaches the 5320 and the destination IP is on a different subnet, the switch's routing engine examines the IP header to decide where to forward the packet.

L1 and L2 always fire. L3 only fires if the destination is outside the local subnet — otherwise L2 forwarding handles it entirely.

Question 3
"You said the frame gets tagged with a VLAN when it leaves the AP. Who actually puts that tag on — is it the AP itself, or does it happen at the switch? And what does the tag look like — is it just a number?"
✓ Model Answer

The AP puts the tag on. The AP is configured (via XIQ Network Policy / SSID-to-VLAN mapping) to tag outgoing frames with the VLAN ID that matches the SSID the client connected to. This happens before the frame hits the switch trunk port.

The tag is a 4-byte field inserted into the Ethernet frame header (802.1Q standard): 2 bytes TPID (0x8100) + 2 bytes TCI containing a 12-bit VLAN ID (0–4094). So yes — it contains a number (the VLAN ID), plus priority bits.

The switch trunk port receives the tagged frame and uses that VLAN ID to decide which VLAN the frame belongs to — it doesn't add the tag, it reads it.

Question 4
"I said the frame has headers to track it to its destination. Am I talking about the MAC destination or the IP destination? What's the difference — and at each hop, which one is changing and which one stays the same?"
✓ Model Answer

Two different jobs running in parallel:

  • IP addresses (L3): Source and destination IPs stay the same end-to-end — from your laptop to Google's server across 10 routers. Never change.
  • MAC addresses (L2): Rewritten at every hop. When a frame leaves your laptop, it's addressed to your gateway's MAC. When the router forwards it to the next hop, it strips that frame and creates a new one addressed to the next router's MAC. Repeat until the destination.

Think of it as: the IP is the final destination address on a package. The MAC is the label on the delivery truck — it changes at every depot, but the destination on the package never changes.

Question 5
"A Layer 2 switch — when it receives a frame, does it actually open up the IP header to decide where to forward it? Or does it just look at the destination MAC and move on without touching anything inside?"
✓ Model Answer

A pure Layer 2 switch never opens the IP header. It looks only at the destination MAC address in the frame header, checks its FDB (Forwarding Database), finds the port that MAC lives on, and forwards the entire frame out that port — untouched.

It doesn't care what's inside. It doesn't know if it's IP, IPX, or anything else. It just moves frames based on MAC addresses within a VLAN.

The 5320 only opens the IP header when it's acting as an L3 gateway (i.e., the frame's destination MAC is the switch's own MAC address on a VLAN SVI).

Question 6
"What actually is the difference between a Layer 2 switch and a Layer 3 device? At what point does a device need to look at the IP header and make a routing decision — versus just forwarding based on MAC?"
✓ Model Answer

L2 switch: Forwards frames based on destination MAC. Operates within one broadcast domain (VLAN). Never looks at IP. Fast — decision is a simple table lookup.

L3 device (router or L3 switch): A frame arrives addressed to the device's own MAC (meaning "this is for me"). The device then strips the L2 frame, examines the destination IP, looks up its routing table, builds a new L2 frame for the next hop, and forwards it. This is routing.

The trigger: the destination MAC in the frame equals the device's own MAC. That's what signals "open the IP header and route this." If the destination MAC is someone else's, just switch it.

Question 7
"I have Device A on port 1 in VLAN 10, and Device B on port 5 in VLAN 20 — same physical 5320. Can Device A reach Device B right now, without any extra configuration? Just yes or no, and why."
✓ Model Answer

No. VLANs are L2 partitions. Even though both devices are on the same physical switch, the switch refuses to forward frames between different VLAN IDs. VLAN 10 frames stay in VLAN 10; VLAN 20 frames stay in VLAN 20. They are in separate broadcast domains.

To cross the boundary, you need an L3 mechanism — something with an IP address on each VLAN that can route between them. That's the role of the SVI (Switch Virtual Interface) / gateway IP.

Question 8 Gap identified
"Device A is 10.10.10.5 on VLAN 10. Device B is 10.20.20.5 on VLAN 20. Device A can't reach Device B directly — it sends to its default gateway. What should that gateway IP be, and where does it live?"
✓ Model Answer

Device A's default gateway should be 10.10.10.1 — an IP address owned by the 5320 itself on VLAN 10. The switch becomes the gateway by having this IP assigned to its VLAN 10 interface (SVI equivalent).

Similarly, VLAN 20 needs 10.20.20.1 assigned to the switch on VLAN 20. Both gateway IPs live on the same physical 5320 — that's what makes it the L3 router for both VLANs.

EXOS command: configure vlan VLAN_10 ipaddress 10.10.10.1/24 followed by enable ipforwarding vlan VLAN_10.

Question 9 Gap identified
"I know there's a command like configure vlan 10 dhcp-options default-gateway 10.10.10.1 — isn't that the same as making the switch the gateway? Or is that doing something different?"
✓ Model Answer — This is the key distinction

These are two completely different things and mixing them up is one of the most common EXOS misunderstandings:

  • configure vlan VLAN_10 dhcp-options default-gateway 10.10.10.1 — tells the DHCP server what gateway IP to hand out to clients in their DHCP lease. The switch is acting as a DHCP server telling clients "your gateway is X." The switch itself does NOT own that IP.
  • configure vlan VLAN_10 ipaddress 10.10.10.1/24 — assigns the IP address directly to the switch's VLAN 10 interface. Now the switch owns that IP and will respond to ARP requests for it. This is what makes it a gateway.

You need both in a complete setup: assign the IP so the switch is the gateway, then configure DHCP options to tell clients where the gateway is. But the switch can't be the gateway without the ipaddress command.

Question 10 Gap identified
"If the 5320 is going to route from VLAN 10 to VLAN 20 — does it only need the gateway IP for VLAN 10, or does it need IP addresses assigned for both VLANs? And do both have to live on the same switch?"
✓ Model Answer

Both VLANs need gateway IPs, and both must be on the same switch for local inter-VLAN routing.

  • configure vlan VLAN_10 ipaddress 10.10.10.1/24 + enable ipforwarding vlan VLAN_10
  • configure vlan VLAN_20 ipaddress 10.20.20.1/24 + enable ipforwarding vlan VLAN_20

Without the VLAN 20 IP: the switch receives the packet on VLAN 10, looks at the routing table for 10.20.20.x, finds no local route, and drops it. It doesn't know VLAN 20 is reachable.

With both IPs: the switch's routing table automatically has connected routes for both subnets, so it knows exactly where to forward traffic between them. Both routes are directly connected — no static routes needed.
