Context — Why EP1, Why Friday

Extreme Platform One (EP1) is the next-generation portal replacing XIQ Classic. The XIQ New UI becomes the mandatory default on July 1, 2026. Classic is fully retired Q4 2026. No official migration guide exists — the only Extreme document is a “UI Transition FAQ” PDF. This creates a knowledge gap that clients will need their SE to fill.

The Friday session is a discovery call with a Sr Director of Solutions Architecture who will validate readiness across: EP1 deployment workflow, GuestEssentials vs. built-in CWP, AirDefense integration, observability via Network360 and AI Canvas, and the migration narrative for enterprise customers.

Critical Fact

EP1 runs on the same underlying platform as XIQ — same API, same RadSec proxy, same IQAgent heartbeat, same PPSK architecture, same EXOS/VOSS switch management. What changed is the front-end only: a modular app launcher replaces the single unified portal. Technology knowledge transfers 1:1. Navigation changes are the skill gap to close.

What Needs Engineering Verification (7 Open Questions)

  1. Database technologies — what specific DB tech does the XIQ/EP1 base platform use for telemetry and config? The p.3 diagram shows "DB" per region but does not name it.
  2. Data Ingestion scope — does the Data Hub ingest config-change events (policy pushes), or only telemetry and observability data?
  3. Anonymization boundary — does anonymization apply to VIQ storage, or only to the copy forwarded to the GDC AI Core? (PII appendix p.11 implies VIQ retains raw data.)

  6. AI Core → CAPWAP handoff — is the coupling synchronous API, async event (message bus), or operator-triggered?

CWP Decision Framework — Built-in vs. GuestEssentials
Built-in CWP (XIQ / EP1 SSID-level)

Simple T&C acceptance (UPA). Redirect to URL. Basic self-registration form. Embedded in SSID policy. No audit trail. No identity capture. No sponsor workflow. No SMS. No branding beyond basic logo/colors.

Use when: walk-in access, T&C only, no compliance requirement, no identity needed.

GuestEssentials (EP1 Guest App)

Full portal designer (HTML/CSS, logos, colors). Self-registration with email/SMS verification. Employee sponsorship + approval workflow. Social login. Voucher codes. Audit trail with identity. HIPAA/PCI-relevant access logging.

Use when: any identity capture, sponsorship, compliance audit, branded experience, SMS delivery, time-limited access.

GuestEssentials Key Facts
  • Portal types: Self-registration (email or SMS), Employee-sponsored, Social login (Google/Facebook), Voucher, MAC Auth bypass
  • Integration: GuestEssentials generates a portal URL → this URL becomes the CWP redirect in the EP1 SSID config
  • SMS delivery: Can send PPSK or time-limited credentials via SMS — requires SMS gateway config
  • Sponsorship flow: Guest submits request → named sponsor receives email → sponsor approves → guest gets access code
  • Audit trail: Every guest session logged: name, email, device MAC, duration, sponsor (if any) — exportable for compliance
  • From PPSK guide: SMS not delivered when PPSK Classification feature is active — this constraint carries over
  • CWP HTML hack (still valid): To hardcode sponsor email alias and hide field from guest, edit index.html in portal designer

Observability — Network360 + AI Canvas

Network360 (Observe App)

  • Client health score: aggregate of RSSI, retry rate, roaming events, auth failures — 0–100
  • AP health: channel utilization, co-channel interference, retry rate, client count
  • PPSK auth events: found in Client 360 > Authentication History per client
  • CWP session data: Guest portal sessions logged in GuestEssentials audit trail (not Network360)
  • Threshold alarms: Configure > Alarms > New Rule > metric + threshold + notification

AI Canvas (EP1-Only)

Natural language interface for dashboard queries. No XIQ equivalent. Key differentiator for EP1.

  • “Show me all clients with RSSI below −75 in the last 2 hours”
  • “Which APs had the most retries yesterday?”
  • “List clients that failed PPSK authentication in the last hour”
  • “What’s the average session duration on Guest SSID this week?”

Step 0 — Morning Study: EP1 APIs + SNMP (Deep)

Complete this before touching the lab. Goal: understand the programmatic + monitoring surface of EP1 so the Sr Director session can go beyond click-through.

  1. EP1 REST API fundamentals
    • Base URL, auth model (OAuth2 / Bearer token), rate limits — User Guide p.1–5 of API Developer Space
    • Key resource groups: Devices, Network Policies, SSIDs, User Management, Alerts, Location
    • Understand CRUD pattern: GET /xapi/v1/devices → POST /xapi/v1/network-policies → PUT .../deploy
    • Pagination, field filtering (?fields=), async job pattern (config push returns a job ID, poll GET /xapi/v1/jobs/{id})
    • EP1 Swagger/OpenAPI: available at extremecloudiq.com/api-docs (requires login)
  2. SNMP in EP1
    • SNMP configured per Network Policy: Additional Settings → Management Servers → SNMP Server (User Guide p.240)
    • Supported: SNMPv2c (community string) and SNMPv3 (auth/priv). Operations: Get + Trap
    • Community string default in lab: hivecommunity (EN default, change in production)
    • What APs trap on: client assoc/disassoc, rogue detection, channel change, AP up/down, config push status
    • Switches (EXOS): SNMP config is pushed via Supplemental CLI or switch template — separate from AP SNMP
    • SNMP MIBs: EP1 uses Aerohive MIBs for APs (downloadable from GTAC → MIB files). Standard IF-MIB + ENTITY-MIB apply to switches
    • Trap receiver: configure your NMS IP as SNMP server in EP1 → APs send traps directly to NMS, not through EP1 cloud
  3. API vs SNMP — when to use which
    • API: provisioning automation, CI/CD pipeline integration, bulk onboarding, custom dashboards pulling live data
    • SNMP: legacy NMS integration (SolarWinds, PRTG, Zabbix), real-time trap alerting, switch port utilisation polling
    • Both together: SNMP for low-latency operational alerts → API for enrichment (pull device context from EP1 when trap fires)
  4. Syslog (related — quick read)
    • Also in Management Servers: Syslog server IP + port + severity threshold
    • APs send syslog directly to server (UDP 514 by default) — not through EP1 cloud
    • Combine with SNMP: syslog = verbose event stream, SNMP trap = actionable alert

RAG references: EP1 Networking User Guide pp.238–242 (SNMP + Syslog config)  •  EP1 API Developer Space (extremecloudiq.com, requires login)
