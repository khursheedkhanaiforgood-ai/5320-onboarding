Session 01 · Apr 20 + Apr 22
OSI Model Fundamentals (Layers 1–3)
Target: Working knowledge
✓ PASS
  • Explain purpose and responsibilities of Layers 1, 2, and 3
  • Describe how traffic flows from physical transmission through switching and routing
  • Identify common troubleshooting areas associated with each layer
Key Answer — L1/L2/L3 Purpose
L1 Physical: Transmission medium / energy source — electromagnetic waves (wireless), light (fiber), voltage (copper). L2 Data Link: Bit streams → frames with start/end delimiters, MAC addressing, VLAN tagging (802.1Q). L3 Network: End-to-end IP routing, source-to-destination packet forwarding across subnets.
Key Answer — Multi-Hop Packet Flow
Source device checks if destination IP is in its subnet. If yes → ARP for destination MAC, send directly. If no → ARP for default gateway MAC, send to gateway. At each hop: router reads destination IP, consults routing table, ARPs for next-hop MAC, rewrites L2 wrapper, forwards. IP header unchanged end-to-end; only L2 MAC rewrites at each hop.
Key Answer — Troubleshooting per Layer
L1: Client unreachable anywhere → cable, interface, physical link issue. L2: Can't reach different VLAN → missing SVI/gateway IP, wrong VLAN on port. L3: Local OK but no internet → return path broken — most often ipforwarding vlan Default disabled (Karl Rule).
Session 02 · Apr 22
Layer 2 — MAC Addresses, Ethernet & VLANs
Target: Working knowledge
✓ PASS
  • Explain how MAC addresses are used in Layer 2 communication
  • Demonstrate understanding of VLAN segmentation and trunk ports
  • Understand FDB learning, flooding, and VLAN 1 loop prevention
Key Answer — MAC Address Fundamentals
MAC addresses are 48-bit hex identifiers burned into NIC firmware. First 3 octets = OUI (Organizationally Unique Identifier) identifying the manufacturer. Switches read Source MAC to learn (populate FDB), read Destination MAC to forward. Unknown destination MAC → flood to all ports on that VLAN. Broadcast MAC (ff:ff:ff:ff:ff:ff) → flood without FDB lookup.
Key Answer — VLAN Segmentation
VLANs are logical L2 partitions. Two devices on the same physical switch in different VLANs cannot communicate at L2 — completely isolated unless an SVI (L3 gateway) exists. Trunk ports carry multiple VLANs (802.1Q tagged). Access ports carry one VLAN (untagged). VLAN 1 must be removed from inter-switch trunk port (configure vlan Default delete ports 10) to prevent bridging loop triangle.
Key Answer — FDB vs ARP Table
FDB: MAC-to-port mapping on switches — L2 concept, per-VLAN. ARP Table: IP-to-MAC mapping on end devices/routers — L3-to-L2 resolution. FDB entries age out (EXOS default 300s). After aging, next frame is flooded again until source is re-learned.
Session 03 · Apr 22
Layer 3 — IPv4, Subnetting & Routing
Target: Working knowledge
✓ PASS
  • Understand IPv4 addressing structure and subnetting basics
  • Explain default gateway behavior and inter-subnet routing via SVIs
  • Differentiate static vs dynamic routing (OSPF reference)
  • Apply campus network design hierarchy
Key Answer — Subnet Math
IPv4 = 32 bits. /24 = 24 network bits, 8 host bits → 256 addresses (254 usable). /25 = 25 network, 7 host → 128 addresses (126 usable). For 10.40.0.50/25: block is 10.40.0.0–10.40.0.127. Network=.0, Broadcast=.127, Hosts=.1–.126. Same IP but /24 spans 10.40.0.0–10.40.0.255.
Key Answer — SVI as Default Gateway
SVI (Switched Virtual Interface) = a virtual L3 interface bound to a VLAN on a switch. It owns the gateway IP that DHCP clients use. In the Horizon lab: SW1 has SVI 10.10.0.1 (VLAN 10), 10.20.0.1 (VLAN 20), 10.30.0.1 (VLAN 30). SW2 has NO SVIs — pure L2 only. All routing decisions happen on SW1.
Key Answer — Static vs Dynamic Routing
Static: Manually configured routing table entries. Simple, no overhead, brittle — if a link fails, packets are stuck. Dynamic (OSPF): Routers advertise directly-connected subnets, learn topology from neighbors, calculate best path via SPF algorithm, auto-reconverge when topology changes. Same forwarding mechanism at each hop — the difference is only how the routing table is populated.
Key Answer — Campus Network Design
Three-tier hierarchy: Access layer (client ports — untagged access ports, VLAN assignment) → Distribution layer (VLAN aggregation + inter-VLAN routing via SVIs) → Core layer (high-speed backbone, L3 forwarding only). IP header never changes end-to-end; only L2 MAC wrapper rewrites at every hop.
Session 04 · Apr 22
Wireless Networking Fundamentals
Target: Working knowledge
✓ PASS
  • Explain how RF conditions affect wireless performance (RSSI, SNR, dBm scale)
  • Identify frequency bands, non-overlapping channels, and DFS behavior
  • Walk through client connection lifecycle using WPA Simulator
  • Compare WiFi 6 vs WiFi 7 (MLO)
Key Answer — RSSI & dBm Scale
dBm is logarithmic: -30 dBm is stronger than -70 dBm (closer to 0 = stronger). Every 10 dB = 10× power change; every 3 dB ≈ 2× (half power). Practical scale: -30 dBm excellent, -70 dBm weak-usable, -90 dBm unusable. RSSI alone is insufficient — SNR (Signal-to-Noise Ratio) matters equally. Strong RSSI with high interference (bad SNR) means frames can't be decoded.
Key Answer — Frequency Bands
2.4 GHz: 2.400–2.4835 GHz, 22 MHz channel width, only 3 non-overlapping (1/6/11), best range but most congested (IoT + legacy). 5 GHz: UNII-1/2A/2C/3, 20/40/80/160 MHz widths, ~25 non-overlap no-DFS / ~45 with DFS. DFS required on UNII-2A/2C (radar avoidance, 10s channel vacate). 6 GHz (WiFi 6E+): 1200 MHz of spectrum, 59 non-overlapping 20 MHz channels, no DFS, PSC channels for discovery. WiFi 6E and 7 only.
Key Answer — WiFi 6 vs WiFi 7
WiFi 6 (802.11ax): OFDMA, 1024-QAM, 160 MHz max, single-band per connection, MU-MIMO 8×8, TWT for IoT battery. WiFi 7 (802.11be): 4096-QAM, 320 MHz aggregation, MLO (Multi-Link Operation) — client simultaneously uses multiple bands. Example: video on 6 GHz (clean) + audio on 5 GHz (fallback) = 240 MHz aggregated. Mandates H2E for SAE — stronger side-channel attack resistance.
Key Answer — DFS & AFC
DFS (5 GHz): AP detects radar chirps in firmware, immediately vacates channel, switches to clean channel, stays off 30+ min. Automatic, no config needed. AFC (6 GHz Standard Power): AP queries cloud database every 24 hours with GPS + power level; server approves/blocks channels based on registered APs. ~30 min propagation delay when APs shut down.

Key Answer — AP Onboarding Flow
(1) Pre-register AP serial number in XIQ. (2) AP powers on, boots into Default VLAN (VLAN 1) untagged — AP lifeline must be configured on Port 3. (3) AP sends DHCP Discover on Default VLAN. (4) AP contacts Aerohive redirector (cloud service). (5) Redirector returns XIQ URL + onboarding token. (6) AP contacts XIQ, proves identity via pre-registered serial. (7) AP downloads config and joins assigned Network Policy with SSIDs, security settings, VLAN mappings, channel/power.
Key Answer — XIQ Client Lookup
Monitor → Clients (global list across all APs). Filter by AP, SSID, or user. Per-client detail: MAC, IP, SSID, VLAN, RSSI, connection time, association status, roam history. Alternative: Devices → [specific AP] → Clients tab.
Key Answer — "Connected but No Internet" Troubleshooting
Check two things first: (1) Is enable ipforwarding vlan Default on SW1? — Karl Rule. (2) Is the DHCP server enabled on trunk Port 10 between SW1 and SW2? — DHCP reach-back for SW2 clients. (3) Are modem static routes pointing VLAN subnets back to SW1 192.168.0.28? These are the three most common return-path issues in a Primary/Secondary architecture.

Troubleshooting Methodology & Packet Capture
Target: Working knowledge
✓ PASS
  • Apply a logical troubleshooting process (L1 → L2 → L3)
  • Identify the OSI layer where an issue is most likely occurring
  • Assist in isolating common network and wireless issues
  • Understand the purpose of packet captures and navigate basic Wireshark
Key Answer — Layered Troubleshooting Sequence
L1 Physical: Is the AP visible from the switch? Cable, link LEDs, show ports 3 information. L2 Association: Can clients associate to the SSID? VLAN 30 tagged on Port 3? MAC in FDB? L3 DHCP: Did the client get an IP in the correct subnet? 169.254.x.x = DHCP failure. L3 Internet: Does SW1 have enable ipforwarding vlan Default? Are modem static routes configured? DHCP reach-back on Port 10?
Key Answer — Purpose of Packet Capture
Ping/traceroute give high-level binary answers ("works or doesn't"). Packet capture gives frame-level visibility: exact payloads (unencrypted), MAC/IP headers, VLAN tags, QoS priority markings, TCP sequence numbers, retransmissions, timing patterns. Enables root-cause diagnosis and trend analysis (congestion, performance degradation, anomalies) that high-level tools cannot reveal.
Key Answer — Where to Capture for Return-Path Diagnosis
Port 1 (uplink to modem): Confirm ICMP Echo Requests leave SW1 AND Echo Replies arrive back from 8.8.8.8. Missing inbound → modem/internet issue. Port 3 (AP port): Confirm Echo Requests from VLAN 30 client reach SW1 AND Echo Replies (tagged VLAN 30) return to client. Missing inbound on Port 3 while present on Port 1 → ipforwarding vlan Default disabled. Wireshark filter: ip.dst == 10.30.0.150

WiFi 6/7 Client Connection Flow — Deep Dive

Five-stage walkthrough: Discovery → Authentication → Association → 4-Way Handshake → Data Plane

◆
Walk through this alongside the WPA Simulator and the 802.11be Simulator. The simulators animate exactly what's described here.
Stage 01
Discovery — iPhone Finds the AP
Passive scan: iPhone hops through channels listening for Beacon frames (broadcast by APs every ~100ms). Each beacon contains SSID, BSSID (AP MAC), supported rates, channel, security type, RSSI.

Active scan: iPhone broadcasts Probe Requests on each channel. Every AP in range responds with Probe Response containing the same info.

iPhone builds "Available Networks" list and user taps their choice. WiFi 7 adds 6 GHz PSC (Preferred Scanning Channels) for faster discovery — 15 priority channels instead of scanning all 59.
Beacon Frame: [Frame Control] [Timestamp] [Beacon Interval=100ms] [Capability Info] [SSID IE: "Horizon-Corp"] [Supported Rates] [Current Channel: 36] [RSSI] [Security IE: WPA3-SAE]
Stage 02
Authentication — SAE Handshake (WPA3)
User enters password and taps SSID. iPhone sends Authentication Request — but the password is never transmitted.

SAE Dragonfly exchange: Both iPhone and AP independently use the password as input to a cryptographic function, generating matching scalars and elements. They exchange these in Commit frames. If both used the same password, the math derives identical PMK on both sides.

Confirm frames: Both sides prove they derived the same PMK via a MIC check. Mutual authentication complete — with forward secrecy. Capturing the entire exchange yields zero information about the password.
SAE Flow (5 frames): 1. iPhone → AP: Auth Request (Algo=SAE) 2. AP → iPhone: SAE Commit (AP's scalar + element) 3. iPhone → AP: SAE Commit (iPhone's scalar + element) 4. iPhone ↔ AP: SAE Confirm (MIC verification both ways) 5. AP → iPhone: Auth Response (Status=Success)
WiFi 7 mandates H2E (Hash-to-Element) method for SAE — stronger side-channel attack resistance.
Stage 03
Association — iPhone Declares Capabilities
iPhone sends Association Request containing all its capabilities: supported rates, HT (WiFi 4+), VHT (WiFi 5+), HE (WiFi 6), EHT (WiFi 7), security RSNE. AP evaluates and responds.

Association Response: AP assigns an AID (Association ID — e.g., AID=5 for "client #5"). iPhone is now associated. Both sides know each other's negotiated parameters.

WiFi 6: Single-band negotiation. WiFi 7: MLO negotiation — iPhone declares "I can do 5 + 6 GHz concurrently," AP agrees to dual-link operation.
Association Request: [Capability Info] [Listen Interval] [SSID] [Supported Rates] [HT Cap] [VHT Cap] [HE Cap] ← WiFi 6 [EHT Cap] ← WiFi 7 (includes MLO capabilities) [RSNE: WPA3-Personal] Association Response: [Status=Success] [AID=5] [Supported Rates + negotiated capabilities]
Stage 04
4-Way Handshake — Key Derivation (PTK + GTK)
The PMK from SAE is temporary. The 4-Way Handshake derives the actual encryption keys for data traffic.

M1 (AP → iPhone): ANonce (AP's random nonce), unencrypted.
M2 (iPhone → AP): SNonce + MIC. Both sides now have ANonce + SNonce + MACs → independently derive identical PTK via PRF function.
M3 (AP → iPhone): GTK (Group Transient Key for broadcast/multicast) encrypted with the newly-derived PTK + MIC confirmation.
M4 (iPhone → AP): ACK. Both sides install PTK and GTK.

After M4, all subsequent data frames are encrypted with PTK using CCMP (AES-128) or GCMP-256 (WiFi 7 192-bit mode).
4-Way Handshake: M1: AP → iPhone [ANonce] M2: iPhone → AP [SNonce, MIC] → both derive PTK M3: AP → iPhone [GTK_encrypted, MIC] → AP confirms PTK install M4: iPhone → AP [ACK] → iPhone confirms install PTK derivation formula: PTK = PRF(PMK, "Pairwise key expansion", Min(MAC_ap, MAC_client) + Max(MAC_ap, MAC_client) + Min(ANonce, SNonce) + Max(ANonce, SNonce))
Stage 05
Data Plane — DHCP → DNS → TCP → Zoom
Encryption is live. iPhone proceeds through the OSI stack for actual application traffic.

Step 1 — DHCP: iPhone broadcasts DHCP Discover (encrypted with PTK in WiFi frame). AP decrypts, forwards unencrypted Ethernet to switch. SW1 DHCP server on VLAN 20 responds with 10.20.0.150, gateway 10.20.0.1, DNS 8.8.8.8.

Step 2 — DNS: iPhone queries DNS for zoom.us → 203.0.113.100.

Step 3 — TCP: iPhone opens socket [10.20.0.150:54321 ↔ 203.0.113.100:443], completes SYN → SYN-ACK → ACK.

Step 4 — Zoom stream: Encrypted WiFi frames carry HTTPS-wrapped Zoom video. QoS priority=5 (high) ensures Corporate traffic gets serviced before Guest (priority=0) during congestion.
Complete stack per Zoom video frame: L7 Application: Zoom video codec output L6 Presentation: TLS/SSL encryption (HTTPS) L5 Session: HTTPS session state L4 Transport: TCP src=54321, dst=443 L3 Network: IP src=10.20.0.150, dst=203.0.113.100 L2 Data Link: WiFi frame with CCMP encryption (PTK) QoS Priority=5, BSSID, AID=5 L1 Physical: 5 GHz channel 36 OFDM modulation
Encryption scope: WiFi CCMP protects iPhone ↔ AP only. Once AP decrypts and forwards to switch, the Ethernet side is unencrypted. HTTPS (L7) provides end-to-end encryption beyond the AP boundary.

Two-capture-point diagnostic logic for return-path problems in the Horizon lab

Capture Point 1 — Port 1 (Uplink)
Mirror Port 1 to a Wireshark capture device. Run traceroute 8.8.8.8 from a VLAN 30 client.
Outbound ICMPEcho Request → 8.8.8.8
Inbound ICMPEcho Reply from 8.8.8.8
If IN missingModem / ISP issue
If both presentUplink path OK
Capture Point 2 — Port 3 (AP Port)
Mirror Port 3 to a Wireshark capture device. Same traceroute running from VLAN 30 client.
Outbound (VLAN 30)Client → 8.8.8.8
Inbound (VLAN 30)Echo Reply → client
If IN missingipforwarding vlan Default
If both presentL3 forwarding OK
✓
Diagnostic matrix:
Port 1 (OUT ✓ IN ✗) → Modem/ISP problem, verify static routes to 10.x.0.0/24.
Port 1 (OUT ✓ IN ✓) + Port 3 (OUT ✓ IN ✗) → enable ipforwarding vlan Default missing.
Port 1 (OUT ✓ IN ✓) + Port 3 (OUT ✓ IN ✓) → Not a routing issue — check DNS, client isolation ACL, firewall.
◆
Wireshark filter tips:
ip.src == 10.30.0.150 or ip.dst == 10.30.0.150 — isolate your client's traffic
icmp — show only ping traffic
vlan.id == 30 — show only VLAN 30 tagged frames
bootp — show DHCP traffic
wlan.fc.type_subtype == 0x04 — show Probe Requests (discovery debugging)

Lessons Learned

Eight permanent takeaways from the Apr 21–22 lab — carry into every future build

01

The Karl Rule — enable ipforwarding vlan Default

SW1 owns the Default VLAN interface (192.168.0.28/24), which is the path to the modem. Without ipforwarding on Default VLAN, return traffic from internet hits Default VLAN, routing is disabled, packet dropped. Clients get DHCP and ping gateway but are completely ghosted by internet. Slide 7 of Karl's PPTX: "The switch checks the Default VLAN state, sees routing is disabled, and explicitly drops the packet."

enable ipforwarding vlan Default
02

DHCP Reach-Back — include Port 10 in enable dhcp command

SW1's DHCP server only listens on ports explicitly listed in the enable dhcp command. If only Port 5 is listed, SW1 ignores DHCP requests arriving from SW2 via trunk on Port 10. Clients on SW2 get 169.254.x.x self-assigned IPs. Always include the trunk port for all data VLANs.

enable dhcp ports 5,10 vlan VLAN_10_Corporate_Wired
03

VLAN 1 Loop Triangle — delete Port 10 from Default VLAN on both switches

Both switches are connected to the modem via Port 1 on VLAN 1. If VLAN 1 is also on the Port 10 trunk, a loop triangle forms: SW1 ↔ Modem ↔ SW2 ↔ SW1. Without STP blocking a path, this creates a broadcast storm. Always remove VLAN 1 from the inter-switch trunk port on BOTH switches.

configure vlan Default delete ports 10 ← run on SW1 AND SW2
04

AP Lifeline — Default VLAN untagged on Port 3

An AP3000 boots with no config — it sends an untagged DHCP request. Port 3 must have VLAN 1 (Default) untagged so the AP gets a 192.168.0.x management IP from the modem and can reach XIQ to download its config. VLAN 20 and 30 ride tagged on the same port for SSID traffic.

configure vlan Default add ports 3 untagged
05

Port Assignments Must Be Explicit — EXOS Does Not Auto-Assign

Creating a VLAN and assigning an IP does not automatically associate ports. Port add commands must be run explicitly for every VLAN on every relevant port. VLANs showed 0 active ports during initial testing because port assignments were omitted.

configure vlan VLAN_10_Corporate_Wired add ports 5 untagged
06

Guest-to-Guest Isolation — Must Use XIQ Station Isolation, Not ACLs

EXOS ACLs operate at L3 and cannot stop L2 intra-VLAN traffic between guests on the same AP. Must use XIQ "Drop traffic between stations" firewall action in Guest_30 User Profile. This is the East-West defense (vs North-South = XIQ L3 Firewall Rules blocking Guest → Corporate).

XIQ → User Profiles → Guest_30 → Firewall → Drop traffic between stations
07

EXOS ACL Syntax — entry containers with semicolons, explicit permit all

Cisco-style one-liners FAIL on EXOS. Must use the policy editor with entry containers, semicolons, and an explicit permit-all at the end. For the Horizon lab: bypassed by using XIQ firewall rules instead of CLI ACLs.

edit policy GUEST_ISOLATION # opens vi editor entry deny-to-corp { if { source-address 10.30.0.0/24; } then { deny; } } entry permit-all { if { } then { permit; } } # Esc + :wq to save
08

XIQ IP Objects Must Exist Before Firewall Rules

When adding firewall rules in XIQ User Profiles, all IP objects referenced must be pre-created in Common Objects. If you try to reference an object that doesn't exist: "IP object must be saved first" error blocks the entire rule. Create Guest_Network, Corp_Wired, Corp_Wireless in Common Objects first.

XIQ → Common Objects → IP Objects → Create (before adding firewall rules)
