Why Starting Clean Matters

Iterative patching hides architectural mistakes. A fresh build from scratch forces every configuration decision to be explicit — and that is how the VLAN tagging bug was discovered clearly rather than buried in layers of partial fixes.

Step 1 — SW1 Factory Reset

SSH to SW1 (192.168.0.28). The factory reset command in EXOS SwitchEngine is:

unconfigure switch all

The switch reboots automatically. On the way back up, EXOS runs a provisioning wizard. The critical question:

Wizard: Disable Unconfigured Ports?

Always answer NO. If you answer YES, EXOS executes disable ports all and every port shuts down — including port 1 (uplink to QF-Modem) and port 3 (AP). IQAgent loses connectivity to XIQ. You then have to SSH in and run enable ports all before anything works. Save yourself the pain: always NO.

Immediately after the wizard, run:

enable ports all
save config
# Default login after factory reset: admin / (blank password)

At this point SW1 is clean. Default VLAN has port 1–20 untagged, gets 192.168.0.x from QF-Modem DHCP, IQAgent connects to XIQ within a minute or two.

The AP’s MAC address (00:E6:0E:55:66:80) appeared on VLAN10, not on the Default VLAN. This was the smoking gun. Understanding why this is wrong requires understanding how VLAN tagging works on a switch port.

VLAN Tagging Deep Dive — What Tagged vs Untagged Actually Means

The Basics: What is a VLAN Tag?

When a frame travels across a network cable, it is just bytes. A VLAN tag (defined by IEEE 802.1Q) is an extra 4-byte label inserted into the Ethernet frame header. It contains the VLAN ID — a number from 1 to 4094 — that tells the network equipment “this frame belongs to VLAN 10” (or 20, or 100, etc.).

  • Tagged frame: has the 4-byte 802.1Q header. Any switch or device that receives it can read the VLAN ID and know which VLAN it belongs to.
  • Untagged frame: a plain Ethernet frame with no VLAN label — the same format a computer or AP sends by default, because most end devices don’t know about VLANs at all.
TAGGED FRAME:
  ┌──────────┬──────────┬────────────────────────┬───────────┬──────────────┐
  │ Dest MAC │ Src MAC  │  802.1Q Tag (4 bytes)  │ EtherType │   Payload    │
  │ 6 bytes  │ 6 bytes  │  TPID=0x8100, VID=10  │  2 bytes  │              │
  └──────────┴──────────┴────────────────────────┴───────────┴──────────────┘
                                ↑ this tells the switch: "I belong to VLAN 10"

UNTAGGED FRAME:
  ┌──────────┬──────────┬───────────┬──────────────┐
  │ Dest MAC │ Src MAC  │ EtherType │   Payload    │
  │ 6 bytes  │ 6 bytes  │  2 bytes  │              │
  └──────────┴──────────┴───────────┴──────────────┘
                   ↑ no VLAN label — belongs to whichever VLAN owns this port

How a Switch Port Handles Tagged vs Untagged

A switch port that carries multiple VLANs (a “trunk” port) has two rules:

  • One untagged (native) VLAN. Frames arriving on this port without a tag are assigned to this VLAN. Frames leaving this port for this VLAN have their tag stripped before transmission. The device on the other end never sees a tag for the native VLAN.
  • One or more tagged VLANs. Frames arriving with a matching tag are put into that VLAN. Frames leaving for these VLANs keep their tag. The device on the other end must understand 802.1Q to read the tag and handle it correctly.
Port 3 — correct architecture (what we want):

SW1 Port 3 ──────────────────────── AP1
                                     │
  Native VLAN = Default (VLAN 1)     │ AP's own traffic (untagged)
  ◄── untagged frames ──────────────►│ AP DHCP, CAPWAP, SSH = no tag
                                     │ → switch assigns to Default VLAN
                                     │ → AP gets 192.168.0.x from QF-Modem
                                     │ → AP can reach XIQ on internet ✅
                                     │
  Tagged VLAN = VLAN10 (tag=10)      │ Client traffic (tagged by AP)
  ◄── [802.1Q tag: VID=10] ─────────►│ When a client joins KB_School_Broadcast,
                                     │ AP wraps client frames with tag VID=10
                                     │ → switch puts on VLAN10
                                     │ → client gets 10.10.0.x from SW1 DHCP ✅

What XIQ Pushed Instead (The Bug)

XIQ configured the AP Port Type with VLAN10 as the untagged (native) VLAN on port 3. The Default VLAN got pushed as tagged. This is the exact opposite of what is needed.

Port 3 — WRONG (what XIQ deployed):

SW1 Port 3 ──────────────────────── AP1
                                     │
  Native VLAN = VLAN10               │ AP's own traffic (untagged)
  ◄── untagged frames ──────────────►│ AP DHCP, CAPWAP, SSH = no tag
                                     │ → switch assigns to VLAN10  ← WRONG
                                     │ → AP gets 10.10.0.x from SW1 DHCP
                                     │ → AP tries to reach XIQ from 10.10.0.x
                                     │ → No route to cloud XIQ from 10.10.0.x ✗
                                     │ → CAPWAP cannot establish ✗
                                     │ → LED ORANGE ✗
                                     │
  Tagged VLAN = Default (VLAN 1)     │ Impossible for AP to use
  ◄── [802.1Q tag: VID=1] ──────────►│ AP does not generate VLAN-tagged mgmt frames

Why Can’t the AP Reach XIQ from 10.10.0.x?

XIQ (ExtremeCloud IQ) is a cloud service on the internet. To reach it, the AP needs:

  1. An IP address on a subnet that has a path to the internet
  2. A default gateway that can route to the internet

When the AP is on the Default VLAN (192.168.0.x), its gateway is the QF-Modem (192.168.0.1) which has a direct internet connection. Path to XIQ: clear.

When the AP is on VLAN10 (10.10.0.x), its gateway is SW1’s SVI (10.10.0.1). SW1 can forward to the QF-Modem via its default route, but the QF-Modem has no concept of forwarding 10.10.0.x management traffic to XIQ’s servers. The 10.10.0.0/24 subnet is intended for client user traffic only — not for AP-to-cloud management.

The Key Insight

The AP has only one physical network interface. That interface sends untagged frames. Whichever VLAN is native/untagged on its switch port is where those frames land — and therefore which subnet the AP lives on. If the AP’s management subnet cannot reach XIQ, CAPWAP cannot establish, full stop. The SSID, the PPSK config, the VLAN assignments — none of that matters if the AP cannot talk to XIQ first.

# VLAN Interface with name "Default" created by user

This confirmed it. VLAN10 owned port 3 as untagged. Default VLAN did not have port 3 at all (or had it tagged, which an AP cannot use). The AP’s management frames had nowhere to go except VLAN10.

Bug 1 Fix — Three CLI Commands, LED Goes Green

The fix is three commands: remove port 3 from VLAN10, add port 3 to Default as untagged (native), add port 3 back to VLAN10 as tagged.

configure vlan VLAN10 delete ports 3
# Removes port 3 from VLAN10 entirely

configure vlan Default add ports 3 untagged
# Adds port 3 to Default VLAN as the native/untagged VLAN
# Now: AP's untagged management frames → Default VLAN → 192.168.0.x

configure vlan VLAN10 add ports 3 tagged
# Adds port 3 to VLAN10 as a tagged VLAN
# Now: frames with 802.1Q tag VID=10 → VLAN10 → 10.10.0.x

save config

Verify the fix

show vlan detail

# Correct output after fix:
# "Default" — Ports (untagged): 1-20, active: 1 *3   ← port 3 active, untagged
# "VLAN10"  — Ports (tagged):   T*3, active           ← port 3 active, tagged

Within seconds of running save config: AP1 LED changed from orange to green. The AP’s management frames were now landing on Default VLAN → AP got a 192.168.0.x IP from QF-Modem → resolved redirector.aerohive.com → CAPWAP tunnel established → XIQ showed AP transitioning to “Connected.”

Why the LED Change Was Immediate

The AP had been continuously sending DHCP DISCOVER frames (broadcast) and CAPWAP discovery packets. The switch was receiving them (6,972 RX frames on port 3, 3,743 broadcasts recorded in port statistics) but delivering them to the wrong VLAN. The moment port 3 was corrected, the AP’s next DHCP DISCOVER landed on Default VLAN, QF-Modem responded with a 192.168.0.x lease, and CAPWAP came up in the same connection cycle.

  • Direct route for 10.10.0.0/24 via VLAN10 — present
  • Direct route for 192.168.0.0/24 via Default VLAN — present

SW1 knew where to send packets. So what was stopping it?

What IP Forwarding Actually Does

By default on EXOS, each VLAN is treated as an isolated Layer 3 island. A packet that arrives on VLAN10 stays on VLAN10. For SW1 to act as a router between VLANs — or to forward a packet from VLAN10 out through the Default VLAN to the QF-Modem — IP forwarding must be explicitly enabled on the outgoing VLAN interface.

Client on VLAN10 wants to reach 8.8.8.8 (internet):

  iPhone (10.10.0.10)
       │ sends packet to gateway 10.10.0.1
       ▼
  SW1 VLAN10 interface (10.10.0.1)
       │ packet arrives, destination = 8.8.8.8
       │ route lookup: default route = 192.168.0.1 via Default VLAN
       │
       ▼ ipforwarding vlan Default = OFF  ← THE BUG
       ✗ SW1 refuses to forward out through Default VLAN
       ✗ packet dropped here, never reaches QF-Modem

  ─────────────────────────────────────────────────────────────

  After fix — ipforwarding vlan Default = ON:

  iPhone (10.10.0.10)
       │
       ▼
  SW1 VLAN10 (10.10.0.1) ──forward──► SW1 Default VLAN (192.168.0.28)
                                              │
                                              ▼
                                       QF-Modem (192.168.0.1)
                                              │
                                              ▼

Note that ipforwarding was enabled on VLAN10 by XIQ (because VLAN10 is a routed SVI). But XIQ does not enable it on the Default VLAN. After a factory reset, Default VLAN forwarding defaults to OFF. It must be enabled manually.

Bug 2 Fix — enable ipforwarding vlan Default 
5320-16P-2MXT-2X-SwitchEngine.40 # enable ipforwarding vlan Default
* 5320-16P-2MXT-2X-SwitchEngine.41 # save config

Internet was restored immediately on both devices. The asterisk (*) in the prompt after the enable command is EXOS confirming the running config has unsaved changes — it disappears after save config.

This Must Be Run After Every Factory Reset

XIQ deploys ipforwarding ON for routed VLAN objects (like VLAN10) because you configured it in the SVI section. It does not configure the Default VLAN’s ipforwarding because Default VLAN is not part of the Routing section in the XIQ policy. After every factory reset, enable ipforwarding vlan Default must be applied manually. Consider adding it to the Supplemental CLI so XIQ pushes it on every deploy.

Bug 3 — XIQ Policy Stuck at 15% on Re-Deploy

Supplemental CLI Ran Twice. EXOS Already Had the Config. XIQ Hung.

After verifying the XIQ Port Type (Native VLAN 1, Allowed VLAN 10 — correct), the policy was deployed again to push the confirmed port config to the switch. The deploy immediately hung at 15%. Same symptom as May 13.

This time the Supplemental CLI commands themselves were syntactically correct — the issue was that DHCP was already configured and enabled on the switch from the earlier deploy. When XIQ re-ran the Supplemental CLI, EXOS returned an error on commands that were trying to configure something already in place (particularly enable dhcp ports 3 vlan VLAN10 — DHCP was already enabled on that port). XIQ interprets any CLI error as a failure and hangs waiting for the job to complete.

Root Cause

Supplemental CLI is not idempotent on EXOS. Commands like enable dhcp ports 3 vlan VLAN10 succeed on first run but error if DHCP is already enabled. XIQ does not handle this gracefully — it hangs at 15% instead of ignoring the error. The Supplemental CLI should be removed from the template once the initial provisioning is confirmed working.

What to Do When Stuck at 15%

  1. Do nothing. Wait 5–10 minutes for the XIQ config job to time out. The lock will clear automatically. The switch is unaffected — its running config and DHCP pool remain intact.
  2. Once the spinning indicator stops, go to the template: Advanced Settings → Supplemental CLI → remove or uncheck VLAN10_DHCP.
  3. Deploy again. It will reach 100% cleanly.
  4. DHCP config on the switch is not lost — it was saved to primary.cfg earlier. The switch will continue serving 10.10.0.x addresses without the Supplemental CLI in the template.

PPSK Confirmed Working

Despite the XIQ policy being stuck, clients remained connected and PPSK continued to work. The switch and AP were not affected by the XIQ deploy lock. iPhone and MacBook both retained internet access throughout.

The Right Sequence for Supplemental CLI — Every Time

Deploy PassSupplemental CLIPurpose
Pass 1OFFConfirm base config (VLAN, ports, SVI, routing) deploys clean to 100%
Pass 2ONPush DHCP commands. Confirm 100%. Verify show dhcp-server.
Pass 3+ (any future deploy)OFFDHCP is already on the switch and saved. Remove Supplemental CLI to avoid re-run errors.

Future Option: Make Supplemental CLI Idempotent

If you want to keep Supplemental CLI permanently in the template (e.g. for disaster recovery after a factory reset), prefix DHCP commands with a disable step to reset state before re-enabling:

disable dhcp ports 3 vlan VLAN10
configure vlan VLAN10 dhcp-address-range 10.10.0.10 - 10.10.0.254
configure vlan VLAN10 dhcp-options default-gateway 10.10.0.1
configure vlan VLAN10 dhcp-options dns-server primary 8.8.8.8
configure vlan VLAN10 dhcp-options dns-server secondary 1.1.1.1
configure vlan VLAN10 dhcp-lease-timer 86400
enable dhcp ports 3 vlan VLAN10
enable ipforwarding vlan Default

This way the CLI is safe to re-run on every deploy: it disables first, reconfigures, then re-enables. The last line also ensures ipforwarding vlan Default is always pushed. Note: this approach requires testing — not yet verified on this switch.

Never Use ${vlan:} Variable in Supplemental CLI

XIQ offers a template variable ${vlan:OBJECT_NAME} to inject VLAN names dynamically. On EXOS switches it resolves to the VLAN ID number (e.g. 10), not the name. EXOS configure vlan commands require the name. Result: “%% A number within the range of 1–4094 is expected”, deploy hangs at 15%.

Always hardcode the name. Verify the exact name on the switch with show vlan after the first deploy. XIQ drops underscores: XIQ object VLAN_10 → switch VLAN name VLAN10. XIQ object Guest_100 → switch VLAN name Guest100.

Lessons Learned — 8 Principles
  1. AP management traffic is always untagged. The AP sends its own DHCP, CAPWAP, and SSH frames without any 802.1Q tag. Whichever VLAN is native/untagged on the AP’s switch port is where those frames land. Get this wrong and the AP cannot reach XIQ, regardless of everything else.
  2. Default VLAN must be native on the AP port. Default VLAN (VLAN 1, 192.168.0.x) has a direct path to QF-Modem and the internet, so AP management can reach XIQ. Client VLANs (VLAN10) must be tagged — the AP tags client frames itself, then sends them tagged to the switch.
  3. show fdb ports 3 is the AP diagnostic smoking gun. AP’s MAC on Default VLAN = correct. AP’s MAC on VLAN10 = tagging inverted, CAPWAP cannot establish.
  4. XIQ does not always get AP port tagging right on first deploy. Always run show vlan detail after deploying a new switch policy with an AP port. Never assume XIQ pushed the correct native/tagged split.
  5. enable ipforwarding vlan Default is mandatory after every factory reset. XIQ enables ipforwarding for VLAN10 (because you configured it in the Routing section) but not for Default VLAN. Without it, SW1 refuses to forward client traffic out to QF-Modem. Clients get DHCP and a gateway, but internet times out. Confirmed May 14, April 7, March 29 2026.
  6. Deploy base config without Supplemental CLI first. A Supplemental CLI error hangs the deploy at exactly 15%. Confirm the base config passes 100% before adding DHCP commands.
  7. Never use ${vlan:} in XIQ Supplemental CLI for EXOS. It resolves to the VLAN ID number (e.g. 10), not the name. EXOS rejects it. Hardcode the VLAN name. XIQ drops underscores when naming VLANs on the switch (VLAN_10 → VLAN10).
  8. EXOS factory reset is unconfigure switch all, not delete config.cfg (that is Fabric Engine / VOSS only). Answer NO to the wizard’s “disable unconfigured ports” question.
  9. Remove Supplemental CLI from the template after initial provisioning. Supplemental CLI is not idempotent on EXOS. Commands like enable dhcp ports 3 vlan VLAN10 fail if DHCP is already enabled — XIQ hangs at 15%. Once the DHCP config is confirmed on the switch and saved to primary.cfg, remove the Supplemental CLI from the template. It will persist on the switch through reboots. Only add it back if doing a full factory reset.

Why No SW1 Changes Were Needed

Port 3 was already configured correctly for this split:

  • Default VLAN (VLAN1) = native/untagged on port 3. When XIQ assigns a staff client to VLAN1, the AP sends their frames with no tag. The switch receives untagged frames on port 3 and assigns them to the native VLAN — Default. Staff client gets 192.168.0.x from QF-Modem.
  • VLAN10 = tagged on port 3. When XIQ assigns a student to VLAN10, the AP wraps their frames with an 802.1Q tag (VID=10). The switch sees the tag and routes to VLAN10. Student gets 10.10.0.x from SW1 DHCP.

Confirmed with show vlan detail before deploying:

Default  →  Untag: *1, *3   ← port 3 untagged, active ✅
VLAN10   →  Tag:   *3       ← port 3 tagged, active ✅

Key Principle

VLAN1 (Default) is the native VLAN on the AP port. Clients assigned to VLAN1 by XIQ receive untagged frames from the AP — the switch’s native VLAN assignment handles them automatically. No additional CLI or template change is required to support a second user group on VLAN1, as long as the native VLAN on the AP port is already Default.
