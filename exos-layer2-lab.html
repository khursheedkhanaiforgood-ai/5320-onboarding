EXOS Layer 2 Switching Lab

Build a 6-switch campus ring from scratch — VLANs, trunks, access ports, management IPs, and VPCS client validation.

Author KK Collaboration
Date March 27, 2026
Duration 60–90 min
Platform EVE-NG / Extreme EXOS SwitchEngine

Learning Objectives

  • Navigate the EXOS CLI and set switch hostnames
  • Create and manage VLANs across a 6-switch ring topology
  • Understand and apply tagged (trunk) vs. untagged (access) port configuration
  • Configure management IP addresses on a dedicated management VLAN
  • Verify Layer 2 connectivity between VPCS clients on different edge switches
  • Understand how STP prevents loops — and what happens when it is misconfigured

Network Topology

Six EXOS switches form a ring. Edge-1 and Edge-2 serve VPCS clients on access ports (3/4/5). The four core switches carry only trunk traffic. STP will block one ring link to prevent a broadcast loop.

Step 1 — Set Hostnames

Step 1 of 8 — All 6 switches

Give Each Switch a Meaningful Name

Why: EVE-NG assigns generic names (ExtremeOs1…). Setting the SNMP sysName changes the CLI prompt so you always know which switch you're on — critical when working across 6 consoles.
configure snmp sysName "Core-1" ↪ Changes CLI prompt to "Core-1 #". Run the equivalent on each switch substituting the correct name. show switch ↪ Verify: look for "SysName: Core-1" in the output.

Step 2 — Create VLANs

Step 2 of 8 — ALL 6 switches

Create Data, Voice, and Management VLANs

Why: A VLAN missing on even one switch in the ring will silently break connectivity for that VLAN across the entire ring. Create all three VLANs before touching any ports.
create vlan Data tag 10 ↪ "Data" is the VLAN name. Tag 10 is the 802.1Q VLAN ID carried in the Ethernet frame header on trunk ports. create vlan Voice tag 20 ↪ Separate VLAN for voice traffic — isolates it from data so QoS policies and troubleshooting are easier. create vlan Kh_Mgmt tag 30 ↪ Dedicated management VLAN. Note: "Mgmt" alone may be reserved in some EXOS builds — use "Kh_Mgmt" or any non-reserved name. show vlan ↪ Verify all three VLANs appear with correct tag IDs before proceeding.

Step 3 — Configure Trunk Ports

Step 3 of 8 — ALL 6 switches, ports 1 and 2

Tag All Three VLANs on Switch-to-Switch Links

Why: Ports 1 and 2 carry traffic between switches. They must be tagged so the receiving switch knows which VLAN each frame belongs to. Removing Default VLAN prevents unintended untagged traffic from leaking across the ring.
configure vlan Data add ports 1,2 tagged ↪ "tagged" = trunk mode. The switch inserts a VLAN 10 tag in every frame leaving these ports, and strips/reads it on arrival. configure vlan Voice add ports 1,2 tagged configure vlan Kh_Mgmt add ports 1,2 tagged ↪ Repeat for all three VLANs. All three must be tagged on every trunk link in the ring. configure vlan Default delete ports 1,2 ↪ Remove Default VLAN from trunk ports — prevents untagged frames from being miscategorized. Best practice on trunk-only ports. show vlan Data ↪ Check output: look for "Tag: *1, *2" — asterisk means the port is active. If you see "!" the port is disabled.

Step 4 — Configure Access Ports

Step 4 of 8 — Edge-1 and Edge-2 only

Assign Untagged Ports for VPCS Clients

Why: VPCS sends plain untagged Ethernet frames — it has no knowledge of 802.1Q. The access port strips/inserts the VLAN tag transparently. The client thinks it's on a normal LAN; the switch handles VLAN membership.
# Run on Edge-1 (ExtremeOs3) AND repeat identically on Edge-2 (ExtremeOs6) configure vlan Data add ports 3 untagged ↪ Port 3 becomes an access port for VLAN 10. Frames arriving untagged on port 3 are placed in the Data VLAN. configure vlan Default delete ports 3 ↪ A port can only be untagged in ONE VLAN at a time. Remove it from Default before adding to Data, or EXOS will reject the command. configure vlan Voice add ports 4 untagged configure vlan Default delete ports 4 configure vlan Kh_Mgmt add ports 5 untagged configure vlan Default delete ports 5 show vlan Data ↪ Expected output: "Untag: *3" — port 3 active and untagged. If you see "Tag: *3" instead, the client will get tagged frames it can't read.

Step 5 — Management IP Addresses

Step 5 of 8 — Each switch gets its own IP

Assign IPs on the Kh_Mgmt VLAN

Why: An IP address on the management VLAN enables SSH access and ping reachability to the switch itself. enable ipforwarding tells the switch to process IP packets destined for its own address.
# Core-1 (repeat for each switch with its own IP) configure vlan Kh_Mgmt ipaddress 10.10.30.1/24 ↪ Assigns the switch's IP address on the management VLAN. Each switch gets a unique IP in the 10.10.30.0/24 subnet. enable ipforwarding vlan Kh_Mgmt ↪ Without this, the switch will not respond to pings or SSH on this VLAN even with an IP assigned. # Core-2: 10.10.30.2 / Edge-1: 10.10.30.3 / Core-4: 10.10.30.4 # Core-5: 10.10.30.5 / Edge-2: 10.10.30.6 ping vr VR-Default 10.10.30.6 ↪ EXOS ping syntax: VR name comes BEFORE the host IP. "from" clause (if used) comes AFTER the host. Any other order returns "Invalid input".

Step 6 — Configure VPCS Clients

Step 6 of 8 — On each VPCS console in EVE-NG

Assign IPs to Virtual PC Clients

Why: VPCS has no IP until assigned. Crucially — the FDB (MAC table) on the switch only learns a device's MAC when that device sends a frame. You must ping or generate traffic after setting the IP, not just set it.
# VLAN10-Client1 — Edge-1 port 3 ip 10.10.10.11/24 ↪ Sets IP and subnet mask. VPCS uses CIDR notation. No gateway needed for same-subnet pings. # VLAN20-Client1 — Edge-1 port 4 ip 10.10.20.11/24 # Kh_Mgmt-Client1 — Edge-1 port 5 ip 10.10.30.11/24 # VLAN10-Client2 — Edge-2 port 3 ip 10.10.10.12/24 # VLAN20-Client2 — Edge-2 port 4 ip 10.10.20.12/24 # Kh_Mgmt-Client2 — Edge-2 port 5 ip 10.10.30.12/24 show ip ↪ Verify the IP is assigned correctly. Also shows the auto-assigned IPv6 link-local — ignore this for now.

Test A — Same-VLAN (expected: success)

Cross-switch ping within the same VLAN

# From VLAN10-Client1 (Edge-1) ping 10.10.10.12 ↪ This ping traverses the full ring: Edge-1 → Core-1 → Core-2 → Edge-2. Proves VLAN 10 is correctly trunked on all four switches in the active path. 84 bytes from 10.10.10.12 icmp_seq=1 ttl=64 time=x.xxx ms ← success

Troubleshooting Case Study  —  March 30, 2026

VPCS Ping Failure — STP Not Bound to VLANs

A real troubleshooting session: VPCS clients could not ping across the ring. The investigation exposed an STP misconfiguration that left the entire ring unprotected from loops.

Symptom ping fails between VLAN10-Client1 and VLAN10-Client2
Root Cause STP bound to no VLANs — ring loop unmitigated
Fix configure stpd s0 add vlan <name> ports 1,2 on all 6 switches

Investigation Timeline

1
Observation

VPCS ping fails — FDB shows MACs on trunk ports, not access ports

After configuring IPs on all VPCS clients, pinging across the ring failed immediately. Running show fdb on Edge-1 revealed that VPCS MACs (OUI 00:50:79) were appearing on ports 1 and 2 (trunk ports), not on ports 3/4/5 (where the clients are physically connected).

ping 10.10.10.12 host (10.10.10.12) not reachable show fdb 00:50:79:66:68:07 Data(0010) d m 2 ← WRONG: port 2 is a trunk port ↪ If a client MAC appears on a trunk port, the switch learned it from the wrong direction. This means the ARP broadcast looped around the ring and arrived back via a trunk — a classic loop signature.
2
Investigation — Port Configuration

Edge-1 port and VLAN config verified correct

Before assuming a loop, we verified the basic port configuration on Edge-1. Both checks passed — the issue was not a misconfigured access port.

show vlan Data Untag: *3 ← port 3 correctly untagged ✓ Tag: *1, *2 ↪ Asterisk (*) = port is active. "Untag" means frames arrive/leave without an 802.1Q tag — correct for VPCS. show ports 3 info 3 Em------e--fMB----- active ← port up, MAC learning enabled ✓ ↪ Flags: E=Enabled, m=MAC Learning on, B=Broadcast flooding on. Port is physically up and ready.
3
Investigation — ARP Tables

One-way ARP: Client2 knows Client1 — Client1 knows nothing

Checking ARP tables on both VPCS clients revealed a critical asymmetry that confirmed the loop hypothesis.

# On VLAN10-Client1 (10.10.10.11) — Edge-1 arp arp table is empty ↪ Client1 sent an ARP broadcast "Who has 10.10.10.12?" — but never received a reply. The reply was lost. # On VLAN10-Client2 (10.10.10.12) — Edge-2 arp 00:50:79:66:68:07 10.10.10.11 expires in 120 seconds ↪ Client2 DID receive Client1's ARP broadcast and recorded Client1's MAC. But Client2's unicast ARP reply never reached Client1.
Key diagnostic pattern: Broadcast succeeds in one direction, unicast reply fails on the return. This is the textbook signature of a Layer 2 loop without STP — broadcasts flood everywhere (including looping), but unicast replies follow FDB entries that are flapping due to the loop.
4
Root Cause Analysis

STP enabled but bound to no VLANs — ring completely unprotected

Running show stpd detail on Edge-2 revealed the core problem. STP was enabled, but the STPD instance had zero ports and zero participating VLANs.

# Edge-2 — BEFORE fix show stpd detail Stpd: s0 Stp: ENABLED Number of Ports: 0 Ports: (none) Participating Vlans: (none) Auto-bind Vlans: Default Designated root: 80:00:50:01:00:06:00:00 ← Edge-2 thinks it IS the root ↪ "Auto-bind Vlans: Default" means only the Default VLAN is automatically protected by STP. But Default VLAN was removed from all trunk ports in Step 3 — so STP is protecting nothing. Data, Voice, and Kh_Mgmt VLANs are all looping freely.
Root Cause: EXOS requires VLANs to be explicitly bound to an STPD instance using configure stpd s0 add vlan <name> ports 1,2. Unlike Cisco IOS (which auto-runs PVST+ per VLAN), EXOS does nothing automatically for non-Default VLANs. The lab had completed Steps 1–6 without this step, leaving the ring in a continuous loop for all three user VLANs.

Why This Happens — The Loop Explained

Without STP blocking one ring link, every broadcast frame circulates indefinitely. Here is the exact sequence of events that caused the symptom:

#EventResult
1Client1 sends ARP broadcast: "Who has 10.10.10.12?"Frame enters Edge-1 port 3, floods into Data VLAN
2Edge-1 floods broadcast out ports 1 and 2 (both trunk paths)Frame enters ring in both directions simultaneously
3Frame loops: Core-1 → Core-2 → Edge-2 and Core-5 → Core-4 → Edge-2Client2 receives the broadcast — learns Client1 MAC ✓
4Frame continues looping around the ring endlesslyEdge-1 sees Client1's MAC arriving on port 2 (trunk) — FDB records wrong port
5Client2 sends unicast ARP reply to Client1's MACEdge-2 looks up MAC in FDB — finds it on port 2 (toward Core-4)
6ARP reply sent toward Core-4 → Core-5 → Edge-1Edge-1 FDB may now show Client1 on port 1 instead of port 2 — reply loops or is dropped
7Client1 never receives the replyARP fails → ping fails → "host not reachable"

The Fix — Bind VLANs to STP on All 6 Switches

Important: This command must be run on every switch in the ring. One missed switch means that switch still has no loop protection, and the loop persists.
Run on: Core-1, Core-2, Core-4, Core-5, Edge-1, Edge-2

Add VLANs and Trunk Ports to STPD

configure stpd s0 add vlan Data ports 1,2 ↪ Binds ports 1 and 2 of the Data VLAN to the STPD instance s0. STP will now send/receive BPDUs on these ports for Data VLAN and can block one of them if needed. configure stpd s0 add vlan Voice ports 1,2 configure stpd s0 add vlan Kh_Mgmt ports 1,2 ↪ Repeat for all three VLANs. All must be protected — a missing VLAN means that VLAN still loops. save configuration ↪ Persist immediately. If EVE-NG resets before you save, the STP binding is lost and the loop returns.

After running on all 6 switches, wait ~30 seconds for STP to converge: switches exchange BPDUs, elect a root bridge, and transition ports from Learning → Forwarding (or Blocking).

Verification — STP Converged & Ping Restored

# On any switch — Core-4 is the one to check (has the blocked port) show stpd detail Ports: 1,2 Participating Vlans: Data, Kh_Mgmt, Voice Data: 1(F), 2(B) ↪ Port 2 BLOCKING = loop broken. This is correct and expected. BridgeID: 80:00:50:01:00:04 CIST Root: 80:00:50:01:00:01 ↪ Core-1 elected root bridge (lowest BridgeID in the ring). # Clear stale FDB entries from when the loop was running clear fdb ↪ FDB entries learned during the loop point to wrong ports. Clearing forces switches to re-learn MAC locations correctly via fresh traffic. # From VLAN10-Client1 — after clearing FDB ping 10.10.10.12 84 bytes from 10.10.10.12 icmp_seq=1 ttl=64 time=x.xxx ms ← RESOLVED ✓ # Verify MAC is now on the correct port show fdb 00:50:79:66:68:0a 00:50:79:66:68:0a Data(0010) d m 3 ← correct: local access port ✓

Key Concepts

IEEE 802.1D

Spanning Tree Protocol

Prevents L2 loops by electing a root bridge (lowest BridgeID) and blocking redundant ring ports. Port states: Blocking → Listening → Learning → Forwarding. Convergence takes 30–50 seconds with classic STP.

→ IEEE 802.1D-2004
EXOS

STPD Binding in EXOS

Unlike Cisco PVST+ (which auto-runs per VLAN), EXOS requires explicit binding: configure stpd s0 add vlan <name> ports 1,2. Without this, STP shows "ENABLED" but does nothing for that VLAN — the ring loops silently.

RFC 826

ARP — Address Resolution Protocol

ARP requests are broadcasts (flood to all ports in VLAN). ARP replies are unicast. One-way ARP — where one host knows the other but not vice versa — is the diagnostic signature of a unicast routing failure on top of a loop.

→ RFC 826
MAC Flapping

FDB Instability in a Loop

In a looping ring, the same MAC arrives from multiple ports alternately. The FDB records whichever port was last seen — this is called MAC flapping. Unicast frames follow the flapping FDB entry and get routed to the wrong port, causing intermittent or total connectivity failure.

IEEE 802.1s

MSTP — Multiple Spanning Tree

EXOS defaults to MSTP operational mode. In this lab, all VLANs share a single CIST (Common Internal Spanning Tree) instance s0. MSTP allows different VLAN groups to use separate STP instances for independent path control.

→ IEEE 802.1Q-2022
Root Bridge Election

Why Core-1 Won

STP elects the switch with the lowest BridgeID as root. BridgeID = Priority (default 32768) + MAC address. All 6 switches had equal priority — Core-1 won because 50:01:00:01 is the numerically lowest MAC in the ring.

Deep Dive  —  Layer 2 Loop Behaviour

Why Some Pings Worked Without STP

Switch-to-switch management pings succeeded. VPCS client pings across the ring failed. Both ran on the same loop-ridden network — here is exactly why they behaved differently.

Topology 6-switch ring, no STP active
Observed March 30, 2026 during VPCS validation

The Question

After completing the lab configuration but before binding VLANs to STP, the following behaviour was observed:

TestResultSurprising?
Core-1 ping Core-2 (adjacent, 1 hop)SUCCESSNo
Edge-1 ping Edge-2 (non-adjacent, 3 hops)UNRELIABLEYes
VLAN10-Client1 ping VLAN10-Client2FAILEDYes — same VLAN, same config
VLAN20-Client1 ping VLAN20-Client2FAILEDYes

The instinct is: "if the network is looping, nothing should work." But that is not quite right. A loop does not break all traffic equally — it breaks traffic that depends on a stable, multi-hop unicast return path.

Why Adjacent Switch Pings Worked

Every adjacent switch pair in the ring is directly connected by a single cable. Core-1 and Core-2 share a direct link on port 2. When Core-1 pings Core-2, the entire exchange happens on that one link — the loop is irrelevant.

Step-by-step ARP + Ping flow (Core-1 → Core-2)

Step 1 — Core-1 sends ARP broadcast: "Who has 10.10.30.2?" Frame exits Core-1 port 2 (direct link to Core-2) Frame also exits Core-1 port 1 (into the ring toward Edge-1) Step 2 — Core-2 receives ARP on port 2 (direct link) Core-2 records: Core-1 MAC is on port 2 → FDB entry stable ✓ Core-2 sends unicast ARP reply → out port 2 → direct to Core-1 Step 3 — Core-1 receives ARP reply on port 2 (direct link) ARP resolves. Ping proceeds. Works every time. ✓ Meanwhile — the broadcast is looping around the ring Core-1 port 1 → Edge-1 → Core-5 → Core-4 → Edge-2 → Core-2 port 1 Core-2 receives Core-1 MAC again — this time on port 1 (ring side) FDB briefly flips: Core-1 MAC now on port 1 instead of port 2 ← noise But the ping reply was already sent and received. No impact.
Key insight: The unicast ARP reply from Core-2 travels on the direct link back to Core-1 — one hop, no ambiguity, arrives before the loop noise can corrupt the FDB entry. The ping completes before the loop has time to cause damage.

Think of it like asking someone next to you a question — they answer you directly before the echo of your voice comes back from the other end of the room. The echo exists, but it arrives too late to matter.

Why VPCS Client Pings Failed

Edge-1 and Edge-2 are not directly connected. Traffic between VLAN10-Client1 (Edge-1) and VLAN10-Client2 (Edge-2) must traverse 3 switches. With no STP, the ARP broadcast floods in both directions around the ring simultaneously.

What happens without STP

Client1 sends ARP broadcast: "Who has 10.10.10.12?" Edge-1 floods out port 1 AND port 2 simultaneously Path A (clockwise): Edge-1 → Core-1 → Core-2 → Edge-2 Arrives at Edge-2 port 1 → Edge-2 records: Client1 MAC on port 1 Path B (counter-clockwise): Edge-1 → Core-5 → Core-4 → Edge-2 Arrives at Edge-2 port 2 → Edge-2 records: Client1 MAC on port 2 ← OVERWRITES Both copies continue looping — arriving repeatedly from alternating directions Edge-2 FDB: Client1 MAC flaps between port 1 and port 2 every few milliseconds Client2 receives broadcast, sends unicast ARP reply to Client1 MAC Edge-2 checks FDB for Client1 MAC → points to port 2 (Core-4 side) at this instant Reply goes: Edge-2 → Core-4 → Core-5 → Edge-1 Edge-1 receives reply on port 2... but its FDB has Client1 (its own VPC!) on port 2 Edge-1 may forward the reply back OUT port 2 into the ring → reply loops Client1 never receives the ARP reply → ping fails → "host not reachable"
Root mechanism: The loop causes MAC address flapping — the same source MAC arriving on alternating ports. Each arrival overwrites the FDB entry. When the unicast reply is sent, the FDB entry points in the wrong direction. The reply either loops endlessly or is forwarded away from the destination.

ARP Flow — With vs Without STP

FDB Flapping — The Core Mechanism

The MAC address table (FDB) is updated every time a frame arrives. In a loop, the same source MAC arrives from multiple ports in rapid succession. The switch keeps overwriting its own table:

TimeEventFDB Entry for Client1 MAC
t=0msClient1 ARP broadcast exits Edge-1 on ports 1 and 2Edge-2: not yet learned
t=1msPath A arrives at Edge-2 port 1 firstEdge-2: Client1 → port 1
t=2msPath B arrives at Edge-2 port 2 (slightly longer path)Edge-2: Client1 → port 2 (overwritten)
t=3msLooped copy of Path A arrives again at Edge-2 port 1Edge-2: Client1 → port 1 (overwritten again)
t=4msClient2 sends unicast reply — FDB says port 2 at this instantReply exits Edge-2 port 2 → goes the wrong way
t=5msLooped copy arrives again — FDB flips backContinuous instability

This is called MAC address flapping. Enterprise switches detect it and generate log alerts. In a ring with no STP, every broadcast causes a flapping storm that lasts until the frame is consumed (it never is — Ethernet has no TTL).

No TTL in Ethernet: Unlike IP packets (which have a TTL field that decrements at each hop and expires), Ethernet frames have no hop counter. A looping Ethernet broadcast circulates at wire speed forever — or until the switch is rebooted, the cable is unplugged, or STP blocks the loop.

The Rule — When a Loop Breaks Traffic

ScenarioWorks without STP?Reason
Adjacent switch ↔ switch (1 hop, direct cable) Usually yes Unicast reply takes the direct link. Arrives before loop noise corrupts the FDB entry. Exchange completes in microseconds.
Non-adjacent switch ↔ switch (2+ hops) Unreliable Reply must choose a path. FDB entry for sender may have been overwritten by loop traffic before reply arrives. Intermittent success possible.
VPCS client ↔ VPCS client (cross-ring, 4 hops) No ARP broadcast floods both ring directions simultaneously. Edge switch FDB flaps before reply can be correctly routed back. Client never gets reply.
Broadcast storm threshold Eventually breaks everything Unmitigated loops compound. Each loop adds another copy of every broadcast. CPU overloads. All traffic eventually fails. In EVE-NG the VM limits slow this effect.

Why EVE-NG masked the worst effects

In a real physical network, an unmitigated ring loop causes a broadcast storm within seconds — link utilisation hits 100%, switch CPUs max out, and the entire network goes dark. In EVE-NG, virtual links have much lower throughput and the hypervisor throttles traffic, so the loop causes FDB flapping and selective failures rather than an immediate full outage. This is why some pings appeared to work — the storm was slow enough to allow short exchanges on direct links to complete.

Key Concepts

MAC Flapping

FDB Instability Under a Loop

When the same MAC address is seen on two different ports in quick succession, the switch overwrites its FDB entry each time. The result is that unicast frames destined for that MAC are forwarded alternately to the wrong port. Enterprise switches log MAC flapping as a warning event.

No TTL in Ethernet

Why Loops Are Permanent

IP packets carry a TTL field that decrements at each router hop and causes the packet to be discarded at zero. Ethernet frames have no equivalent. A looping frame circulates at full wire speed indefinitely — only STP, cable removal, or reboot can stop it.

→ IEEE 802.1D — Bridge Loops
IEEE 802.1D

Why STP Was Invented

STP (1985, Radia Perlman) was designed specifically to solve the loop problem in Ethernet bridged networks. It provides a distributed algorithm for switches to collectively agree on a loop-free tree topology — without manual configuration of which link to block.

→ IEEE 802.1D-2004
RFC 826

ARP — The Loop's Victim

ARP relies on broadcasting a request and receiving a single unicast reply. This protocol has zero tolerance for the MAC instability caused by a loop — if the FDB entry for the requester flaps between the request and the reply, the reply is misrouted and the entire name resolution fails.

→ RFC 826
Broadcast Storm

The Compounding Effect

Each switch floods a broadcast out all ports in a VLAN. In a loop, every copy of the broadcast is flooded again, creating exponentially more copies. A single broadcast becomes thousands within milliseconds. This is a broadcast storm — it saturates links and overwhelms switch CPUs.

EVE-NG Effect

Why the Lab Didn't Storm Immediately

EVE-NG virtualises links at low speeds with CPU-based forwarding. The hypervisor throttles frame rates, so a loop that would storm a physical network in <1 second instead causes slow FDB flapping and selective failures. This is useful for learning but can mask the true severity of a loop.

IEEE 802.1D-2004 Media Access Control Bridges — Spanning Tree Section 17: Loop detection and STP algorithm. The original motivation for STP was exactly the broadcast storm scenario described in this tab. Radia Perlman's spanning tree algorithm is described in full.
IEEE 802.1Q-2022 Virtual Bridged LANs Section on MSTP (Multiple Spanning Tree). Explains how modern switches handle per-VLAN loop protection and why explicit VLAN binding to STP instances is necessary in MSTP mode.
RFC 826 An Ethernet Address Resolution Protocol The ARP specification. Read Section 3 (Packet Format) to understand why ARP replies are unicast and why MAC instability directly causes ARP failure.
Perlman, R. (1985) An Algorithm for Distributed Computation of a Spanning Tree in an Extended LAN The original academic paper that introduced spanning tree to networking. Radia Perlman ("Mother of the Internet") describes the loop problem and its solution from first principles. Accessible and historically significant.
Cisco Press CCIE Routing and Switching — Chapter: Spanning Tree Vendor-neutral deep dive into STP, RSTP, and MSTP mechanics including MAC flapping, TCN (Topology Change Notification), and how different vendors implement BPDU handling.

Investigation Timeline

  • Confirmed VPCS IPs set correctly — show ip on each VPC passed
  • Ping attempted, failed — "host (10.10.10.12) not reachable"
  • Port config verified OK — Edge-1 port 3 untagged in Data VLAN, port active
  • ARP asymmetry found — Client2 had Client1 in ARP table, Client1 ARP was empty. Classic loop signature.
  • FDB on Edge-2 — Client1 MAC learned on port 2 (trunk, wrong direction)
  • Root cause — show stpd detail on Edge-2: Ports: (none), Participating VLANs: (none). EXOS requires explicit VLAN binding — unlike Cisco PVST+.
  • Fix applied on all 6 switches — configure stpd s0 add vlan Data/Voice/Kh_Mgmt ports 1,2
  • STP converged (~30s) — Core-4 port 2 in BLK state, all others Forwarding
  • Ping restored — clear fdb then ping succeeded ✓

Fix Commands

configure stpd s0 add vlan Data ports 1,2 configure stpd s0 add vlan Voice ports 1,2 configure stpd s0 add vlan Kh_Mgmt ports 1,2 save configuration # ~30s later: Core-4 port 2 → BLK ✓ then: clear fdb

Key Lesson

Unlike Cisco IOS (PVST+ runs automatically per VLAN), EXOS requires explicit VLAN binding to a Spanning Tree Domain. STP showing "ENABLED" does not mean it is protecting your VLANs. Always verify show stpd detail shows participating VLANs and ports after config.
