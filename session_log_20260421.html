Key Concepts Worked Through

Claude AI — Claude_InterVLAN_Apr21.docx — ARP + MAC rewrite
"It sends an ARP request, and the gateway responds saying it has the MAC address for the destination — 'send your packet to me.' When it reaches L3 at the gateway, it strips the VLAN 40 L2 header and puts in the VLAN 70 L2 header."
Claude AI — Claude_InterVLAN_Apr21.docx — L2 packet wrapping
"The VLAN 40 client wraps the L2 header around the IP packet. Source and destination IPs never change. The L2 header provides the next-hop address. The client doesn't need to know about VLANs."
NotebookLLM — NoteBook_LLM_InterVLAN_Apr21.docx — Option A Breakthrough
"In our earlier labs, we were using 'Option B,' where we were basically at the mercy of the home modem for everything. In this new plan, we're moving to Option A, where Switch 1 (Primary) handles DHCP internally for all three VLANs. This is huge because it means the modem doesn't need to understand our internal VLAN structure anymore; it just needs to know where to send return traffic."
NotebookLLM — NoteBook_LLM_InterVLAN_Apr21.docx — The Golden Rule
"The Golden Rule: We can't forget enable ipforwarding on every single VLAN on Switch 1, or our clients will be stranded with an IP but no way to talk to the next hop."
NotebookLLM — NoteBook_LLM_InterVLAN_Apr21.docx — The Modem Handoff
"Since our Quantum Fiber modem only knows about the 192.168.0.x network, we have to tell it manually that to find 10.10.0.0, 10.20.0.0, or 10.30.0.0, it has to talk to Switch 1's IP address."

SW1 — Primary L3

SW1 Complete Config — 192.168.0.28 — All SVIs + DHCP + Routing

NotebookLLM — NoteBookLLM_KarlLab_Apr21.docx v1 — Port 10 DHCP
"EXOS is extremely literal about where it listens for DHCP requests. For Switch 1 to 'hear' the request coming from Switch 2, the trunk port itself must be enabled for DHCP. On Switch 1, the command must include the trunk port: enable dhcp ports 5,10 vlan VLAN_10. If we only enabled it on Port 5, Switch 1 would only provide IPs to devices plugged directly into itself, and the UE on Switch 2 would stay in a 'Searching...' state."
NotebookLLM — NoteBookLLM_KarlLab_Apr21.docx v1 — The Karl Rule
"Don't forget enable ipforwarding vlan Default on Switch 1. If you miss that one, your clients will get an IP address and look perfectly fine, but they'll be 'ghosted' by the internet because the return traffic won't know how to cross back from the modem's subnet to yours."
NotebookLLM — NoteBookLLM_KarlLab_Apr21.docx v1 — Native VLAN on Port 3
"On Port 3, keep the Default VLAN (1) untagged for the AP3000 to 'call home' and get its management IP, while your SSIDs for the Corporate and Guest networks come in tagged."

SW2 — Secondary L2

SW2 Complete Config — 192.168.0.11 — Pure L2, No SVIs

NotebookLLM — NoteBookLLM_KarlLab_Apr21_v2.docx — VLAN 1 Triangle Risk
"If you also add VLAN 1 to the Port 10 Trunk, you create a physical triangle: SW1 ↔ Modem ↔ SW2 ↔ SW1. Without Spanning Tree (STP) to block one of those paths, a single broadcast frame would loop around that triangle indefinitely — that's the 'ping-ponging.' In EXOS, the Default VLAN is usually assigned to STPD s0 (Spanning Tree Domain 0) by default to prevent this exact nightmare."
NotebookLLM — NoteBookLLM_KarlLab_Apr21_v2.docx — EXOS delete ports syntax
"In EXOS, you don't 'no switchport trunk allowed vlan 1' like in Cisco; you explicitly tell the VLAN to 'delete' that port from its membership list: configure vlan Default delete ports 10"

QoS + Isolation

Guest Isolation + QoS Priority — Strategy

NotebookLLM — NoteBookLLM_FinalItems_April21.docx — Two-pronged strategy
"Guest isolation is tricky because traffic flows in two different directions: sideways (Guest to Guest) and upward (Guest to Corporate). North-South (Guest → Corporate): This requires a Policy-Based ACL on SW1 because it's a routing decision. East-West (Guest → Guest): We confirmed this must be done in PlatformOne (XIQ) using the Client Isolation toggle on the SSID, because that traffic never leaves the AP's radio to hit the switch."
NotebookLLM — NoteBookLLM_FinalItems_April21.docx — CLI vs XIQ division
"You were using the CLI for the 'Wired Side' (VLAN 10) because that's where the granular control is needed, while XIQ is best for the 'Wireless Side' (SSIDs). If you had done the Guest Isolation in XIQ, you wouldn't have hit that 'Missing keyword entry' error because the cloud would have written the code for you!"
Claude AI — Claude_NewLab_Karl_Dialogue_Apr21_v3.docx — L2 vs L3 insight
"Both clients are on VLAN 30 (same subnet). So SW1 doesn't make an L3 routing decision here — it just switches the traffic at L2 within VLAN 30. That's why an ACL at L3 on VLAN 30 won't catch it — the traffic never crosses a VLAN boundary, so the routing engine never sees it."

Lessons

Key Lessons — All Sources — April 21, 2026

Lesson 1 — The Karl Rule (most critical)

⚠ enable ipforwarding vlan Default — ALWAYS add to SW1 script
Without this, clients get DHCP addresses and can ping the gateway but have NO internet. Traffic can't cross from data VLANs to the Default VLAN (modem path). This was NOT in the original config script despite NotebookLLM flagging it explicitly in v2.
NotebookLLM — NoteBookLLM_KarlLab_Apr21_v2.docx
"The 'Karl Rule': Don't forget enable ipforwarding vlan Default on Switch 1. If you miss that one, your clients will get an IP address and look perfectly fine, but they'll be 'ghosted' by the internet because the return traffic won't know how to cross back from the modem's subnet to yours."

Lesson 2 — DHCP must include port 10 (trunk)

⚠ enable dhcp ports 5,10 vlan VLAN_10 — not just port 5
If port 10 is not in the enable dhcp command, SW1 ignores DHCP requests arriving from SW2. Clients on SW2 get 169.254.x.x self-assigned addresses.

Lesson 3 — VLAN 1 Triangle Loop Risk

⚠ configure vlan Default delete ports 10 — run on both switches
Both switches connect to modem via VLAN 1 on port 1. If VLAN 1 also exists on port 10 (trunk), a loop forms: SW1 ↔ Modem ↔ SW2 ↔ SW1. Broadcast storm without STP. Always remove Default VLAN from trunk port.
NotebookLLM — NoteBookLLM_KarlLab_Apr21_v2.docx
"In EXOS, you don't 'no switchport trunk allowed vlan 1' like in Cisco; you explicitly tell the VLAN to 'delete' that port from its membership list: configure vlan Default delete ports 10"

Lesson 4 — AP Onboarding needs Default VLAN untagged on port 3

AP3000 boots dumb — sends untagged DHCP request
AP has no config at boot. Sends untagged DHCP → switch assigns to VLAN 1 → modem gives 192.x.x.x → AP calls home to XIQ. Once onboarded, AP carries VLAN 20 and 30 tagged.
Claude AI — Claude_NewLab_Karl_Dialogue_Apr21_v3.docx — AP boot flow
"AP boots → untagged DHCP request → port 3 (VLAN 1 untagged) → port 1 uplink → modem → DHCP 192.x.x.x → AP onboards to XIQ. And once onboarded, the AP carries: VLAN 20 tagged → Corporate SSID traffic · VLAN 30 tagged → Guest SSID traffic"

Lesson 5 — ACL syntax: EXOS uses policy files, not inline create access-list

⚠ create access-list "ip-source-address..." fails in EXOS 33.x
The attribute "ip-source-address" is not valid. EXOS requires policy files (edit policy NAME) using source-address / destination-address keywords. For this lab, XIQ Firewall Rules avoided the CLI syntax entirely.

Lesson 6 — L2 vs L3 isolation: ACLs can't stop intra-VLAN guest-to-guest traffic

Claude AI — Claude_NewLab_Karl_Dialogue_Apr21_v3.docx
"Both clients are on VLAN 30 (same subnet). So SW1 doesn't make an L3 routing decision here — it just switches the traffic at L2 within VLAN 30. That's why an ACL at L3 on VLAN 30 won't catch it — the traffic never crosses a VLAN boundary, so the routing engine never sees it. The feature you're looking for is called Client Isolation — it's configured on the AP/SSID in PlatformOne."

Lesson 7 — ipforwarding on SW2 is correctly DISABLED

Claude AI — Claude_FullDay_Apr21.docx — SW2 ipforwarding check
"ipforwarding on SW2 is of course disabled because there is NO L3 forwarding... all traffic go to SW1. Exactly right — and good self-correction. SW2 is pure L2, so IPv4 Forwarding: Disabled on SW2 is correct and expected. No fix needed there."

Lesson 8 — LLDP for trunk verification

NotebookLLM — NoteBookLLM_KarlLab_Apr21_v2.docx — LLDP as verification tool
"LLDP is the carrier for Fabric Attach (FA). When you plug the AP3000 into Port 3, it uses LLDP to 'advertise' its presence. In our Port 10 Trunk test, if you run show lldp neighbors detailed, you can actually see the Management IP of the other switch, which is a great way to verify Layer 2 connectivity even if pings aren't working yet."
