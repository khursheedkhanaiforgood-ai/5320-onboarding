The Trigger

Refined Definition: Wired Interface Arrival in the SW1 Broadcast Domain

Yesterday's framing of the trigger was vague — "MacBook joined Wi-Fi" — and that turned out to be wrong. The deterministic trigger is far more specific:

Move the work MacBook's Ethernet (en8) from QF Modem (DMZ side) to SW1 Port 5 (lab side), while the same MacBook's en0 is associated to Corporate_Wireless via AP1.

The moment Port 5 link comes up and the work MacBook starts DHCP'ing on VLAN 10, iPhone DHCP fails. Disconnecting Port 5 (or simply waiting 2–4 minutes) recovers everything. This is the deterministic, reproducible trigger.

Why Did TK/GTK Corrupt?

Socratic Round — The Trigger Layer Below Yesterday's Mechanism Layer

Yesterday we identified mechanism = (b) WPA2 key state. Today we asked the deeper question: what corrupted it in the first place? Four hypotheses:

#HypothesisWhat it claims
(A)Stale per-MAC key slot in AP crypto engineiPhone's PTK still bound to MAC slot; MacBook joins, double-binds, silent fail
(B)GTK rotation triggered by perceived duplicate clientAP defensive deauth + GTK rotation; group key handshake to other clients wedges
(C)macOS dual-interface state changeWired + Wi-Fi both active → macOS aborts/restarts EAPOL → AP left with half-installed TK
(D)All three layered (C → A → B cascade)C sets the stage; A is the immediate cause for the trigger MAC; B propagates to lagging clients

User committed to (D). Today's pcap then narrowed it further: in this specific repro, the LAA collision condition for (A) is absent (work MBP and personal MBP both use UAA hardware MACs; only iPhone uses LAA). So today is a clean test of C → B, with (A) ruled out by MAC distinctness.

The Smoking Gun

3-Client Failure Fingerprint — Proof of (B)

This is the strongest evidence today produced. Look at which clients failed:

ClientFailed?Recovery time
Work MBP (trigger device, 84:2f:57:94:bd:cf)NO — stayed connected throughoutn/a
Personal MBP (74:a6:cd:8b:19:60)YEST+256s (4.3 min)
iPhone (62:75:85:f6:4e:3e)YEST+163s (2.7 min)

Two-of-three clients failed. The immune client is the trigger device. This asymmetry is unique to GTK rotation propagation:

  • (A) per-client TK collision — would only break one client → doesn't match (two broke)
  • (B) GTK rotation propagation — trigger device is the active client at the moment of GTK rotation, gets GTK_v2 first; lagging clients miss the M1/M2 group-key-handshake exchange → exactly this fingerprint ✓
  • (C) dual-interface race — sets up the conditions, doesn't directly cause silence
  • (D) C → B cascade — confirmed by today's data

The trigger device gets GTK_v2 successfully because it's the "active" client (it's just emitted EAPOL traffic). Lagging clients (iPhone, personal MBP) miss the GTK_v2 push due to M2-ACK race or dropped frames. AP starts encrypting broadcasts with GTK_v2 → lagging clients can't decrypt → iPhone's DHCP renewal fails → falls to APIPA.

Self-Recovery

The Yesterday vs. Today Difference

Yesterday, after observing the failure, you rebooted — which fixed it but consumed the evidence. Today, you let it ride. It self-recovered in 2–4 minutes without any intervention. en8 stayed plugged into Port 5 the entire time.

This means yesterday's reboot may have been unnecessary — the AP has built-in recovery mechanisms:

  • Periodic GTK refresh timer — AP rotates GTK on a schedule (default 1–24 hours, faster on disassoc events)
  • Client re-association on lease expiry — iPhone tried to renew, failed, re-associated, fresh PTK and GTK
  • AP's auto-retry GTK push — resends GTK to lagging clients after timeout

For future incidents: wait 5 minutes before rebooting. If the cascade is GTK-rotation-based (which today's repro confirms), it self-heals.

Lessons

Operational Lessons

  1. Never enable streaming kdebug on the only SSH path. The _kdebug wifi-driver msglevel wsec command flooded the SSH buffer on a busy WPA3 BSSID; SSH dropped with "Can't assign requested address". For future kdebug use, access via XIQ Device 360 → Tools → SSH/CLI (cloud-relay path is more resilient).
  2. The cascade self-recovers in 2–4 minutes. Yesterday's reboot may have been unnecessary. Wait it out before destroying evidence.
  3. The trigger device is immune to its own cascade. Useful diagnostic asymmetry — if 2 of N clients fail and the immune one is the most recently active, suspect GTK desync.
  4. The trigger isn't "joining Wi-Fi" — it's specifically a wired interface arriving in the same SW1 broadcast domain, while the same machine's Wi-Fi is associated.
  5. Capture en0 from a wired-already client to see the cascade externally; gratuitous ARPs are the recovery timestamp for each affected client.
  6. ADSP sensor mode blocks capture interface wifi*. Capture on Eth0 instead — you see post-decrypt traffic from all clients, which is actually MORE useful for DHCP forensics.
