AP3000 Onboarding Guide

Complete session record: 5320-16P-2MXT-2X switch configuration via CLI → VLAN design → PoE → IP discovery → ExtremeCloud IQ adoption. Two AP3000s deployed on ports 3 and 5.

Context

Overview & Context

Yesterday (March 25, 2026) — The 5320 was onboarded using an AI-assisted agent: factory reset from Fabric Engine 9.2 to EXOS SwitchEngine 33.5.2 via ZTP+. During that session, four VLANs were created (VLAN_0010 through VLAN_0040) and the switch management IP was set to 192.168.0.24/24 on the Default VLAN. The switch is connected to a home router at 192.168.0.1 which provides DHCP for the 192.168.0.0/24 subnet.

Today (March 26, 2026) — Two Extreme AP3000 access points are being onboarded. The goal: cable them to ports 3 and 5, power them via PoE, rename the VLANs descriptively, configure the port VLAN membership (tagged/untagged), verify PoE delivery, discover the AP IP addresses, and adopt both APs into ExtremeCloud IQ with a Network Policy mapping three SSIDs to VLANs 20/30/40.

VLAN Design — Four VLANs are planned: Default (1) for switch management and AP provisioning, Mgmt (10) reserved by EXOS for the OOB management port, Corporate (20) for corporate WiFi clients, Guest (30) for guest WiFi, and IoT (40) for IoT devices. Ports 3 and 5 carry the Default VLAN untagged (so APs get DHCP) plus Corporate/Guest/IoT tagged (for multi-SSID after XIQ adoption).

Session Flowchart — Major Milestones

Phase 1

SSH to the Switch

The session began by SSHing to the primary 5320 at 192.168.0.24. Both switches immediately rejected the connection with a host key negotiation error.

Error — SSH Host Key Negotiation Failure
Last login: Wed Mar 25 15:46:00 on ttys008
khukhan@EXT-L63PXM76R6 ~ % ssh admin@192.168.0.24
Unable to negotiate with 192.168.0.24 port 22: no matching host key type found. Their offer: ssh-rsa
khukhan@EXT-L63PXM76R6 ~ % ssh admin@192.168.0.28
Unable to negotiate with 192.168.0.28 port 22: no matching host key type found. Their offer: ssh-rsa
Why This Happens

macOS ships with a modern OpenSSH client that removed ssh-rsa from the default list of accepted host key algorithms for security reasons (SHA-1 based). EXOS SwitchEngine 33.5.2 only offers ssh-rsa. The fix: explicitly re-enable it with command-line flags.

Fix — Add HostKeyAlgorithms Flags

ssh -o HostKeyAlgorithms=+ssh-rsa -o PubkeyAcceptedAlgorithms=+ssh-rsa admin@192.168.0.24

Successful Connection

khukhan@EXT-L63PXM76R6 ~ % ssh -o HostKeyAlgorithms=+ssh-rsa -o PubkeyAcceptedAlgorithms=+ssh-rsa admin@192.168.0.24

The Mgmt Trap

Renaming VLAN_0010 to "Mgmt" was accepted by EXOS but locked the VLAN to the physical OOB management port. Subsequent attempts to use it failed:

* .14 # configure vlan Mgmt add ports 3 untagged
Error: Cannot configure VLAN Mgmt!

* .17 # configure vlan Mgmt name APMgmt
Error: Cannot configure VLAN Mgmt

* .19 # delete vlan Mgmt
Error: VLAN "Mgmt" cannot be deleted from the system!

The show vlan detail output revealed the reason: Untag: Mgmt-port on is down — the 5320 has a dedicated physical OOB management port and "Mgmt" is permanently bound to it.

Fixes Applied

* 5320-16P-2MXT-2X-SwitchEngine.20 # configure vlan Corportate name Corporate
← Fixed the typo
Lesson: EXOS Reserved VLAN Names

In EXOS SwitchEngine, Management and Mgmt are reserved keywords tied to the physical OOB management port. They cannot be used as names for data VLANs. Use descriptive alternatives: APMgmt, WifiMgmt, AccessMgmt.

The "Mgmt" VLAN (VID 10) in this deployment is permanently locked — it cannot be deleted, renamed, or have data ports added. AP management uses the Default VLAN for provisioning instead.

Phase 4

Port Configuration — Tagged vs Untagged

Concept: Untagged vs Tagged

Untagged — The port strips the 802.1Q VLAN tag before sending frames out. The connected device receives plain Ethernet and needs no VLAN awareness. Used for the native/provisioning VLAN. Only one untagged VLAN is allowed per port.

Tagged — Frames carry the 802.1Q VLAN tag intact. The connected device (the AP) must understand VLANs. Used for data VLANs that APs map to SSIDs. Multiple tagged VLANs per port are allowed.

Why Default VLAN untagged for AP provisioning? — The home router provides DHCP on 192.168.0.0/24 and connects to port 1 untagged on Default VLAN. APs on Default VLAN (untagged) get a 192.168.0.x IP from the home router, reach the internet, and connect to XIQ cloud for adoption. After adoption, XIQ pushes Corporate/Guest/IoT SSID config which uses the tagged VLANs.

PoE Class 4 Explained

Class 4 = IEEE 802.3at (PoE+), up to 30W allocated per port. Initial draw of 2–2.5W is normal during early boot sequence. Full AP3000 power consumption under load is typically 12–15W. The 5320's 180W budget supports up to ~12 simultaneous Class 4 APs at full load.

What nmap -sn Does

Network Mapper ping scan. Sends multiple probe types to every IP in the /24 range (256 addresses): ICMP echo request, TCP SYN to port 443, TCP ACK to port 80, ICMP timestamp request. Reports all hosts responding to any probe. Much more reliable than a simple broadcast ping alone.

Network Policy — Creation Steps

  1. XIQ → Configure → Network Policies → Add Network Policy
  2. Name: 5320-Home-Policy → Next
  3. Add Wireless Network: Corporate · WPA2-PSK · VLAN 20 → Save
  4. Add Wireless Network: Guest · WPA2-PSK · VLAN 30 → Save
  5. Add Wireless Network: IoT · WPA2-PSK · VLAN 40 → Save
  6. Monitor → Devices → Select both APs → Actions → Assign Network Policy → 5320-Home-Policy
  7. Click Update Devices → APs reboot briefly → SSIDs broadcast
