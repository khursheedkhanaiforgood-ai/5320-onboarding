QoS Egress Validation: A Three-Layer Architecture Bug Confirmed

Mission: validate the boss's claim that "all traffic goes to best-effort despite XIQ priority policy." Result: bug fully confirmed and decomposed into three independent layers, two of which are architectural. The Apr 21 lab work mapped VLANs to queues correctly, but downstream YouTube traffic still lands in QP1 because the EXOS VLAN-to-qosprofile mapping is ingress-side only and the L3-routed return path keeps its WAN-side queue assignment.

The Mission

Validate the QoS Bug Claim

Boss observation:

"In Network Policy when QoS is enabled for particular traffic (Corp higher than Guest), apparently the tagging of the AP→EXOS traffic on the egress is missing the specific tags or priority assigned by XIQ. All traffic goes to best effort in the egress."

Apr 21 lab work configured: VLAN 20 (Corp) → QP6, VLAN 30 (Guest) → QP1, with XIQ User Profile scheduling weights of 100 (Corp) and 10 (Guest). Test goal: prove or disprove that priority is enforced end-to-end.

The Stack

The Classification Chain — What Has to Work

For end-to-end Corp > Guest priority, every layer must do its job:

LayerFunctionConfigured?
1. AP marks frames on egress (PCP/DSCP)Wireless → wireNo (XIQ User Profile QoS marker not set)
2. SW1 reads markings on Port 3 ingress802.1p / DSCP examinationYes (802.1p enabled by default)
3. SW1 maps to queueVLAN qosprofile, ACL, etc.Yes (VLAN 20→QP6, VLAN 30→QP1)
4. SW1 schedules between queuesWRR / Strict-Priority weightsAll weights = 1 (no differentiation)

Three independent failure modes were uncovered along this chain.

Finding #1 — FIXED

SW1 QoS Profile Weights Were All Equal (Weight = 1)

Initial state on SW1, system-wide:

show qosprofile

QP1    Weight = 1
QP6    Weight = 1
QP8    Weight = 1

Even with VLAN 20 mapped to QP6 and VLAN 30 mapped to QP1, the queue scheduler treats them equally under load because both have weight 1. This means even when traffic IS in different queues, no actual priority differentiation happens.

The Apr 21 lab work mapped VLANs to queues but never differentiated queue weights from default.

Fixed via:

configure qosprofile QP6 maxbuffer 100 weight 10
save configuration

After fix:

QP1    Weight =  1
QP6    Weight = 10    ← changed
QP8    Weight =  1

QP6 now has 10× the scheduling weight of QP1 under load.

Status: Fixed in lab. Saved to running config.

Finding #2 — ARCHITECTURAL

EXOS VLAN qosprofile is Ingress-Side Only

The most consequential finding. The EXOS command:

configure vlan VLAN_20_Corporate_Wireless qosprofile QP6

...classifies traffic based on the ingress VLAN tag. It does NOT apply at egress when traffic is L3-routed INTO the VLAN from another port.

DirectionPathResult
Upstream (client → Internet)Port 3 ingress (VLAN 20 tag) → classified to QP6 → routed via SVI → egress Port 1 in QP6QP6 Tx counter incrementing on Port 1
Downstream (Internet → client)Port 1 ingress (untagged from QF Modem, PCP=0) → classified to QP1 → L3-routed to VLAN 20 → egress Port 3 stays in QP1QP6 Tx counter on Port 3 = 0

This direction-asymmetry is the crux of the bug. Most user-facing traffic (YouTube, web browsing, video streaming) is dominated by downstream traffic. The VLAN qosprofile mapping handles only the upstream half. Downstream return traffic keeps its WAN-side queue assignment all the way through to the AP-facing egress port.

Architectural limitation: EXOS documentation does not surface this directional behavior. Users reasonably expect configure vlan X qosprofile QPn to enforce queue assignment for all VLAN-X traffic regardless of direction. It does not.

Finding #3 — DATA-PLANE

DSCP Marking Absent End-to-End

To work around Finding #2, one fallback is to enable DSCP-based classification at Port 1 ingress. But this requires inbound traffic to carry DSCP markings.

Pcap analysis of Corp and Guest YouTube traffic:

Source pcapFramesDSCP=0 (BE)Non-zero DSCP
Corp (work MBP en0)193,49699.85%198 frames @ DSCP=46 (Apple system protocols, NOT YouTube)
Guest (personal MBP en0)135,21199.96%6 frames @ DSCP=40, 4 @ DSCP=56 (system protocols)

YouTube/CDN traffic carries DSCP=0 from origin. The QF Modem doesn't preserve or add DSCP either. So even if SW1 enables diffserv examination on Port 1, there is no DSCP value to differentiate on. This rules out DSCP-based classification as a workaround.

The AP3000 doesn't add DSCP either — the XIQ User Profile QoS Marker setting (Class of Service / DSCP) was never configured during the Apr 21 lab work.

Fix Path

ACL-Based Classification on Port 1 Ingress

The remaining viable fix for the downstream side: explicit ACL-based classification at WAN ingress. Match by destination subnet, force into QP6 before the frame enters SW1's switch fabric.

# Conceptual EXOS access-list policy (requires policy file editor)
edit policy "Corp_Priority"
# Inside the vi-style editor:
entry Corp_DL_to_QP6 {
  if match all {
    destination-address 10.20.0.0/24 ;
  }
  then {
    qosprofile QP6 ;
  }
}
# Save with Esc :wq

configure access-list "Corp_Priority" ports 1 ingress
save configuration

This forces all traffic destined to the Corp subnet (10.20.0.0/24) into QP6 at WAN ingress, BEFORE it routes through. The queue assignment then sticks all the way through to Port 3 egress.

Ungainly but correct. The cleaner architectural answer would be either:
(a) EXOS extending VLAN qosprofile to apply at egress as well (firmware change), or
(b) AP3000 marking inbound traffic with DSCP before forwarding (XIQ feature support unclear), or
(c) Upstream routers/CDNs marking traffic that survives ISPs (out of scope for lab).

UX Gap

The XIQ Disconnect

XIQ's Network Policy User Profile has a "Scheduling Weight" field (set to 100 for Corp, 10 for Guest in this lab). This sounds like a global priority knob, but it isn't — it controls only AP-side WMM airtime scheduling. There is no automatic translation to switch-side queue weights or marker policy.

A user setting "Corp scheduling weight 100" reasonably expects end-to-end priority enforcement. Three additional configurations are required, none of which the UI surfaces:

  1. User Profile QoS Marker — Class of Service (PCP) and DSCP must be explicitly set so the AP marks frames on the wire. (Not configured in this lab.)
  2. Switch Template QoS Profiles — queue weights must be differentiated. (Not configured in this lab.)
  3. WAN-side ACL classification if downstream traffic is the dominant load. (Not configured in this lab.)

For the GTAC ticket, this UX gap is itself a meaningful finding to surface to Extreme R&D.
