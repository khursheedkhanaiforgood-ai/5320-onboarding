Clean role separation: SW2 (VOSS) is the local Fabric intelligence — it maps services to I-SIDs and hands traffic off via a physical cable from its Transit Port (Port 1) to the IPE. SW2 uses a Static Route or IP Shortcuts pointing RDC-destined traffic to the IPE's IP as next-hop. The IPE (physical box) owns everything beyond that — Fabric Extend tunnel encapsulation, IPsec, and the WAN path to the RDC. SW2 has no IAH, no tunnel config — it just routes to the IPE.

IPE — The Black Box (Physical Extreme Networks Appliance)

Architecture confirmed Apr 30 2026 (NoteBookLLM dialogue): IPE is a physical external hardware appliance. SW2 does NOT run IAH internally. SW2 has a dedicated Transit Port (Port 1) that cables directly to the IPE. SW2 uses Static Route / IP Shortcuts to forward RDC-bound traffic to the IPE as next-hop. The IPE owns the Fabric Extend tunnel — SW2's VOSS config has zero tunnel or IAH involvement.
Physical Placement

SW2 Transit Port → IPE → HomeModem

IPE is cabled: SW2 Port 1 (Transit Port) → IPE LAN port. IPE WAN port → HomeModem LAN port. It is the physical WAN gateway for all Path B traffic. SW2 sees the IPE as its next-hop for RDC-destined routes.

Fabric Extend Owner

FE-Tunnel Terminates on IPE

The Fabric Extend tunnel is configured on the IPE — not on SW2. IPE encapsulates I-SIDs in IP/IPsec headers and sends them over the HomeModem to the RDC in Raleigh. SW2 simply routes traffic to the IPE and the IPE does the heavy lifting.

Application Hosting

IAH Runs on the IPE

The "Integrated Application Hosting" feature belongs to the IPE device itself. The IPE hosts Docker-based applications at the edge — service connectors, analytics, RDC-managed agents — all running on the IPE hardware, not on SW2.

Traffic Flow — Corrected Step by Step

① iPhone → AP2 (Wi-Fi association)
② AP2 → SW2 port (Fabric Attach — AP2 LLDP FA TLVs signal I-SID request)
③ SW2 auto-sense maps AP2 service to I-SID → zero-touch, no manual port config
④ SW2: I-SID frame forwarded per Fabric rules toward RDC destination
⑤ SW2 Static Route / IP Shortcuts → next-hop = IPE IP → exits via Transit Port (Port 1)
⑥ Physical cable: SW2 Port 1 → IPE LAN port  (the "Secure Edge Handoff")
⑦ IPE: Fabric Extend encapsulation → wraps I-SID frame in IP + IPsec header
⑧ IPE WAN port → HomeModem → Internet  (encrypted Fabric Extend tunnel)
⑨ Internet → Extreme Networks RDC, Raleigh NC  (far-end FE endpoint on RDC)
⑩ RDC decapsulates → delivers traffic to service offering

SW2 VOSS Role vs IPE Role — Clear Boundary

SW2 (VOSS) — Local Intelligence

Fabric Edge: I-SIDs + Routing

IS-IS Control Plane, Nickname, I-SID mapping, Fabric Attach on AP2 port. Defines Transit Port (Port 1) as the physical link to IPE. Configures Static Route or IP Shortcuts so RDC-bound traffic exits via Transit Port to IPE as next-hop. No tunnel config. No IAH. No FE on SW2.

IPE (Physical Box) — Heavy Lifting

Tunnel + WAN + App Hosting

Owns the Fabric Extend tunnel (FE-Tunnel configured here). IPsec encapsulation of I-SID frames. WAN uplink to HomeModem. Fabric Extend to RDC in Raleigh. Hosts IAH applications (Docker). SW2 knows nothing about the tunnel — it just routes to the IPE.

Still needed for IPE: Device model/part number, management access method (CLI / XIQ / dedicated UI), VOSS/FE version, management IP scheme, and RDC endpoint configuration details (IP address in Raleigh, IPsec keys/certs).

Fabric Architecture — VOSS / SPB on SW2

IS-IS Control Plane

SW2 Fabric Identity

First task after VOSS upgrade: establish IS-IS Control Plane. SW2 gets Nickname 0.00.02 — its unique identity in the Fabric domain. IS-IS discovers topology and builds the SPB forwarding tree across the Fabric Extend tunnel to the RDC.

I-SIDs

Service Identifiers (replace VLANs)

I-SIDs are the service abstraction in Fabric Connect. Each wireless service (SSID) on AP2 maps to an I-SID on SW2. Fabric Extend wraps these I-SIDs in IP headers so they traverse the internet to the RDC as if on a local cable.

Fabric Extend (FE)

WAN Tunnel — SPB over IPsec

FE-Tunnel configured on SW2: source = SW2/IPE WAN IP, destination = RDC gateway Raleigh. Carries I-SIDs transparently across the HomeModem. No VPN client needed — FE is native VOSS. IS-IS adjacency forms across the tunnel.

Fabric Attach (FA)

AP2 Zero-Touch Provisioning

fa enable + auto-sense enable on the SW2 port connected to AP2. AP3000 sends FA LLDP TLVs requesting its I-SIDs. SW2 auto-provisions — no manual VLAN or I-SID tagging on the port. True zero-touch AP onboarding.

Phase 1 — SW2 VOSS Upgrade (Pre-Flight)

Prerequisite: Delete SW2 and AP2 from XIQ before rebooting — otherwise ZTP+ re-pushes EXOS config on first boot.
  • 1
    Delete SW2 + AP2 from XIQ
    XIQ → Manage → Devices → select SW2 (192.168.0.11) + AP2 (192.168.0.25) → Delete. Confirm.
  • 2
    Console cable to SW2
    screen /dev/cu.usbserial-A9VKJO11 115200
  • 3
    Reboot and catch bootloader
    boot from EXOS CLI (or power cycle). Press Space Bar immediately at the bootloader prompt. Timing is critical — if missed, power cycle and retry.
  • 4
    Select VOSS conversion
    In bootloader menu: select "Change the switch OS to Fabric Engine (VOSS)". Confirm. Switch downloads VOSS image, applies, reboots. All EXOS config is wiped.
  • 5
    First VOSS login
    Default: rwa / rwa. Run show sys-info to confirm VOSS version and hostname.
  • 6
    Execute VOSS Pre-Flight CLI
    Hostname, management IP, SSH keys, IS-IS, Nickname 0.00.02, B-VLANs. Script ready to generate on your go-ahead.
  • 7
    Clear Mac SSH known_hosts
    ssh-keygen -R 192.168.0.11 before SSH to new VOSS management IP.

Phase 2 — IPE Physical Integration (Secure Edge Handoff)

Blocked on: IPE device model + RDC endpoint details. Steps below are the logical sequence — no IAH config on SW2, only Transit Port + routing.
  • P
    Confirm IPE device model + receive unit
    Get the exact Extreme Networks IPE part number. Confirm management access method (CLI / dedicated UI / XIQ). Note: the FE-Tunnel is configured ON the IPE — not on SW2.
  • P
    Physical cabling: SW2 Port 1 → IPE → HomeModem
    SW2 Port 1 (Transit Port) → IPE LAN port. IPE WAN port → HomeModem LAN port. Verify IPE gets a WAN IP from HomeModem (DHCP or static). Verify IPE can ping 8.8.8.8.
  • P
    Configure SW2 Transit Port (Port 1)
    Define SW2 Port 1 as the routed/L3 interface toward the IPE. Configure Static Route or IP Shortcuts: ip route <RDC-subnet> <IPE-LAN-IP>. This is the only SW2 config needed for the IPE handoff — no IAH, no tunnel.
  • P
    Verify Secure Edge Handoff
    From SW2: ping IPE LAN IP via Transit Port. Confirm Static Route is installed. Trace route to RDC subnet — first hop should be IPE. All tunnel logic lives on IPE from this point forward.
  • P
    Configure Fabric Extend on IPE (not SW2)
    On the IPE: configure FE-Tunnel with source = IPE WAN IP, destination = RDC gateway in Raleigh. IPsec keys/certs from Extreme Networks team. Once FE-Tunnel is UP on IPE, I-SIDs flow transparently to RDC.

Phase 3 — Fabric Extend + RDC Tunneling

Blocked on: RDC gateway IP and any pre-shared keys/certs from Extreme Networks team (resolves Q2 remainder + Q3).
  • P

  • P

Phase 4 — AP2 Fabric Attach (Zero-Touch)
