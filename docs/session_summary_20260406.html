Factory Reset & ZTP+ Reconnect

Both 5320 switches factory reset (Option A — clean slate for XIQ-only deployment). Previous CLI-configured VLANs wiped. Switches powered on, ZTP+ activated automatically on any RJ45 port with internet access.

Factory Reset Method (EXOS)

delete /intflash/config.cfg
boot

After reboot: all ports on VLAN 1, DHCP on Default VLAN, switch phones home to XIQ automatically via ZTP+.

XIQ Deployment

XIQ-Only Deployment — VLANs + APs

All configuration pushed exclusively through ExtremeCloud IQ. No CLI, no serial console used after factory reset.

Deployment Flow

  1. Switches appeared in XIQ automatically via ZTP+ (pre-onboarding state)
  2. Claimed both switches in XIQ → assigned to Network Policy
  3. Created VLANs 20, 30, 50, 60 in Network Policy
  4. Assigned port profiles: port 3 on each switch as AP-facing (tagged VLANs + VLAN 1 untagged)
  5. Pushed config to switches via XIQ
  6. APs powered up via PoE on port 3 → received DHCP on VLAN 1 → phoned home to XIQ
  7. Adopted APs in XIQ → assigned WLAN policy
  8. XIQ pushing SSID/radio config to APs (AP2 at 37% at session close)

VLAN 10 (Management) — Sprint 2

VLAN 10 is reserved for in-band switch management. Currently not configured. Planned for Sprint 2 to isolate management traffic from user data VLANs.

ConceptExplanation
Physical MGMT portDedicated RJ45 on 5320 faceplate. Out-of-band. Never carries user traffic. Plug a laptop directly for emergency access.
In-band VLAN 10A VLAN on data ports specifically for management (SSH, SNMP, XIQ). Planned SW1: 10.10.0.1, SW2: 10.10.0.2
EXOS naming caveat"Mgmt" is a reserved keyword — use "SwMgmt" or "APMgmt" as the VLAN name in XIQ

Current state is fine: VLAN 1 handling management is acceptable for lab scale. VLAN 10 adds enterprise-grade isolation — implement in Sprint 2 after APs are fully verified.

⚠ Critical Lesson — Internet Not Working After DHCP + Default Route
Always run enable ipforwarding vlan Default on every switch

Return traffic from the internet arrives on VLAN 1 (Default) — the uplink to the Quantum Fiber router. Without IP forwarding enabled on Default VLAN, the switch cannot route that return traffic back to clients on VLANs 20/30/50/60. Result: clients get DHCP, get a gateway, but Safari times out.

This was confirmed working in the March 29 lab session and re-discovered April 7. It must be run after every factory reset before testing internet from client devices.

enable ipforwarding vlan Default    ← SW1
save configuration

enable ipforwarding vlan Default    ← SW2
save configuration
⚠ Critical Lesson — Default Route is Automatic via DHCP, ipforwarding is NOT
Do NOT manually add configure iproute add default 192.168.0.1 — EXOS installs it automatically from DHCP lease

When the switch gets its management IP via DHCP (e.g. 192.168.0.11 from Quantum Fiber), EXOS automatically installs a BOOTP/DHCP default route (origin bo in show iproute). You can verify: look for bo Default Route 192.168.0.1.

What IS required and is NOT automatic:

enable ipforwarding vlan Default    ← manual, required on every switch after factory reset
save configuration
