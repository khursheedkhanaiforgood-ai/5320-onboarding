Method

Diagnostic Framework

Every troubleshooting walk follows this four-step loop. Skipping any step degrades to "guessing" or "tool-chasing." Today's incident was solved by running this loop five times — once per hypothesis tested.

Step 1
Symptoms
What is observable? Capture every symptom — even contradictions. Contradictions are clues, not noise.
Step 2
Hypothesis
What single mechanism explains all symptoms? If a hypothesis only explains some, it's incomplete.
Step 3
Test
Pick the test that distinguishes between competing hypotheses. Pcaps, CLI dumps, counter diffs.
Step 4
Validation
Did the data confirm or break the hypothesis? Scripts · Debugs · Pcaps · Counter snapshots · Repro.

"Tools to collect data don't replace understanding. The framework is what turns data into diagnosis."

Hypothesis Arena

Five Hypotheses Tested, One Wins

H1 — Alex/NotebookLLM: "MAC Flap between Port 5 and Port 3"  REJECTED

Claim: Same MAC appears on both wired (Port 5) and wireless (via AP, Port 3); SW1 FDB locks; DHCP listener stalls.

Test: Look up the MacBook's actual MACs.

Validation: Three different MACs on the same device — wired UAA, plus a unique LAA per SSID. Nothing flaps. All three of Alex's downstream Socratic questions collapse with this premise.

H2 — Alex Q1: "Wrong VLAN landing — modem leak via 192.168.1.1"  PARTIAL

Claim: MacBook traffic landed on Default VLAN; modem responded with 192.168.1.1.

Test: Compare both bad pcaps for non-MacBook traffic.

Validation: v4 (Guest) shows one IGMP from 192.168.1.1 ✓. v5 (Corporate) is completely silent — zero non-MacBook traffic. Theory only explains 50% of the failure.

H3 — Bridge MAC FDB at AP3000  REJECTED

Claim: AP's internal Linux software bridge has a bad/stale FDB entry for the MacBook MAC; frame dropped.

Test: What does a Linux bridge do with unknown unicast destinations?

Validation: Bridges flood unknowns to all ports — they don't drop. So a missing FDB entry would still result in SW1 Port 3 receiving the frame. SW1 didn't, so frames died upstream of the bridge FDB lookup — at the decryption layer.

H4 — Per-client TX queue / buffer  REJECTED

Claim: AP's per-client TX queue (powersave hold, A-MPDU buffer) wedged for MacBook.

Test: What direction is the TX queue? Does the symptom span both directions?

Validation: TX queue is AP→client (downstream-only). But SW1's FDB never saw upstream DISCOVERs either — a downstream-only fault cannot break upstream. Queue wedges typically affect all clients on the radio, not one MAC.

H5 — WPA2 Key State (TK + GTK) at AP3000  WINS

Claim: 4-way EAPOL handshake wedged → bad TK install for MacBook (silent decrypt failure both directions). GTK rotation triggered by MacBook's deauth → Group Key Handshake to iPhone wedges → iPhone deaf to broadcasts.

Test: Does this explain MacBook silence AND iPhone failure AND silent nature AND reboot fix?

Validation: ✓ Bidirectional (TK encrypts both ways). ✓ MAC-specific (each client has unique PTK). ✓ Silent (encrypted-frame-discard has no client-visible error). ✓ Reboot = full re-handshake. ✓ GTK extension explains all-clients failure when one client's handshake fails. Tie-in to May 4 EOD Concepts #4 (KEK vs TK) and #5 (GTK Impact).

AP-Internal Tables

4-Option Elimination Matrix

The AP3000's software bridge has multiple in-memory tables. Frames pass through them in sequence. We tested each:

#TableDirectionSilent?Verdict
(a)WLAN client state machineBothSometimes (logs assoc events)Runner-up
(b)WPA2 key state (TK + GTK)BothYes✓ WINS
(c)Bridge MAC FDBForwarding onlyNO — floodsEliminated
(d)Per-client TX queueDownstream onlySometimesEliminated

Why (b) wins: Bidirectional, per-client OR all-clients (TK vs GTK), silent by design, clearable by re-handshake. Only option that survives every elimination test simultaneously.

Plot Twist

Why iPhone Failed Too — GTK Rotation

Single-client TK theory cannot explain why iPhone broke. Resolution: 802.11 has two keys per client session.

KeyScopeEncrypts
TK (Temporal)Per-client unicastAP ↔ this-one-client
GTK (Group)Per-BSSID broadcastALL multicast/broadcast frames

The likely sequence that broke iPhone

T0  iPhone has GTK_v1, working
T1  MacBook 4-way handshake wedges/times out
T2  AP deauths MacBook (handshake never completed)
T3  AP rotates GTK → GTK_v2 (security: prevent failed
    client from decrypting future broadcasts)
T4  AP runs Group Key Handshake to push GTK_v2 to iPhone
    M1: AP sends GTK_v2 (encrypted with iPhone's KEK)
    M2: iPhone ACK back to AP
T5  M1 or M2 wedges (lost, retried, replay counter desync)
T6  AP commits to GTK_v2; iPhone still on GTK_v1
T7  DHCP OFFER (broadcast) → AP encrypts with GTK_v2 →
    iPhone decrypts with GTK_v1 → MIC fail → silent discard
T8  iPhone lease expires → can't renew → DISCOVER → broken

Reboot fixes it because the AP wipes all key state. Every client re-runs full 4-way handshake. Fresh GTK is distributed cleanly to everyone.

Outcomes

Concepts Mastered (9)

  1. Apple per-SSID privacy MAC — same device shows different LAA MAC per SSID; can revert to UAA after long disconnect or setting change.
  2. Linux software bridges flood unknowns — unknown unicast destinations are flooded, not dropped. Bridge FDB cannot be the cause of silent loss.
  3. TX queue is downstream-only — per-client TX buffers govern AP→client. Cannot explain client→AP→wire loss.
  4. TK = unicast, per-client — bad TK install = one client deaf in BOTH directions (encryption symmetric).
  5. GTK = broadcast, per-BSSID — bad GTK = ALL clients deaf to broadcasts. DHCP OFFER (broadcast) is the most common casualty.
  6. Encryption mismatch is silent — failed MIC discards happen at hardware crypto layer; no client-visible error, no log by default.
  7. Reboot consumes evidence — capture AP-side diagnostics BEFORE rebooting. Reboot is a reset, not a diagnosis.
  8. GTK rotation on disassoc — AP rotates group key when any client leaves; group key handshake to remaining clients can wedge silently.
  9. Symptom → Hypothesis → Test → Validation framework — every bug-hunt loops this 4-step pattern. Skipping it = guess and tool-chase.
