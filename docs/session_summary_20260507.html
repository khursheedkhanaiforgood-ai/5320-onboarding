01 Diagnostic Framework — The Meta-Pattern

Every troubleshooting walk follows this loop. Skipping any step degrades to "guessing" or "tool-chasing." Today's DHCP incident was solved by running this loop three times — once for each rejected hypothesis, then a fourth time for the winner.

STEP 1
SYMPTOMS
What is observable?
(Pcap, logs, "AP not responding", reboot fixed it.) Capture EVERY symptom, even ones that seem contradictory — the contradictions are clues.
→
STEP 2
HYPOTHESIS
What single mechanism explains EVERY symptom?
If a hypothesis only explains some symptoms, it's incomplete. Force it to fit ALL data — or move on.
→
STEP 3
TEST
What's the most diagnostic question?
Pcaps, CLI dumps, counter diffs, reproduce-the-failure tests. Pick the test that distinguishes between competing hypotheses.
→
STEP 4
VALIDATION
Did the data confirm or break the hypothesis?
Scripts · Debugs · Pcaps · Counter snapshots · Repro. If validation fails → loop back to Step 2 with a refined hypothesis.

"Tools to collect data don't replace understanding. The framework is what turns data into diagnosis."

04 Hypothesis Arena — 4 Tested, 1 Wins
H1 — Alex/NotebookLLM: "MAC Flap between Port 5 and Port 3"
✗ REJECTED — premise is factually wrong
Claim: same MAC appears on both wired (Port 5) and wireless (via AP, Port 3) → SW1 FDB locks → DHCP listener stalls.
Test: Look up the MacBook's actual MACs. Wired = 80:69:1a (UAA, Apple OUI). Wi-Fi Corp = e2:68:dc (LAA, privacy). Wi-Fi Guest = 4e:60:81 (LAA, privacy).
Validation: Three different MACs, no overlap. Nothing flaps. All three of Alex's Socratic Qs collapse with this premise.
H2 — Alex Q1: "Wrong VLAN landing (192.168.1.1 modem leak)"
⚠ PARTIAL — fits one pcap, not the other
Claim: MacBook traffic landing on Default VLAN, modem responds with 192.168.1.1.
Test: Compare both bad pcaps for non-MacBook traffic.
Validation: v4 (Guest) shows one IGMP from 192.168.1.1 ✓. v5 (Corporate) is completely silent — zero non-MacBook traffic. Theory only explains 50% of failure.
H3 — Bridge MAC FDB at AP3000
✗ REJECTED — bridges FLOOD, don't drop
Claim: AP's internal Linux software bridge has bad/stale FDB entry for MacBook MAC → frame dropped.
Test: What does a Linux bridge do with unknown unicast destination MAC?
Validation: brctl-style bridges flood unknowns to all ports. So a missing FDB entry would cause frames to be flooded to wired Ethernet → SW1 Port 3 would still receive and learn the MAC. SW1 didn't learn the MAC ⇒ frames died upstream of bridge FDB lookup.
H4 — Per-client TX queue / buffer
✗ REJECTED — downstream-only, can't explain bidirectional loss
Claim: AP's per-client TX queue (powersave hold, A-MPDU buffer) wedged for MacBook → frames not delivered.
Test: What direction is TX queue? Does symptom span both?
Validation: TX queue is AP→client (downstream). But SW1 FDB never saw upstream DISCOVERs either. Queue alone can't break upstream. Also, queue wedges typically affect all clients on the radio, not one MAC. Reboot would fix once but recur if cause persists — user saw single recovery.
H5 — WPA2 Key State (TK + GTK) at AP3000
✓ WINS — explains every fact, including iPhone breaking
Claim: 4-way EAPOL handshake wedged → bad TK install for MacBook (silent decrypt failure both directions). Then GTK rotation triggered by MacBook's deauth → Group Key Handshake to iPhone wedges → iPhone deaf to broadcasts.
Test: Does this explain BOTH MacBook silence AND iPhone failure AND silent nature AND reboot fix?
Validation: ✓ Bidirectional (TK encrypts both ways). ✓ MAC-specific (each client has unique PTK). ✓ Silent (encrypted-frame-discard has no client-visible error). ✓ Reboot = full re-handshake. ✓ GTK extension explains all-clients failure when one client's handshake fails. Tie-in to May 4 EOD Concepts #4 (KEK vs TK) and #5 (GTK Impact).

Why (b) wins: Bidirectional, per-client OR all-clients (TK vs GTK), silent by design, clearable by re-handshake. Only option that survives every elimination test simultaneously.

06 Plot Twist — Why iPhone Also Failed (GTK rotation)

Single-client TK theory cannot explain why iPhone broke. Resolution: 802.11 has TWO keys per client session.

KeyScopeEncrypts
TK (Temporal) Per-client unicast AP ↔ this-one-client
GTK (Group) Per-BSSID broadcast ALL multicast/broadcast frames
T0 iPhone has GTK_v1, working T1 MacBook 4-way handshake wedges/times out T2 AP deauths MacBook (handshake never completed) T3 AP rotates GTK → GTK_v2 (security: prevent failed client from decrypting future broadcasts) T4 AP runs Group Key Handshake to push GTK_v2 to iPhone M1: AP sends GTK_v2 (encrypted with iPhone's KEK) M2: iPhone ACK back to AP T5 M1 or M2 wedges (lost, retried, replay counter desync) T6 AP commits to GTK_v2; iPhone still on GTK_v1 T7 DHCP OFFER (broadcast) → AP encrypts with GTK_v2 → iPhone decrypts with GTK_v1 → MIC fail → silent discard T8 iPhone lease expires → can't renew → DISCOVER → broken

Reboot fixes: AP wipes ALL key state. Every client re-runs full 4-way handshake. Fresh GTK distributed cleanly.

08 Concepts Mastered Today (9)
✓
Apple per-SSID privacy MAC — same device shows different LAA MAC per SSID; can revert to UAA after long disconnect or setting change.
✓
Linux software bridges flood unknowns — unknown unicast destinations are flooded, not dropped. Bridge FDB cannot be the cause of silent loss.
✓
TX queue is downstream-only — per-client TX buffers govern AP→client. Cannot explain client→AP→wire loss.
✓
TK = unicast, per-client — bad TK install = one client deaf in BOTH directions (encryption symmetric).
✓
GTK = broadcast, per-BSSID — bad GTK = ALL clients deaf to broadcasts. DHCP OFFER (broadcast) is the most common casualty.
✓
Encryption mismatch is silent — failed MIC discards happen at hardware crypto layer; no client-visible error, no log by default.
✓
Reboot consumes evidence — capture AP-side diagnostics BEFORE rebooting. Reboot is a reset, not a diagnosis.
✓
GTK rotation on disassoc — AP rotates group key when any client leaves; group key handshake to remaining clients can wedge silently.
✓
Symptom→Hypothesis→Test→Validation framework — every bug-hunt loops this 4-step pattern. Skip it = guess and tool-chase.
