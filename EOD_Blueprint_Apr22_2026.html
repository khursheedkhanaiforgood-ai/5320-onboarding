Session 01 · Apr 20 + Apr 22
OSI Model Fundamentals (Layers 1–3)
Target: Working knowledge
✓ PASS
  • Explain purpose and responsibilities of Layers 1, 2, and 3
  • Describe how traffic flows from physical transmission through switching and routing
  • Identify common troubleshooting areas associated with each layer
Key Answer — L1/L2/L3 Purpose
L1 Physical: Transmission medium / energy source — electromagnetic waves (wireless), light (fiber), voltage (copper). L2 Data Link: Bit streams → frames with start/end delimiters, MAC addressing, VLAN tagging. L3 Network: End-to-end IP routing, source-to-destination packet forwarding across subnets.
Key Answer — Multi-Hop Packet Flow
Source device checks if destination IP is in its subnet. If yes → ARP for destination MAC, send directly. If no → ARP for default gateway MAC, send to gateway. At each hop: router reads destination IP, consults routing table, ARPs for next hop MAC, rewrites L2 wrapper, forwards. IP header stays unchanged end-to-end; only L2 MAC rewrites at each hop. Same mechanism whether routing table is populated statically or dynamically (OSPF).
Key Answer — Troubleshooting per Layer
L1: Client unreachable anywhere → cable, interface, physical link issue. L2/L3: Can't reach different VLAN → missing SVI/gateway IP. L3: Local OK but no internet → return path broken (often ipforwarding vlan Default disabled).

Layer 2 Networking Concepts
Target: Working knowledge
✓ PASS
  • Explain how MAC addresses are used in Layer 2 communication
  • Demonstrate understanding of VLAN segmentation
  • Understand Ethernet frame structure and switching behavior
Key Answer — MAC Address Fundamentals
MAC addresses are 48-bit hexadecimal identifiers burned into NIC firmware at manufacturing. First 3 octets = OUI (Organizationally Unique Identifier) identifying the manufacturer. Can be spoofed via software. Uniquely identify L2 devices within a broadcast domain.
Key Answer — FDB vs ARP Table
FDB (Forwarding Database): MAC-to-port mapping on switches — L2 concept. ARP Table: IP-to-MAC mapping on end devices/routers — L3-to-L2 resolution. Switches read Source MAC to learn (populate FDB), read Destination MAC to forward. Unknown destination MAC → flood to all ports on that VLAN. Broadcast MAC (ff:ff:ff:ff:ff:ff) → flood without FDB lookup.
Key Answer — VLAN Segmentation
VLANs are logical L2 partitions, not physical. Two devices on the same physical switch in different VLANs cannot communicate at L2 — they are completely isolated unless an SVI (L3 gateway) exists on a device that owns gateway IPs in both VLANs.
Session 03 · Apr 22
Layer 3 — IPv4, Subnetting & Routing
Target: Working knowledge
✓ PASS
  • Understand IPv4 addressing structure and subnetting basics
  • Explain default gateway behavior and inter-subnet routing
  • Differentiate static vs dynamic routing mechanisms
Key Answer — Subnet Math
IPv4 = 32 bits. /24 = 24 network bits, 8 host bits → 256 addresses (254 usable). /25 = 25 network, 7 host → 128 addresses (126 usable). For 10.40.0.50/25: block is 10.40.0.0 – 10.40.0.127. Network = .0, Broadcast = .127, Hosts = .1 – .126. Same IP but /24 would span 10.40.0.0 – 10.40.0.255.
Key Answer — Gateway Behavior
Device at 10.40.0.50/25 wants to reach 10.40.0.150. Since 150 is outside the /25 block (ends at .127), device must send to default gateway. Device does NOT route itself — it ARPs for the gateway MAC, sends the frame to gateway, gateway makes the L3 routing decision.
Key Answer — Static vs Dynamic Routing
Static: Manually configured routing table entries. Simple, no overhead, but brittle — if a link fails, packets are stuck. Dynamic (OSPF): Routers advertise directly-connected subnets, learn topology from neighbors, calculate best path via SPF algorithm, automatically reconverge when topology changes. Same forwarding mechanism at each hop (ARP next hop, rewrite L2) — the difference is only how the routing table is populated.
Key Answer — Campus Network Design
Two-building campus: Building A (10.10.0.0/24) + Building B (10.20.0.0/24), central gateway router. Client in A reaches client in B: (1) Client ARPs for A's gateway, (2) Frame reaches gateway, (3) Gateway consults routing table, (4) Gateway ARPs for B's edge router MAC, (5) Rewrites frame, forwards, (6) B's edge router looks up destination in FDB, forwards to local port, (7) Destination client receives. IP header never changes; MAC wrapper changes at every hop.

Wireless Networking Fundamentals
Target: Working knowledge
✓ PASS
  • Explain how RF conditions affect wireless performance
  • Identify common causes of wireless connectivity issues
  • Demonstrate understanding of how clients connect and communicate with APs
Key Answer — RSSI & dBm
dBm is logarithmic: -30 dBm is stronger than -70 dBm (closer to 0 = stronger). Every 10 dB = 10× power change; every 3 dB ≈ 2× (half power). Practical: -30 dBm excellent, -70 dBm weak-usable, -90 dBm unusable. RSSI alone is insufficient — SNR (Signal-to-Noise Ratio) matters equally. Strong RSSI with bad SNR means frames can't be decoded.
Key Answer — Frequency Bands
2.4 GHz: 2.400–2.4835 GHz, 22 MHz channel width, only 3 non-overlapping (1/6/11), best range but most congested. 5 GHz: UNII-1/2A/2C/3, 20/40/80/160 MHz widths, ~25 non-overlap (no DFS) / ~45 with DFS, DFS required on UNII-2A/2C (radar avoidance). 6 GHz (WiFi 6E+): 5.925–7.125 GHz (1200 MHz!), 59 non-overlapping 20 MHz channels, 15 PSC channels for discovery, no DFS, WiFi 6E and 7 only.
Key Answer — WiFi 6 vs WiFi 7 Architecture
Both run on top of the 2.4/5/6 GHz bands. WiFi 6 (802.11ax): OFDMA, 1024-QAM, 160 MHz max, single-band connection. WiFi 7 (802.11be): 4096-QAM, 320 MHz aggregation, MLO (Multi-Link Operation) — client uses multiple bands simultaneously. Example: Video on 6 GHz (clean), audio on 5 GHz (fallback) — 240 MHz aggregated.
Key Answer — DFS & AFC
DFS (5 GHz): AP detects radar chirps in firmware, immediately vacates the channel, switches to clean channel, stays off 30+ min. Automatic, no config. AFC (6 GHz, Standard Power): AP queries cloud database every 24 hours with GPS + power level; server approves or blocks channels based on other registered APs. Not real-time — ~30 min propagation delay when APs shut down.

WiFi 6/7 Client Connection Flow — Deep Dive

Five-stage walkthrough: Discovery → Authentication → Association → 4-Way Handshake → Data Plane

◆
Walk through this alongside the WPA Simulator and the 802.11be Simulator. The simulator animates exactly what's described here.
Stage 01
Discovery — iPhone Finds the AP
Passive scan: iPhone hops through channels listening for Beacon frames (broadcast by APs every ~100ms). Each beacon contains SSID, BSSID (AP MAC), supported rates, channel, security type, RSSI.

Active scan: iPhone also broadcasts Probe Requests on each channel. Every AP in range responds with Probe Response containing the same info.

iPhone builds "Available Networks" list and the user taps their choice. WiFi 7 adds 6 GHz PSC (Preferred Scanning Channels) for faster discovery — 15 priority channels instead of scanning all 59.
Beacon Frame: [Frame Control] [Timestamp] [Beacon Interval=100ms] [Capability Info] [SSID IE: "Horizon-Corp"] [Supported Rates] [Current Channel: 36] [RSSI] [Security IE: WPA3-SAE]
Stage 02
Authentication — SAE Handshake (WPA3)
User enters password and taps SSID. iPhone sends Authentication Request — but the password is never transmitted.

SAE Dragonfly exchange: Both iPhone and AP independently use the password as input to a cryptographic function, generating matching scalars and elements. They exchange these in Commit frames. If both used the same password, the math derives identical PMK on both sides.

Confirm frames: Both sides prove they derived the same PMK via a MIC check. Mutual authentication complete — with forward secrecy. Capturing the entire exchange yields zero information about the password.
SAE Flow (5 frames): 1. iPhone → AP: Auth Request (Algo=SAE) 2. AP → iPhone: SAE Commit (AP's scalar + element) 3. iPhone → AP: SAE Commit (iPhone's scalar + element) 4. iPhone ↔ AP: SAE Confirm (MIC verification both ways) 5. AP → iPhone: Auth Response (Status=Success)
WiFi 7 mandates H2E (Hash-to-Element) method for SAE — stronger side-channel attack resistance.
Stage 03
Association — iPhone Declares Capabilities
iPhone sends Association Request containing all its capabilities: supported rates, HT (WiFi 4+), VHT (WiFi 5+), HE (WiFi 6), EHT (WiFi 7), security RSNE. AP evaluates and responds.

Association Response: AP assigns an AID (Association ID — e.g., AID=5 for "client #5"). iPhone is now associated. Both sides know each other's negotiated parameters.

WiFi 6: Single-band negotiation (5 GHz 80 MHz max). WiFi 7: MLO negotiation — iPhone declares "I can do 5 + 6 GHz concurrently," AP agrees to dual-link operation with 240 MHz aggregated capacity.
Association Request: [Capability Info] [Listen Interval] [SSID] [Supported Rates] [HT Cap] [VHT Cap] [HE Cap] ← WiFi 6 [EHT Cap] ← WiFi 7 (includes MLO capabilities) [RSNE: WPA3-Personal] Association Response: [Status=Success] [AID=5] [Supported Rates + negotiated capabilities]
Stage 04
4-Way Handshake — Key Derivation (PTK + GTK)
The PMK from SAE is temporary. The 4-Way Handshake derives the actual encryption keys for data traffic.

M1 (AP → iPhone): ANonce (AP's random nonce), unencrypted.
M2 (iPhone → AP): SNonce + MIC. Both sides now have ANonce + SNonce + MACs → independently derive identical PTK via PRF function.
M3 (AP → iPhone): GTK (Group Transient Key for broadcast/multicast) encrypted with the newly-derived PTK + MIC confirmation.
M4 (iPhone → AP): ACK. Both sides install PTK and GTK.

After M4, all subsequent data frames are encrypted with PTK using CCMP (AES-128) or GCMP-256 (WiFi 7 192-bit mode).
4-Way Handshake (4 frames): M1: AP → iPhone [ANonce] M2: iPhone → AP [SNonce, MIC] → both derive PTK M3: AP → iPhone [GTK_encrypted, MIC] → AP confirms PTK install M4: iPhone → AP [ACK] → iPhone confirms install PTK derivation formula: PTK = PRF(PMK, "Pairwise key expansion", Min(MAC_ap, MAC_client) + Max(MAC_ap, MAC_client) + Min(ANonce, SNonce) + Max(ANonce, SNonce))
Stage 05
Data Plane — DHCP → DNS → TCP → Zoom
Encryption is live. iPhone proceeds through the OSI stack for actual application traffic.

Step 1 — DHCP: iPhone broadcasts DHCP Discover (encrypted with PTK in WiFi frame). AP decrypts, forwards unencrypted Ethernet to switch. SW1 DHCP server on VLAN 20 responds with 10.20.0.150, gateway 10.20.0.1, DNS 8.8.8.8.

Step 2 — DNS: iPhone queries DNS for zoom.us → 203.0.113.100.

Step 3 — TCP: iPhone opens socket [10.20.0.150:54321 ↔ 203.0.113.100:443], completes SYN → SYN-ACK → ACK handshake.

Step 4 — Zoom stream: Encrypted WiFi frames carry HTTPS-wrapped Zoom video. QoS priority=5 (high) ensures Corporate traffic gets serviced before Guest (priority=0) during congestion.
Complete stack per Zoom video frame: L7 Application: Zoom video codec output L6 Presentation: TLS/SSL encryption (HTTPS) L5 Session: HTTPS session state L4 Transport: TCP src=54321, dst=443 L3 Network: IP src=10.20.0.150, dst=203.0.113.100 L2 Data Link: WiFi frame with CCMP encryption (PTK) QoS Priority=5, BSSID, AID=5 L1 Physical: 5 GHz channel 36 OFDM modulation
Key insight on encryption scope: WiFi CCMP encryption protects iPhone ↔ AP only. Once the AP decrypts and forwards to switch, the Ethernet side is unencrypted. HTTPS (L7) provides end-to-end encryption that survives beyond the AP boundary.

Key Answer — AP Onboarding Flow
(1) Pre-register AP serial number in XIQ. (2) AP powers on, boots into Default VLAN (VLAN 1) untagged. (3) AP sends DHCP Discover on Default VLAN. (4) DHCP server (modem or local) assigns 192.168.0.x. (5) AP contacts Aerohive redirector (cloud service). (6) Redirector returns XIQ URL + onboarding token. (7) AP contacts XIQ, proves identity via pre-registered serial. (8) AP downloads initial config and joins assigned Network Policy with SSIDs, security settings, VLAN mappings. Critical: Port 3 must have VLAN 1 (Default) untagged or AP cannot phone home.
Key Answer — XIQ Client Lookup
Monitor → Clients (global list across all APs). Filter by AP, SSID, or user. Per-client detail includes MAC, IP, SSID, VLAN, RSSI, connection time, association status. Alternative: Devices → [specific AP] → Clients tab.
Key Answer — "Connected but no internet" Troubleshooting
Check two things first: (1) Is the DHCP server enabled on the trunk port (Port 10) between SW1 and SW2? Clients on SW2 need DHCP reach-back. (2) Are modem static routes pointing the VLAN subnets back to SW1's management IP? These are the two most common return-path issues in a Primary/Secondary architecture.

Troubleshooting Methodology & Packet Capture
Target: Working knowledge
✓ PASS
  • Apply a logical troubleshooting process
  • Identify the OSI layer where an issue is most likely occurring
  • Assist in isolating common network and wireless issues
  • Understand the purpose of packet captures and navigate basic Wireshark
Key Answer — Layered Troubleshooting Sequence
L1 Physical: Is the AP visible from the switch? Cable, link LEDs, port status. L2 Association: Can clients associate to the SSID? Open SSID should auto-connect. L3 DHCP: Did the client get an IP in the correct subnet? 169.254.x.x indicates DHCP failure. L3 Internet: Does SW1 have enable ipforwarding vlan Default? Are modem static routes pointing to SW1? Does the trunk port allow DHCP reach-back for SW2 clients?
Key Answer — Purpose of Packet Capture
Ping/traceroute give high-level binary answers ("works or doesn't"). Packet capture gives frame-level visibility: exact payloads (unencrypted), MAC/IP headers, VLAN tags, QoS priority markings, TCP sequence numbers, retransmissions, timing patterns. Enables root-cause diagnosis and trend analysis (congestion, performance degradation, anomalies) that high-level tools cannot reveal.
Key Answer — Where to Capture for Return-Path Diagnosis
Port 1 (uplink to modem): Confirm outbound ICMP Echo Requests leave SW1 and that Echo Replies arrive back from 8.8.8.8. Missing inbound → modem/internet issue. Port 3 (AP port): Confirm outbound requests from VLAN 30 client reach SW1 AND that Echo Replies (tagged VLAN 30) return to the client. Missing inbound on port 3 while present on port 1 → ipforwarding vlan Default disabled. In Wireshark filter by ip.dst == 10.30.0.150 to isolate your client's traffic.

Packet Capture Method — Reference

Two-capture-point diagnostic logic for return-path problems

Capture Point 1 — Port 1 (Uplink)
Mirror Port 1 to a Wireshark capture device. Run simultaneous traceroute 8.8.8.8 from a VLAN 30 client.
Outbound ICMPEcho Request → 8.8.8.8
Inbound ICMPEcho Reply from 8.8.8.8
If IN missingModem / ISP issue
If both presentUplink path OK
Capture Point 2 — Port 3 (AP Port)
Mirror Port 3 to a Wireshark capture device. Same traceroute running from VLAN 30 client.
Outbound (VLAN 30)Client → 8.8.8.8
Inbound (VLAN 30)Echo Reply → client
If IN missingipforwarding vlan Default
If both presentL3 forwarding OK
✓
Diagnostic logic matrix:
Port 1 (OUT ✓ IN ✗) → Modem/ISP problem, verify static routes.
Port 1 (OUT ✓ IN ✓) + Port 3 (OUT ✓ IN ✗) → enable ipforwarding vlan Default missing.
Port 1 (OUT ✓ IN ✓) + Port 3 (OUT ✓ IN ✓) → Not a routing issue; check DNS, client isolation ACL, firewall.
◆
Wireshark filter tips:
ip.src == 10.30.0.150 or ip.dst == 10.30.0.150 — isolate your client's traffic.
icmp — show only ping traffic.
vlan.id == 30 — show only VLAN 30 tagged frames.
bootp — show DHCP traffic.
wlan.fc.type_subtype == 0x04 — show Probe Requests (discovery debugging).

Seven permanent takeaways from today's lab — carry into every future build

01

The Karl Rule — enable ipforwarding vlan Default

SW1 owns the Default VLAN interface (192.168.0.28/24), which is the path to the modem and internet. Without ipforwarding enabled on Default VLAN, traffic from VLAN 10/20/30 cannot cross into the Default VLAN to reach the modem. Clients get DHCP and can ping the gateway but are completely ghosted by the internet. Must be in every SW1 config.

enable ipforwarding vlan Default
02

DHCP Reach-Back — include Port 10 in enable dhcp command

SW1's DHCP server only listens on ports explicitly listed in the enable dhcp command. If only Port 5 is listed, SW1 ignores DHCP requests arriving from SW2 via the trunk on Port 10. Clients on SW2 get 169.254.x.x self-assigned IPs. Always include the trunk port for all data VLANs.

enable dhcp ports 5,10 vlan VLAN_10_Corporate_Wired
03

VLAN 1 Loop Triangle — delete Port 10 from Default VLAN

Both switches are connected to the modem via Port 1 on VLAN 1. If VLAN 1 is also on the Port 10 trunk between the switches, a loop triangle forms: SW1 ↔ Modem ↔ SW2 ↔ SW1. Without STP blocking a path, this creates a broadcast storm. Always remove VLAN 1 from the inter-switch trunk port on both switches.

configure vlan Default delete ports 10
04

AP Onboarding — Default VLAN untagged on Port 3

An AP3000 boots with no config — it sends an untagged DHCP request. The switch assigns untagged traffic to whatever VLAN owns that port untagged. Port 3 must have VLAN 1 (Default) untagged so the AP gets a 192.168.0.x management IP from the modem and can reach XIQ. VLAN 20 and 30 ride tagged on the same port for SSID traffic.

configure vlan Default add ports 3 untagged
05

Port Assignments Must Be Explicit — EXOS Does Not Auto-Assign

Creating a VLAN and assigning an IP does not automatically associate ports to that VLAN. Port add commands must be run explicitly. VLANs showed 0 active ports during initial testing because port assignments were omitted from the build sequence.

configure vlan VLAN_10_Corporate_Wired add ports 5 untagged
06

dhcp-options default-gateway ≠ ipaddress on the switch

configure vlan VLAN_10 dhcp-options default-gateway 10.10.0.1 only tells DHCP clients what gateway IP to use. It does NOT assign that IP to the switch. For the switch to own the gateway IP and respond to ARP, you must separately run configure vlan VLAN_10 ipaddress 10.10.0.1/24.

configure vlan VLAN_10_Corporate_Wired ipaddress 10.10.0.1/24
07

MacBook Multi-Interface Conflict

MacBook with both Wi-Fi and Ethernet active creates a routing table conflict. The correct interface for the wired lab connection was en7 (USB-C Ethernet adapter), not en0. Always verify the interface with networksetup -listallhardwareports and disable Wi-Fi during wired-only testing.

route get default → check gateway and interface
