Data Flow

Complete End-to-End Data Flow — iPhone to google.com

The diagram below shows the full path a packet takes from the iPhone to Google and back, with both failure points annotated exactly where they occurred and how each was resolved.

Data Flow

Outbound Path — iPhone to Google

The outbound direction works without any special configuration because the switch has a default route to the home router. Every hop knows what to do.

HopDeviceWhat HappensPacket State
1 iPhone User opens Safari → google.com. iPhone looks up DNS (8.8.8.8). Traffic destined for 8.8.8.8. Sends to default gateway 10.30.0.1. src=10.30.0.102 · dst=8.8.8.8
2 AP3000 (port 5) AP receives the WiFi frame from the iPhone on the Guest SSID. Tags it with 802.1Q VLAN 30. Sends the tagged Ethernet frame up the trunk to the switch port 5. 802.1Q tag 0x8100 VLAN 30 added
3 5320 — Guest VLAN 30 Switch receives tagged frame on port 5. Strips the VLAN tag. Passes to Guest VLAN interface (10.30.0.1). ipforwarding ON — switch routes the packet. src=10.30.0.102 · dst=8.8.8.8
4 5320 — Routing engine Looks up 8.8.8.8 in routing table. No specific route. Matches default route: 0.0.0.0/0 via 192.168.0.1 on Default VLAN. Forwards out port 1. Next-hop: 192.168.0.1 · exit: port 1
5 Quantum Fiber modem Receives packet from LAN. Source IP is 10.30.0.102. Applies NAT: rewrites source to public WAN IP (71.212.x.x). Records NAT mapping: WAN-port-X ↔ 10.30.0.102. Forwards to internet. src=71.212.x.x · dst=8.8.8.8
6 Internet / Google Packet arrives at 8.8.8.8. Google processes the request. Sends reply to the WAN IP (71.212.x.x). src=8.8.8.8 · dst=71.212.x.x
Data Flow

Return Path — Google to iPhone (Two Failure Points)

The return path is where both issues lived. The outbound direction only requires a default route — but the return path requires specific knowledge at two devices.

HopDeviceWhat Needs to HappenBefore FixAfter Fix
1 Quantum Fiber De-NAT the reply: swap dst from 71.212.x.x back to 10.30.0.102. Now needs to DELIVER the packet to 10.30.0.102. Works (NAT table intact) Works
2 Quantum Fiber — routing Look up 10.30.0.102 in routing table. Where is 10.30.0.0/24? If no route exists, the modem drops the packet — it has no idea where that subnet lives. DROPPED — no route to 10.30.0.0/24 ✗ Static route: 10.30.0.0/24 → 192.168.0.24 ✓
3 5320 — Default VLAN Packet arrives at 192.168.0.24 (Default VLAN interface). Destination is 10.30.0.102. Switch must ROUTE it from Default VLAN to Guest VLAN 30. Requires ipforwarding ON. DROPPED — ipforwarding OFF on Default VLAN ✗ enable ipforwarding vlan Default ✓
4 5320 — Guest VLAN Switch looks up 10.30.0.102 in ARP table. Finds it on port 5 (AP2). Sends packet out port 5 tagged VLAN 30. Never reached Delivered to AP ✓
5 AP3000 → iPhone AP strips VLAN 30 tag. Delivers frame to iPhone over WiFi on Guest SSID. Never reached iPhone receives reply ✓
Why the Switch Ping Worked But iPhone Didn't

When the switch runs ping 8.8.8.8 from 10.30.0.1, the reply from Google is addressed to 10.30.0.1 — the switch's own interface. The switch receives it on the Default VLAN interface (192.168.0.24) and recognises the destination as itself. No routing needed — the packet is accepted locally.

When the iPhone sends traffic, the reply is addressed to 10.30.0.102 — a client behind the switch. The switch receives it on Default VLAN but now needs to route it onwards to the Guest VLAN. That requires ipforwarding. Two completely different code paths in the OS — one works, one doesn't.

Root Cause 1

Issue 1 — Missing Static Route on Quantum Fiber

The Problem

The Quantum Fiber modem's routing table only knew about its own subnets (192.168.0.0/24 and its WAN ranges). It had no entry for 10.20.0.0/24 or 10.30.0.0/24. After de-NATting a reply from Google, it had no way to forward the packet to the correct device — so it dropped it silently.

Diagnosis

TestResultWhat It Proved
ping 8.8.8.8 from 10.30.0.1 on switch0% lossOutbound works AND switch's own address gets replies (no routing needed on return)
iPhone tries 1.1.1.1 in SafariUnreachableReturn traffic for a CLIENT address (not switch interface) fails — routing problem upstream
ping 10.30.0.102 from switchSuccessLocal network is fine — problem is at the upstream boundary

Fix Applied

Two static routes added on the Quantum Fiber modem admin UI (Advanced → Static Routing):

DestinationMaskGatewayPurpose
10.20.0.0255.255.255.0 (/24)192.168.0.24Corporate WiFi clients → route replies to switch
10.30.0.0255.255.255.0 (/24)192.168.0.24Guest WiFi clients → route replies to switch
Note — First Attempt Failed

The first attempt at adding the routes used the wrong gateway IP. The Quantum Fiber UI accepted the entry but the route was either rejected silently or saved with incorrect data. Re-entering with gateway 192.168.0.24 (the switch management IP on Default VLAN) resolved the issue. Always verify routes appear in the static routes table after saving.

Root Cause 2

Issue 2 — ipforwarding Disabled on Default VLAN

The Problem

After the static route was added, return traffic successfully reached the switch at 192.168.0.24 (the Default VLAN interface). But the Default VLAN had ipforwarding disabled — meaning the switch could accept traffic for itself on that interface but could not route traffic that arrived there to another VLAN. The packet was silently dropped.

Evidence from show ipconfig

Interface    IP Address          Flags
Corporate    10.20.0.1       /24 EUf--R--g---    ← f = ipforwarding ON ✓
Default      192.168.0.24    /24 EU---R--gB--    ← NO f = ipforwarding OFF ✗
Guest        10.30.0.1       /24 EUf--R--g---    ← f = ipforwarding ON ✓

Why This Matters

Scenarioipforwarding needed?Works without it?
Switch pings 8.8.8.8 from 10.30.0.1 (its own interface)No — reply is TO the switch itselfYes ✓
iPhone at 10.30.0.102 sends traffic to internetYes — reply arrives on Default VLAN, must be routed to Guest VLANNo ✗
SSH to switch from Mac (192.168.0.x)No — management traffic stays on Default VLANYes ✓

Fix Applied

enable ipforwarding vlan Default
save configuration

With forwarding enabled, the Default VLAN interface can now act as an inter-VLAN routing gateway. Return traffic arriving at 192.168.0.24 destined for 10.30.0.102 is routed to the Guest VLAN interface and delivered to the iPhone via port 5.

Reference

Concept Glossary

TermPlain Language Definition
ipforwardingA switch setting that allows a VLAN interface to act as a router — forwarding packets that arrive on that interface to a different subnet. Without it, the switch only accepts traffic addressed to itself on that interface.
Static routeA manual entry in a router's routing table that says "to reach subnet X, send traffic to gateway Y." Required when a router doesn't automatically know where a subnet lives.
NAT (Network Address Translation)A router function that rewrites the source IP of outbound packets from a private address (10.x.x.x) to the router's public WAN IP. Return traffic is rewritten back. Allows many private devices to share one public IP.
Default routeA catch-all routing entry (0.0.0.0/0) — "if you don't know where to send a packet, send it here." The switch uses this to forward all internet-bound traffic to the Quantum Fiber modem.
802.1Q VLAN taggingA standard for adding a 4-byte tag to an Ethernet frame to identify which VLAN it belongs to. The AP tags client traffic when it leaves the AP toward the switch. The switch uses the tag to route traffic to the correct VLAN interface.
Return pathThe route a reply packet takes from the internet back to the original sender. More complex than outbound because every device along the path must know how to reach the original private IP address.
DHCPProtocol that automatically assigns an IP address, gateway, and DNS server to a device when it joins the network. The 5320 is the DHCP server for Guest (10.30.0.x) and Corporate (10.20.0.x). The Quantum Fiber modem is the DHCP server for the Default VLAN (192.168.0.x).
Inter-VLAN routingThe process of forwarding traffic between two different VLANs through a Layer 3 switch. Requires ipforwarding to be enabled on the VLAN interfaces involved.
