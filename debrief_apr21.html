SLIDE 05 The DHCP Reach-Back Flow
S3 L3 RoutingS6 TroubleshootingLab Build
1. The Ask: Client on AP (connected to SW2) sends DHCP Discover.
2. The Transit: SW2 has no DHCP server — request travels tagged across Port 10 trunk.
3. The Catch: SW1 (DHCP Server) MUST have DHCP listening on Port 10, or the request is silently dropped.
The SnagWithout adding port 10 to this command, SW1 ignores the trunk and SW2 clients are left stranded! (169.254.x.x APIPA addresses)

SLIDE 06 The Snag: Internet Ghosting
S6 TroubleshootingS3 L3 RoutingKarl Rule
Symptoms: Clients connected · DHCP assigned 10.x.x.x IPs · Pings to 10.x.x.1 gateways succeeded · Safari timed out — no internet.

The Mystery: IPv4 Forwarding was enabled on VLANs 10, 20, 30. SW1 could ping the internet. The modem had static routes pointing back. Where were the packets dying?

SLIDE 07 The Fix: The "Karl Rule"
Karl RuleS6 TroubleshootingS3 L3 Routing
What We Thought: Traffic leaves data VLAN → goes to modem → modem knows how to come back.
What the Switch Did: Return traffic hits Default VLAN (modem uplink). Switch checks Default VLAN state — routing disabled — explicitly drops packet.
Root Cause: IPv4 Forwarding disabled on Default VLAN.
The Karl Rule"You must enable IP Forwarding on EVERY created interface in the fab — not just the data VLAN!"
Apr 22 LinkS6 Q2 verbatim: "Isn't it IP forwarding on Default?" → Claude: "YES — exactly! The Karl Rule." Also S1 Q3: "My problems have been in the forwarding and the default forwarding that I hadn't done."
→ Session 6 Q2: Karl Rule Identification   → Session 1 Q3: L3 Troubleshooting
Config Pushed — The Karl Rule Fix (SW1)
# The fix — ipforwarding on EVERY routed VLAN including Default
enable ipforwarding vlan VLAN_10
enable ipforwarding vlan VLAN_20
enable ipforwarding vlan VLAN_30
enable ipforwarding vlan Default    # ← THE KARL RULE — this was missing
# Default route to modem (SW1 only — SW2 has no routes)
configure iproute add default 192.168.0.1
# Verify — all VLANs should show IPv4 Forwarding: Enabled
show ipconfig

SLIDE 12 Technical Field Notes: Strict EXOS ACL Syntax
S6 TroubleshootingLab Build
The Mistake: Cisco-style one-liner ACLs fail on EXOS 5320.
Apr 22 LinkS6 Q3 — Packet capture purpose: "What can it show that ping/traceroute cannot?" ACL drops are invisible to ping — only a packet capture reveals silently dropped traffic.
→ Session 6 Q3: Packet Capture Purpose
Config Pushed — EXOS ACL Syntax (Correct vs Wrong)
# ✗ WRONG — Cisco-style one-liner FAILS on EXOS
access-list 100 deny ip 10.30.0.0 0.0.0.255 10.10.0.0 0.0.0.255

# ✓ CORRECT — EXOS requires named entry structure with semicolons
create access-list guest_isolation "entry deny_guest_to_corp {
  if {
    source-address 10.30.0.0/24;
    destination-address 10.10.0.0/24;
  } then { deny; }
} entry permit_all {
  if { } then { permit; }
}"
# Apply to VLAN 30 ingress — catch traffic at the edge
configure access-list guest_isolation vlan VLAN_30 ingress
# Critical: always end with permit all — implicit deny kills internet!
SLIDE 13 Technical Field Notes: OWE Transition Mode
S4 WirelessS5 XIQ OpsLab Build
The Goal: Secure Guest network without a shared password using Enhanced Open (OWE).
The Snag: Legacy devices don't understand OWE — they fail to connect to the SSID entirely.
The Solution: Enable OWE Transition Mode.
How It Works: AP broadcasts a standard open network alongside a hidden OWE network. Capable devices auto-migrate to encrypted channel; legacy gear stays on open.
Apr 22 LinkS4 — WiFi security progression: WPA2 → WPA3-SAE → OWE (for open networks). KK covered SAE Dragonfly in depth. OWE is the open-network equivalent — same philosophy: no password transmitted, forward secrecy.

Critical Field Lessons — Errors That Bit Us
THE KARL RULE
enable ipforwarding vlan Default
Without this, return traffic from the modem is dropped at the unrouted Default VLAN. Internet reaches SW1 on Port 1 (Default) — but replies can't route back because Default has no ipforwarding. Symptom: clients ping 192.168.0.1 (modem) but can't reach 8.8.8.8. Fix: enable ipforwarding on EVERY VLAN that carries routable traffic, including Default.
DHCP REACH-BACK — PORT 10 MUST BE IN DHCP ENABLE
enable dhcp ports 3,10 vlan VLAN_20
SW2 clients send DHCP Discover → Port 10 trunk → SW1. SW1's DHCP server only listens on ports listed in the enable command. If Port 10 is omitted, SW2 clients never get an IP. Symptom: SW1-side clients get IPs, SW2-side clients stuck at APIPA (169.x). Fix: include port 10 in every DHCP enable command for VLANs used across the trunk.
LOOP TRIANGLE RISK — DELETE DEFAULT FROM PORT 10
configure vlan Default delete ports 10
If Default VLAN remains on Port 10 (the inter-switch trunk), a loop forms: Modem → SW1 Port 1 → Default VLAN → Port 10 → SW2 → back to modem. This creates a broadcast storm. The Default VLAN (192.168.0.x range) should ONLY exist on Port 1 (WAN uplink) and Port 3 (AP lifeline). Never on the inter-switch trunk.
AP LIFELINE — DEFAULT UNTAGGED ON PORT 3
configure vlan Default add ports 3 untagged
An AP boots as a dumb device — no VLAN tag, just a DHCP request on the wire. If Port 3 (the AP port) doesn't have Default VLAN untagged, the AP can't reach XIQ to get its config. Once XIQ pushes the config, the AP knows to use tagged VLANs 10/20/30 for SSIDs. The untagged Default is the bootstrapping lifeline.

Modem Static Routes
Return Path Configuration — Without This, Nothing Works
Why modem routes are required: SW1 knows all subnets (it has SVIs). The modem does NOT. When a guest on 10.30.0.x pings Google, the reply comes back to modem from ISP. The modem looks at the source (10.30.0.x) and doesn't know where to send it — unless you add a static route pointing all three subnets to SW1's WAN IP.

PURE L2 · L6 The Verification Challenge: AP2 Phone-Home
S6 TroubleshootingS4 WirelessS5 XIQ Ops
The Problem: With Port 1 physically disconnected on SW2, the direct path to the modem's DHCP server is gone. How does AP2 (connected to SW2 Port 3) acquire its management IP address from the Quantum Fiber modem and establish its connection back to ExtremeCloud IQ (XIQ)?

The Answer: Default VLAN tagged on the Port 10 trunk. AP2's untagged management traffic enters SW2 Port 3, gets an 802.1Q tag applied, travels the trunk to SW1, reaches the modem via SW1 Port 1, gets a 192.168.0.x IP, then phones home to XIQ.
Key InsightThe AP doesn't need a direct path to the modem — it needs a routable path. Once Default VLAN is on the Port 10 trunk, SW2 provides that routable path through SW1.

PURE L2 · L11 Lesson 08: Pure L2 Means a Single Uplink
S2 L2 SwitchingS1 OSI / L1–L3Karl Rule
"The dual-path physical architecture was the true root cause of the broadcast storm, not VLAN 1 itself."
You cannot maintain two active, parallel uplinks to an unmanaged modem without Spanning Tree Protocol (STP). Secondary access switches must rely entirely on the primary L3 core for upstream routing.

The VLAN 1 delete was a symptom fix, not the root fix. Deleting Default from Port 10 broke the loop — but it also broke AP2's management path and left the dual-uplink physical risk in place. The real fix is eliminating one of the physical uplinks.
Lesson Codified"If SW2 has a direct wire to the modem AND a wire to SW1, you have a loop by design — regardless of VLAN config. VLAN config can mask the loop temporarily, but only disconnecting the physical cable eliminates it."
