SLIDE 05 The DHCP Reach-Back Flow
S3 L3 RoutingS6 TroubleshootingLab Build
1. The Ask: Client on AP (connected to SW2) sends DHCP Discover.
2. The Transit: SW2 has no DHCP server — request travels tagged across Port 10 trunk.
3. The Catch: SW1 (DHCP Server) MUST have DHCP listening on Port 10, or the request is silently dropped.
enable dhcp ports 3,10 vlan VLAN_10 enable dhcp ports 3,10 vlan VLAN_20 enable dhcp ports 3,10 vlan VLAN_30
The SnagWithout adding port 10 to this command, SW1 ignores the trunk and SW2 clients are left stranded! (169.254.x.x APIPA addresses)

SLIDE 07 The Fix: The "Karl Rule"
Karl RuleS6 TroubleshootingS3 L3 Routing
What We Thought: Traffic leaves data VLAN → goes to modem → modem knows how to come back.
What the Switch Did: Return traffic hits Default VLAN (modem uplink). Switch checks Default VLAN state — routing disabled — explicitly drops packet.
Root Cause: IPv4 Forwarding disabled on Default VLAN.
enable ipforwarding vlan Default
The Karl Rule"You must enable IP Forwarding on EVERY created interface in the fab — not just the data VLAN!"

SLIDE 12 Technical Field Notes: Strict EXOS ACL Syntax
S6 TroubleshootingLab Build
The Mistake: Cisco-style one-liner ACLs fail on EXOS 5320.
access-list 100 deny ip 10.30.0.0 0.0.0.255 10.10.0.0 0.0.0.255 ✗ FAILS
The Reality — EXOS requires entry structure:
entry deny_guest_to_corp { if { source-address 10.30.0.0/24; destination-address 10.10.0.0/24; } then { deny; } } entry permit_all { if {} then { permit; } }
CRITICAL RulesMust declare entry <name> { }. Conditions inside if { }. Every action ends with semicolon. Must include permit all at end — implicit deny kills internet access.

SLIDE 13 Technical Field Notes: OWE Transition Mode
S4 WirelessS5 XIQ OpsLab Build
The Goal: Secure Guest network without a shared password using Enhanced Open (OWE).
The Snag: Legacy devices don't understand OWE — they fail to connect to the SSID entirely.
The Solution: Enable OWE Transition Mode.
How It Works: AP broadcasts a standard open network alongside a hidden OWE network. Capable devices auto-migrate to encrypted channel; legacy gear stays on open.

Tech Blueprint Doc · §8
Alex Vault
Critical Field Lessons — Errors That Bit Us
THE KARL RULE
enable ipforwarding vlan Default
Without this, return traffic from the modem is dropped at the unrouted Default VLAN. Internet reaches SW1 on Port 1 (Default) — but replies can't route back because Default has no ipforwarding. Symptom: clients ping 192.168.0.1 (modem) but can't reach 8.8.8.8. Fix: enable ipforwarding on EVERY VLAN that carries routable traffic, including Default.
DHCP REACH-BACK — PORT 10 MUST BE IN DHCP ENABLE
enable dhcp ports 3,10 vlan VLAN_20
SW2 clients send DHCP Discover → Port 10 trunk → SW1. SW1's DHCP server only listens on ports listed in the enable command. If Port 10 is omitted, SW2 clients never get an IP. Symptom: SW1-side clients get IPs, SW2-side clients stuck at APIPA (169.x). Fix: include port 10 in every DHCP enable command for VLANs used across the trunk.
LOOP TRIANGLE RISK — DELETE DEFAULT FROM PORT 10
configure vlan Default delete ports 10
If Default VLAN remains on Port 10 (the inter-switch trunk), a loop forms: Modem → SW1 Port 1 → Default VLAN → Port 10 → SW2 → back to modem. This creates a broadcast storm. The Default VLAN (192.168.0.x range) should ONLY exist on Port 1 (WAN uplink) and Port 3 (AP lifeline). Never on the inter-switch trunk.
AP LIFELINE — DEFAULT UNTAGGED ON PORT 3
configure vlan Default add ports 3 untagged
An AP boots as a dumb device — no VLAN tag, just a DHCP request on the wire. If Port 3 (the AP port) doesn't have Default VLAN untagged, the AP can't reach XIQ to get its config. Once XIQ pushes the config, the AP knows to use tagged VLANs 10/20/30 for SSIDs. The untagged Default is the bootstrapping lifeline.
Tech Blueprint Doc · §9
Modem Static Routes
Return Path Configuration — Without This, Nothing Works
Why modem routes are required: SW1 knows all subnets (it has SVIs). The modem does NOT. When a guest on 10.30.0.x pings Google, the reply comes back to modem from ISP. The modem looks at the source (10.30.0.x) and doesn't know where to send it — unless you add a static route pointing all three subnets to SW1's WAN IP.
Required Static Routes on Modem (192.168.0.1)
Destination Subnet Mask Gateway VLAN

10.20.0.0255.255.255.0192.168.0.28VLAN 20 — Corp Wireless
10.30.0.0255.255.255.0192.168.0.28VLAN 30 — Guest
SW1 WAN IP = 192.168.0.28 (assigned by modem DHCP or set static)
