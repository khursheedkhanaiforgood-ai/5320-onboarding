ADR-006 Three-Layer Data Architecture pgvector for semantic search + adjacency tables (→Neo4j) for graph traversal + standard PostgreSQL for structured data. All in one Railway Postgres instance. No single layer solves all three query types needed: "what feature is the user asking about?" (semantic), "what must I configure before PPSK?" (graph traversal), "how many bins are 75% verified?" (SQL aggregate). Each layer is best-in-class for its query type. pgvector-only (can't do dependency chains); pure graph DB (loses semantic search); pure SQL (no efficient traversal); separate services for each (ops overhead on Day 1) Session 2 →
ADR-007 Graph Implementation Path Week 1: PostgreSQL adjacency tables designed graph-compatible from Day 1. Week 2+: migrate to Neo4j AuraDB free tier. Migration = data export + import, not schema redesign. Neo4j adds a new service to provision on Day 1. Adjacency tables in PostgreSQL give graph semantics immediately with zero additional infrastructure. When traversal queries get complex (multi-hop dependencies), Neo4j's native Cypher pays off. Start simple, graduate when needed. Neo4j AuraDB from Day 1 (Day 1 complexity budget too high); NetworkX in-memory only (not persistent); pure SQL recursive CTEs forever (unreadable at depth >3 hops) Session 2 →
ADR-008 Agent Orchestration LangGraph StateGraph. Nodes = agents, edges = conditional routing. Supports cycles (retry loops), async execution, checkpointing. The agent workflow has conditional branches (gap detected? → fallback; coverage ≥ 75%? → switch mode), retry loops (fallback tier escalation), and async side effects (heat map update, bibliography append). LangGraph handles all three natively; a custom state machine requires writing that infrastructure from scratch. Custom Python state machine (more code, less robust); CrewAI (role-based, less routing control); AutoGen (heavier, Microsoft-opinionated); simple function chain (no conditional routing) Session 2 →
ADR-009 Fallback Pattern 3-tier ported from Cisco→EXOS agent: Tier 1 bin-scoped DB + graph, Tier 2 full semantic search, Tier 3 web crawl → user prompt → write back to DB. The Cisco→EXOS fallback was battle-tested across hundreds of queries. No-match cases are critical for a system mapping a product (EP1) whose docs may lag reality. User prompt → DB contribution loop means the system learns from every gap it encounters. 2-tier only (no web crawl, loses new discoveries); fail gracefully with "not found" (doesn't improve over time); ask user immediately (skips DB — wastes existing knowledge) Session 2 →
ADR-010 31 Bins (expanded from 25) Add 6 bins: #26 Policy Push/Deploy, #27 EP1 AI Canvas + NL Query, #28 EP1 Agent Manager, #29 Smart RF, #30 Troubleshooting + Diagnostics, #31 MSP + Multi-Tenancy. E2E KB_School scenario traced (build → push → observe → tune → automate → troubleshoot). Original 25 bins had zero coverage for the critical PUSH phase and zero coverage for all EP1-specific AI/automation capabilities. A new user hitting these gaps would be lost. Keep 25 (incomplete — new users fail at Push and Tune phases); collapse into existing bins (features buried, under-served in search and heat map) Session 3 →
ADR-011 Browser Automation Validation Playwright browser agents actually navigate XIQ and EP1 GUIs, screenshot each step, and verify that features exist at the stated nav path. API validation confirms data exists but cannot validate GUI navigation paths. This engine's primary output IS a navigation path. A user who follows a wrong path has a worse experience than no answer at all. Ground truth requires navigating the actual GUI. API-only validation (doesn't validate nav paths); manual testing only (not scalable for 31 bins × multiple features); trust docs without verification (EP1 docs may lag reality) Session 5 →
ADR-012 Dual-Stream Validation Both streams run in parallel: doc-based RAG/graph AND live Playwright browser validation. Every doc claim gets empirically tested. Divergences are written back to RAG corpus. No one has exhaustively validated EP1 features before. We do not know if the docs are accurate. Running both streams means the system can be launched (doc stream provides immediate coverage) while simultaneously building ground truth (browser stream validates each claim). Neither stream is trusted exclusively at the start. Doc-based only (docs may be wrong — EP1 is too new to trust blindly); browser-only (no searchable index, no RAG); sequential not parallel (too slow — 4-week window to July 1) Session 6 →
ADR-013 75% Phase Gate When db_coverage (COUNT browser-verified / COUNT all mappings) ≥ 0.75: switch from PARALLEL mode to RAG_VALIDATED mode. Browser agents become on-demand only. Running both streams indefinitely is expensive and adds latency. At 75% empirical validation, the doc-based patterns have been corrected sufficiently to be trusted for the remaining 25%. The 75% threshold balances confidence (enough corrections accumulated) against timeline (must reach this before July 1). Calibrated conservatively — not 50% (too few corrections) and not 90% (diminishing returns). 50% threshold (too few corrections — docs still unreliable); 90% (too late — diminishing returns, browser validation of niche features is expensive); no phase gate (browser agents run indefinitely — high cost, high latency forever) Session 6 →
ADR-014 Self-Calibrating Loop When browser agent finds EP1 nav path differs from doc claim: the correction is written to the RAG corpus as a new indexed document. RAG re-indexes. Future doc-based predictions improve automatically. A wrong doc claim is not just an error to flag — it is information. "EP1 moved this feature from path X to path Y" is a fact about EP1's naming convention that, once learned, improves predictions for all similar features. Without the corpus feedback loop, the same mistake would repeat across related features. Ignore divergence and return browser-verified answer only (docs stay wrong, pattern never learned); flag for manual review only (doesn't auto-improve); delete incorrect doc entry (loses context about what changed) Session 6 →
ADR-015 Dual Heat Map Two confidence values per bin: doc-confidence (RAG/graph estimate) + verified-confidence (browser-validated). Both displayed side-by-side. Gap between them = active investigation zone. A single composite score hides the most important signal: "is the doc estimate trustworthy for this bin?" Doc-green + verified-red is an alert — the docs say we have coverage but live testing disagrees. That gap is where EP1 has changed from what was documented. Single scores mask this; dual columns surface it immediately. Single composite confidence (masks doc vs empirical divergence); separate pages for each view (hard to compare); text-only status column (not scannable across 31 bins) Session 6 →

Graph Database Decision

Claude Code (Sonnet 4.6) — Graph Analysis

XIQ→EP1 is not a lookup table — it is a navigation graph. Features have dependency chains. Graph traversal answers "what is the full dependency chain to accomplish X in EP1" — vector search alone cannot.

Three tools, three purposes:

ToolWhat It IsUsed For
LangGraphPython library — agent workflow IS a directed graphOrchestrating 8 sub-agents. Control flow, NOT data storage.
pgvectorPostgreSQL extension — vector embeddingsSemantic search: "what feature is the user asking about?"
Neo4j / graph DBDatabase — knowledge stored as nodes + edgesXIQ↔EP1 relationships, nav trees, dependency chains. Data storage.

Implementation path decided: Week 1 = PostgreSQL adjacency tables (graph-shaped schema, no new service). Week 2+ = migrate to Neo4j AuraDB free tier. Schema is graph-compatible from Day 1 — migration = data export + import, not schema redesign.

Multi-Agent AI Architecture

Fallback Agent — Ported from Cisco→EXOS Agent

This is the same 3-tier search escalation that existed in cisco-en-cli-agent. In that agent: bin-scoped → full-DB → web crawl to Cisco/Extreme docs. Graph Education — Three-Layer Summary

Claude Code — Graph Education

A relational database stores rows and columns; relationships via JOINs. A graph database stores nodes (entities) and edges (relationships between them). The difference is what queries become natural:

QuestionSQLGraph (Cypher)
"Where is PPSK in EP1?"Single SELECTSingle MATCH
"What must I configure before PPSK?"Recursive CTE with 3-table JOINMulti-hop MATCH in one line
"Full workflow: AP policy push in EP1"App-layer path reconstructionPath traversal — native

Graph wins on path-finding and dependency chain queries — exactly what this engine solves.

Implementation path: Week 1 = PostgreSQL adjacency tables (no new service). Week 2+ = Neo4j AuraDB free tier (200K nodes / 400K edges — enough). Migration = data export + import. Schema designed graph-compatible from Day 1.

React explained (framed for Khursheed's background): React components = XIQ templates. A Network Policy template defines a pattern applied to many devices with different parameters. A React component defines a UI pattern rendered many times with different data (props). In plain HTML you copy 31 bin cards. In React you write 1 BinCard component and pass 31 data objects. Claude Code — Dual-Stream + Phase Gate Architecture

In sync. This is the right call. The core insight: because no one has validated EP1 exhaustively, you cannot trust the docs alone to be ground truth. Running both streams in parallel means every doc claim gets empirically tested — and every discrepancy between what the docs say and what the portal actually does becomes a NEW FINDING fed back into the RAG corpus. The system is self-calibrating.

Two Parallel Validation Streams

StreamSourceMethodConfidence TypeSpeed
Stream 1 — Doc-Based XIQ + EP1 published docs pgvector semantic search + graph traversal Estimated (doc-based) Fast, always on, no credentials
Stream 2 — Live Validated Actual XIQ + EP1 portals Playwright browser automation Empirical (ground truth) Slower, requires credentials, may hit downtime

Phase Gate Logic — 75% Threshold

-- Coverage metric (calculated live)
db_coverage = COUNT(mappings WHERE verified_by='browser_agent') / COUNT(all mappings)

if db_coverage < 0.75:
    run_mode = "PARALLEL"          ← debug / optimization guardrail
    # Every query triggers BOTH streams
    # Browser agent validates doc-based answer before returning to user
    # Discrepancy → NEW FINDING → written to corpus + DB
    # Heat map shows: doc-confidence (left) vs verified-confidence (right) side-by-side
    # Self-calibrating: doc patterns improve as ground truth accumulates

elif db_coverage >= 0.75:
    run_mode = "RAG_VALIDATED"     ← break out of guardrail
    # Primary: RAG/Graph answers directly (docs are now trusted)
    # Browser agents: spot-check only, on-demand or when confidence < 0.6
    # 75% verified → doc patterns are reliable enough for production use

The Self-Calibration Loop

  1. Doc stream generates mapping: "XIQ PPSK → EP1 path X, confidence 0.78"
  2. Browser agent navigates to path X in EP1 — takes screenshot
  3. If path X confirmed: doc confidence promoted to 1.0, written to DB
  4. If path X wrong (real path is Y): doc corpus updated with correction, DB updated, RAG re-indexes
  5. After ~30 corrections: RAG learns EP1's naming patterns and starts predicting correctly
  6. Coverage meter climbs. At 75%: guardrail lifts. Docs now trustworthy.

The system gets smarter with every divergence it finds. A wrong doc entry is not an error — it is a training signal.
