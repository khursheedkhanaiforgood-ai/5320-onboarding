1 Why Railway?

The original agent runs entirely on a local Mac — the operator reads instructions in a terminal and types commands in a screen session. This works perfectly for a solo operator. But it has one hard limit: only the person holding the Mac can see what's happening.

Railway unlocks the use case where a field tech is physically at the switch site and an engineer needs to guide them remotely — or where a manager needs to observe a live onboarding session from a browser.

ScenarioLocalRailway
Solo operator, own Mac✓ PerfectOverkill
Manager watches live from browser✗✓
Field tech onsite, engineer remote✗✓
No API key needed by field tech✗ Each user needs one✓ One key on server
Audit trail — central session log✗ Local only✓ Visible to all

2 Architecture

The key insight: you cannot access a physical USB serial port from a cloud server. The solution is a hybrid architecture — a tiny local bridge script handles the serial I/O, and everything else (state machine, Claude API, web UI) runs on Railway.

User's Mac (field tech)              Railway (cloud)
─────────────────────────            ──────────────────────────────
bridge.py                            FastAPI server
  ├── opens /dev/cu.usbserial-*       ├── session_manager.py
  ├── reads serial @ 115200 baud      │     ├── StateMachine (20 states)
  ├── streams lines → wss://          │     ├── ConsoleAnalyzer (Claude API)
  └── writes commands ← wss://        │     └── HTML report builder
                                      ├── /ws/bridge  (bridge endpoint)
                                      ├── /ws/ui      (browser endpoint)
                                      └── static/     (web UI)
                                            ├── index.html
                                            ├── style.css
                                            └── app.js

Browser (engineer / observer)
  └── opens https://app.railway.app
        ├── Left pane: live switch console
        ├── Right pane: agent instructions
        └── Prompts: yes/no buttons for XIQ confirm etc.

WebSocket Message Protocol

DirectionMessage TypePurpose
Bridge → Serverserial_lineOne decoded line from the switch console
Bridge → Serverbridge_helloIdentifies the bridge (hostname, port, baud)
Server → Bridgeserial_sendCommand to write to the serial port
Server → UIconsole_lineForward serial line to browser
Server → UIinstructionClaude's action/command/explanation
Server → UIstate_updateCurrent state machine state
Server → UIpromptAsk operator a yes/no question
Server → UIreport_readyHTML report as download blob
UI → Serverprompt_responseOperator's answer to a prompt
UI → Servermanual_commandCommand typed in the browser input bar

3 What Was Built

FileWhat it does
bridge/bridge.py~150 lines. Auto-detects serial port, opens pyserial, streams lines to Railway via WebSocket, writes received commands back to switch. Auto-reconnects with exponential backoff.
bridge/requirements.txtTwo packages only: pyserial + websockets
server/app.pyFastAPI entry point. Mounts /ws/bridge and /ws/ui WebSocket endpoints, serves static files, validates bridge token.
server/config.pyReads ANTHROPIC_API_KEY and BRIDGE_TOKEN from Railway env vars.
server/serial_proxy.pyAbstraction layer. Provides the same get_lines() / get_raw_buffer() interface as the original LogfileMonitor — so session_manager doesn't care that I/O goes over a network.
server/session_manager.pyAsync port of main.py. time.sleep() → await asyncio.sleep(). click.prompt() → WebSocket prompt messages. screen -X stuff → serial_proxy.send_command(). HTML report sent as download blob instead of saved to local disk.
server/static/index.htmlTwo-pane web UI. Left: scrolling console. Right: instruction card + prompt area.
server/static/app.jsVanilla JS WebSocket client. Handles all message types, renders instruction cards, shows prompt buttons, generates report download blob.
server/static/style.cssDark theme matching the existing config report style.
railway.tomlTells Railway: uvicorn server.app:app --host 0.0.0.0 --port $PORT
requirements.txt (root)Copy of server requirements at repo root — required for Nixpacks auto-detection.

Phase 3 — Deploy Debugging
K
Khursheed Khan#3

Are you testing them before you commit to Git and before it is picked up by Railway?

C
Claude Code

Honest answer: No — Docker and nixpacks CLI were not installed locally, so the Railway build environment could not be simulated. Three failed deployments resulted from guessing at the Nixpacks environment:

  • Attempt 1: railway.json with custom install phase → Railpack: no start command detected
  • Attempt 2: nixpacks.toml → ignored by Railway
  • Attempt 3: railway.toml with pip install → pip: command not found
  • Attempt 4: pip3 → pip3: command not found
  • Attempt 5: /opt/venv/bin/pip → still failed
  • Fix: removed custom install phase entirely, copied requirements.txt to repo root so Nixpacks auto-detected it, kept only startCommand in railway.toml. Build succeeded.

Lesson: Nixpacks works best when you let it auto-detect. Override only the start command, not the install phase.

K
Khursheed Khan#4

Successful deployment, now where is the webpage from Railway?

C
Claude Code

Instructed: Railway project → service → Settings → Networking → Generate Domain. Railway provides a public HTTPS URL. The web UI is then accessible at that URL from any browser.

5 Deploy Steps — Complete Reference

Server (Admin — done once)

  1. Go to railway.app → New Project → Deploy from GitHub
  2. Select repo khursheedkhanaiforgood-ai/5320-onboarding, branch feature/railway
  3. Railway → service → Variables → add ANTHROPIC_API_KEY and BRIDGE_TOKEN
  4. Railway → service → Settings → Networking → Generate Domain
  5. Share the URL and BRIDGE_TOKEN value with field techs

Bridge (Field Tech — run per session)

pip install pyserial websockets
python3 bridge/bridge.py --server wss://YOUR-APP.railway.app --token YOUR_TOKEN

Observer (Engineer — any browser)

https://YOUR-APP.railway.app

6 Running Today

The Railway deployment is live at:

https://5320-onboarding-production.up.railway.app

To connect a switch session:

pip install pyserial websockets
python3 ~/5320-onboarding/bridge/bridge.py \
  --server wss://5320-onboarding-production.up.railway.app \
  --token 5320bridge

Note: Change the BRIDGE_TOKEN set to 5320bridge — update this if you ever rotate the token in Railway → Variables.
