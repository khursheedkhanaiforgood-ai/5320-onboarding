5320 Onboarding Agent — Build Tutorial

A complete record of how an AI-assisted network switch onboarding agent was designed and built through live field testing on an Extreme Networks 5320-16P-2MXT-2X. Every prompt, every bug, every architectural decision — in sequence.

1. Project Goal

An Extreme Networks 5320-16P-2MXT-2X arrives from the factory running Fabric Engine Pre-GA firmware. To put it into production it must be migrated to EXOS SwitchEngine via Extreme’s cloud zero-touch provisioning platform (XIQ ZTP+). The process requires factory resetting the switch, letting it phone home to XIQ, receiving a firmware push, rebooting into EXOS, completing a setup wizard, and saving configuration.

The goal was to build a single-command AI agent that:

  • Watches the serial console in real time — no manual attention needed between steps
  • Detects each phase with local regex patterns in microseconds — no API latency during critical boot sequences
  • Calls Claude (Anthropic API) only on state transitions — plain-English instructions for the operator
  • Detects when a switch is already onboarded and pivots to a configuration verification and capture workflow
  • Produces a self-contained dark-themed HTML report with tabs, opened automatically in the browser

2. Architectural Principles

These four principles were established before a single line of code was written and held through every iteration:

Read-only on the wire

The agent never opens the serial port for writing. It tails a logfile written by a screen session that the human operator controls. This eliminates [Errno 16] Resource busy errors entirely — only one process (screen) ever holds the port exclusively.

Regex-first, Claude-second

A local 20-state machine classifies every console line with compiled regex patterns in microseconds. Claude is invoked via API only on state transitions — roughly once every few minutes during onboarding. Token cost is negligible; latency is invisible during fast phases like U-Boot and firmware install.

Human stays in control

The agent displays instructions; the human types commands in a separate Terminal window. Auto-send mode (screen -X stuff) is used only in the already-onboarded verification flow — a clearly bounded, read-only operation with no misconfiguration risk.

Single rolling buffer

Console output is stored in a 4000-character rolling deque. Only this window is sent to Claude — never the full session log. This bounds token spend and keeps context focused on the current switch state.

Key internal functions in main.py

FunctionPurpose
_wait_for_state()Blocks until state machine reaches a target state. Checks current state immediately before blocking — avoids hang when already in target.
_wait_for_settle()Waits for console activity to START, then for it to go quiet for settle seconds. Adapts to actual output speed — never fires on stale queue.
_wait_for_settle_escape()Like _wait_for_settle but sends Space automatically when --More-- is detected — used for show port configuration.
_find_screen_session()Uses lsof -t port + screen -ls to find the screen session ID holding the serial port.
_send_to_screen()Sends a command + Enter to a screen session via screen -X stuff.
_run_command()Full auto-send sequence: flush queue → blank CR → 400ms → flush → capture start_pos → send command → wait settle → read logfile.
_filter_port_config()Strips port lines with no active link from show port configuration output before HTML generation.
_parse_switch_info()Regex extraction of SysName, System Type, firmware, and MAC from show switch output for the HTML report header.
_build_html_report()Generates self-contained dark-themed HTML with switch metadata header, tabbed layout, and per-tab scrollable <pre> blocks.
_run_onboarded_verification()Full already-onboarded flow: login → XIQ banner → disable paging → 6 captures → filter → HTML → browser.

Already-Onboarded Detection Logic

_ONBOARDING_IN_PROGRESS_STATES

ZTD_MODE, DHCP_ACQUIRING, XIQ_CONNECTING, FIRMWARE_DOWNLOADING, FIRMWARE_INSTALLING
If any of these were seen during the session, fresh onboarding was happening.

_EXOS_ACTIVE_STATES

EXOS_LOGIN_PROMPT, EXOS_LOGGED_IN, EXOS_SETUP_WIZARD, EXOS_SAVE_CONFIG, ONBOARDED
Landing here without the onboarding states = already running EXOS when the agent connected.

The likely_already_onboarded property is a negative inference: absence of onboarding journey evidence + presence of an EXOS active state = already onboarded. The state machine records every state ever seen in _seen_states: set[SwitchState] to make this determination.

Root cause and fix

  • pyserial and screen both open the serial port with O_RDWR. Only one process can hold it exclusively at a time — an OS-level constraint.

Key Technical Decisions

DecisionAlternative ConsideredWhy This Approach
LogfileMonitor (tail logfile)Direct pyserial readOnly one process can hold the serial port exclusively. screen already owns it. Tailing the logfile is zero-conflict and lets the operator type freely.
Regex state machine → Claude on transitionCall Claude on every lineRegex runs in microseconds. Claude latency is 1–3 seconds. State machine filters 99% of lines — Claude only sees meaningful transitions. Token cost is negligible.
osascript to open TerminalManual instruction to operatorZero operator setup steps. Session ID tracked automatically. Stale sessions killed before opening.
_wait_for_settle (activity then quiet)time.sleep(N) fixed delayFixed sleep is either too short (misses slow output) or too long (slows every command). Settle-based wait adapts to actual output speed.
Blank CR + 400ms before each commandSend command directlyscreen drops the first character of a command sent immediately after the previous one. Without the delay: show ipconfig → how ipconfig.
Space to dismiss --More--Escape keyEscape aborts all remaining output — ports after the first page are lost before the filter can run. Space advances to the next page, capturing everything.
verification_done flagRe-check likely_already_onboardedThe property stays True permanently. Without the flag, every subsequent state transition re-enters the verification flow.
10s wait after username before sending passwordSend username + password in sequenceEXOS can accept admin with blank password instantly. If the password is sent after login completes, EXOS treats it as a command — “Invalid input”.

Test Coverage

21 unit tests in tests/test_state_machine.py — all passing throughout every iteration:

  • Initial state (UNKNOWN)
  • Individual detection for all major states: U-Boot, FE login, FE login blocked, FE logged in, ZTD, DHCP, XIQ connecting, firmware download, firmware install, EXOS boot, EXOS login, onboarded
  • Boot-complete flag lifecycle
  • likely_already_onboarded — three scenarios: detected correctly, not triggered after full ZTP+ flow, not triggered before EXOS states
  • Full 14-step onboarding sequence in order from U-Boot to ONBOARDED

The state machine was the one module that never broke across any change in the entire build. The test suite caught the duplicate pattern bug immediately in Phase 1.

python3 -m pytest tests/ -v
# 21 passed in 0.03s

Running the Agent Today

  1. Connect the USB-C serial console cable to the Mac and the switch console port (RJ45 labeled CON/CONSOLE)
  2. cd /path/to/5320-onboarding-agent
  3. source .venv/bin/activate
  4. python3 -m agent.main

The agent auto-detects the port, kills any stale session, opens a Terminal window with screen logging to /tmp/screenlog.0, and starts monitoring.
