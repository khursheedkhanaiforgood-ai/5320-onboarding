Root Cause — Full Explanation

What Broke

AP1 (IQ Engine OS) accumulated corrupted data plane state after a sequence of events on May 12-13:

  1. XIQ firewall policy deployed to AP1 (wrong rule order)
  2. Manual AP power cycles (×3–4 during debugging)
  3. XIQ firewall policy removed/redeployed
  4. QoS policy removed by boss → AP reset
  5. Additional XIQ policy deploys (PPSK, Corp, Guest user profiles modified)

After this sequence, AP1's data plane ended up in a state where it:

  • ✅ Broadcast frames wireless→wired: forwarded normally (DHCP, gratuitous ARPs)
  • ✅ Broadcast frames wired→wireless: forwarded (DHCP offers as broadcasts)
  • ❌ Unicast frames wired→wireless: silently dropped (ARP replies, ICMP, all data)

Why DHCP Masked It

iPhones and Macs set the BROADCAST flag in DHCP DISCOVER. This instructs the DHCP server to send OFFER and ACK as Ethernet broadcasts (dest MAC = ff:ff:ff:ff:ff:ff), not unicast. So DHCP completed successfully without unicast wired→wireless delivery. Clients got valid IPs, looked like everything was fine — until they tried to use the network.

Why Manual Resets Didn't Fix It

A power cycle alone causes the AP to reconnect to XIQ and re-download policy. But if the policy being pushed is in an inconsistent state (multiple partial deploys, mixed-version firewall/QoS objects), the AP rebuilds the same broken data plane. A XIQ-triggered reboot ensures XIQ pushes a single coherent policy snapshot to the AP at boot time, rebuilding the data plane cleanly.

Diagnostic Chain — The Four-Step Runbook

This is the definitive sequence for diagnosing "clients have DHCP but no internet" when SW1 routing is confirmed healthy:

StepCommandWhat It Proves
1 show fdb vlan <vlan> on SW1 Client MACs learned on correct port? If yes → wireless→wired works. The AP IS forwarding broadcast frames to SW1.
2 show iparp vlan <vlan> on SW1 ARP entries populated from client gratuitous ARPs? If yes → clients sent traffic post-DHCP. Still doesn't prove wired→wireless works.
3 ping vr VR-Default <client_ip> on SW1 100% loss despite valid ARP = SW1 is sending unicast toward AP1 but frames not reaching client. AP data plane suspect.
4 arp -n on MacBook Empty cache = smoking gun. MacBook's ARP requests reached SW1 (FDB proves it) but SW1's unicast ARP replies never arrived at MacBook = AP1 dropping wired→wireless unicast.
Important EXOS Gotcha
ping vr VR-Default 8.8.8.8 from <ip> — the from address MUST be one of SW1's own configured interface IPs. Using a client IP (e.g., 10.30.0.101) returns: "Error: Invalid from address supplied, make sure it exists." This means the "ping from SVI" test only proves SW1's own traffic flows — it does NOT simulate client-forwarded traffic end-to-end.

  • When multiple policy changes need to be made in XIQ (firewall, QoS, VLAN), batch them and do a single XIQ deploy + single XIQ-triggered AP reboot at the end.
  • Do NOT use manual power cycles mid-policy-change-sequence — the AP comes back with partial state.

Lessons Learned

1. DHCP Working ≠ Data Plane Healthy

DHCP uses broadcasts. Clients set the BROADCAST flag. The AP can have completely broken unicast delivery and DHCP will still work perfectly. Never use "clients have IPs" as proof that the AP data plane is healthy.

2. XIQ-Triggered Reboot > Power Cycle

A cold power cycle brings the AP back up with whatever state it can reconstruct. A XIQ-triggered reboot causes XIQ to push a fresh policy snapshot during the CAPWAP handshake. When the AP has accumulated partial/conflicting policy state, only the XIQ-triggered reboot rebuilds the data plane correctly.

3. arp -n on the Client Is the Definitive Test

If the client's ARP cache is empty after attempting to ping the gateway, the AP is dropping wired→wireless unicast. This single command cuts through all the routing, NAT, and firewall noise and points directly at the AP data plane.

4. The from Option in EXOS Ping Is Not Client Simulation

ping vr VR-Default 8.8.8.8 from 10.30.0.1 proves SW1's own traffic flows — not client-forwarded traffic. The source must be SW1's own interface IP. This test created a false confidence that the return path was working when it wasn't.

5. ARP Conflicts After Reboot

MacBooks cache their previous DHCP lease and send gratuitous ARPs on reconnection, potentially claiming an IP that the DHCP server re-assigned to a different device. After reboots in a lab, clear MAC/IP bindings in the DHCP server or wait for the ARP conflict to resolve before assuming ARP table data is canonical.

6. Socratic Diagnostic Method Works

Rather than guessing and rebooting, the Socratic layered approach — confirm each hop individually before moving up the stack — isolated the problem precisely. The key was asking "what is the ONE difference between SW1 pinging 10.30.0.1 (works) vs SW1 pinging the client (fails)?" That question led directly to the AP.

Deep Dive — Why the AP Broke and What the Reboot Fixed

Question — Khursheed

I need to understand this clearly. I got DHCP, client devices assigned IP addresses. I could ping .1 for 20/30/100. My return path was good on QF-Modem. Removed QoS / Firewall restrictions from PPSK. But nothing worked. Now, I could not ping my .101 on 20/30 from .28. So, are we saying that AP1 could not forward traffic put on port 3 by SW1 to client devices? If so, why not? What really happened when the 'fresh policy was pushed' just in AP1 to solve this.

What the AP actually does in the wired→wireless direction

When SW1 wants to deliver a unicast frame to a wireless client, the path is:

SW1 eth1/3  →  [tagged VLAN 30 unicast, dst MAC = client]  →  AP eth0
                                                                   ↓
                                             AP looks up dst MAC in station table
                                                                   ↓
                                             Retrieves PTK for that association
                                                                   ↓
                                             Encrypts frame with client's PTK
                                                                   ↓
                                             Transmits over-the-air to client

Broadcast/multicast (DHCP, gratuitous ARPs) follow a different path — they use the GTK (Group Temporal Key), one key shared across all clients on the BSS. GTK is stable, installed once, and does not depend on any specific client's association state.

Unicast uses the PTK (Pairwise Temporal Key) — established per-client during the 4-Way Handshake, stored in a station table entry keyed by client MAC.

Why your specific evidence proved exactly where the break was

ObservationWhat it proves
show fdb vlan Guest_30 → MacBook MAC on port 3AP IS forwarding wireless→wired (upstream works)
show iparp vlan Guest_30 → ARP entry for .101Client's ARP REQUEST (broadcast) reached SW1
ping 10.30.0.101 from SW1 → 100% lossSW1's unicast ARP REPLY never reached the client
arp -n on MacBook → emptyClient never received the unicast reply — AP dropped it silently
DHCP workedGTK path (broadcast) still functional

The break is exactly at the AP's station table lookup in the wired→wireless unicast path. AP receives the frame from SW1, looks up the destination MAC, and either cannot find it or the PTK entry is invalid — so it drops the frame. No error, no log, just silence.

What caused the station table corruption

Multiple XIQ policy changes while clients are associated — firewall add/remove, QoS add/remove — each cause XIQ to push a live CAPWAP config update. The AP applies the update while running, which forces a partial data-plane re-init: it tears down and rebuilds SSID security profiles, VLAN mappings, user profile enforcement.

If a client was mid-session during one of these partial rebuilds, the AP's association FSM can land in a state where:

  • The station table entry exists (so the client believes it is associated, and upstream/broadcast frames still work)
  • But the PTK is stale, null, or installed into the wrong hardware key slot in the crypto engine

The AP then silently drops any unicast frame addressed to that client MAC, because encryption would fail with an invalid key.

Why power cycling alone didn't fix it

AP reboots → reconnects CAPWAP → XIQ sees: "I know this AP, already provisioned"
           → pushes only delta updates from last known state
           → AP rebuilds data plane from the SAME corrupted policy basis
           → station table re-corrupts as clients re-associate under the same broken config

XIQ's "last known state" was itself the product of those sequential partial pushes — it was not clean.

What the XIQ-triggered reboot actually did differently

XIQ: Reboot command → AP shuts down completely
AP:  Comes back up cold — empty station table, empty crypto key tables, blank VLAN mappings
AP:  Reconnects CAPWAP to XIQ
XIQ: "This AP just rebooted via my command — send FULL policy, not delta"
XIQ: Pushes complete current config as a single coherent bundle:
       • SSID definitions
       • VLAN→BSS mappings
       • Security profiles (WPA2-AES, key management)
       • User profiles (firewall, QoS)
       • Radio settings
AP:  Initializes data plane from scratch against this clean config
AP:  SSIDs come up → clients re-associate → fresh 4-Way Handshakes
     → PTKs installed into clean crypto context → unicast delivery restored

Clients then go through a complete new 4-Way Handshake, PTKs are installed correctly into a freshly initialized key table, and the station table entries are clean from the start. The wired→wireless unicast path works again because the PTK lookup succeeds.

One-Sentence Summary

The AP's internal station table had valid entries (so association and broadcasts worked) but corrupted PTK key material (so unicast delivery failed silently) — accumulated across multiple live policy pushes; a XIQ-triggered reboot forced a full policy re-provisioning that rebuilt the data plane from a clean, coherent state.
