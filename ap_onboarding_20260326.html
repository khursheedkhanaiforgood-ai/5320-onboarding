Network Design

VLAN Routing & DHCP Architecture

Each wireless SSID maps to a VLAN. When a client connects, the AP tags its traffic with the VLAN ID and sends it to the 5320. The switch acts as the gateway for each VLAN, runs the DHCP server that assigns client IPs, and routes traffic to the internet via the home router.

How the Data Flow Works

StepWhat Happens
1. Client associatesPhone/laptop connects to Corporate, Guest, or IoT SSID on the AP3000
2. AP tags trafficAP3000 tags all frames from that client with the VLAN ID (20, 30, or 40) and sends them tagged up the trunk to port 3 or 5
3. DHCP requestClient broadcasts DHCP Discover — switch intercepts it on the correct VLAN interface and responds from its built-in DHCP server
4. IP assignedSwitch assigns the client an IP from the pool (e.g. 10.30.0.105 for a Guest client) with gateway 10.30.0.1 and DNS 8.8.8.8
5. RoutingClient sends traffic to its default gateway (10.30.0.1 = switch VLAN 30 interface). Switch routes to Default VLAN and forwards out port 1
6. NAT to internetHome router receives traffic, NATs the source IP to its WAN IP, and forwards to internet

Why This Happens

macOS ships with a modern OpenSSH client that removed ssh-rsa from the default list of accepted host key algorithms for security reasons (SHA-1 based). EXOS SwitchEngine 33.5.2 only offers ssh-rsa. The fix: explicitly re-enable it with command-line flags.

Fix — Add HostKeyAlgorithms Flags

ssh -o HostKeyAlgorithms=+ssh-rsa -o PubkeyAcceptedAlgorithms=+ssh-rsa admin@192.168.0.24

Lesson: EXOS Reserved VLAN Names

In EXOS SwitchEngine, Management and Mgmt are reserved keywords tied to the physical OOB management port. They cannot be used as names for data VLANs. Use descriptive alternatives: APMgmt, WifiMgmt, AccessMgmt.

The "Mgmt" VLAN (VID 10) in this deployment is permanently locked — it cannot be deleted, renamed, or have data ports added. AP management uses the Default VLAN for provisioning instead.

Phase 4

Port Configuration — Tagged vs Untagged

Concept: Untagged vs Tagged

Untagged — The port strips the 802.1Q VLAN tag before sending frames out. The connected device receives plain Ethernet and needs no VLAN awareness. Used for the native/provisioning VLAN. Only one untagged VLAN is allowed per port.

Tagged — Frames carry the 802.1Q VLAN tag intact. The connected device (the AP) must understand VLANs. Used for data VLANs that APs map to SSIDs. Multiple tagged VLANs per port are allowed.

Why Default VLAN untagged for AP provisioning? — The home router provides DHCP on 192.168.0.0/24 and connects to port 1 untagged on Default VLAN. APs on Default VLAN (untagged) get a 192.168.0.x IP from the home router, reach the internet, and connect to XIQ cloud for adoption. After adoption, XIQ pushes Corporate/Guest/IoT SSID config which uses the tagged VLANs.

PoE Class 4 Explained

Class 4 = IEEE 802.3at (PoE+), up to 30W allocated per port. Initial draw of 2–2.5W is normal during early boot sequence. Full AP3000 power consumption under load is typically 12–15W. The 5320's 180W budget supports up to ~12 simultaneous Class 4 APs at full load.

Phase 8

DHCP & IP Gateway Configuration

After both APs were adopted in XIQ and SSIDs were broadcasting, the next step was to give WiFi clients actual IP addresses. The 5320 runs a built-in DHCP server — each VLAN needs an IP address (its gateway), a DHCP address pool, a default gateway option, DNS, and DHCP enabled per port.

Important: XIQ Does Not Know About These CLI Changes

XIQ manages the switch by pushing configuration down. It has no mechanism to detect what you type manually at the CLI. If you push a Network Policy update from XIQ that covers VLANs, it will overwrite the DHCP/IP configuration below. Run save configuration after every CLI change. Long-term, re-enter this config into XIQ's Switch Policy so XIQ is the single source of truth.

Key EXOS DHCP Syntax Lessons

Wrong commands attempted
# These do NOT work on EXOS SwitchEngine:
configure dhcp-server vlan Guest low 10.30.0.100 high 10.30.0.200   ← does not exist
enable dhcp-server vlan Guest                                        ← does not exist
configure vlan Guest ipaddress 10.30.0.1/24                         ← error if IP already set
unconfigure vlan Guest ipaddress                                     ← must clear first
Correct EXOS DHCP syntax
# Correct pattern on EXOS SwitchEngine 33.5.2:
configure vlan <name> dhcp-address-range 10.X.0.100 - 10.X.0.200   ← dash separator (not "to" or "high")
configure vlan <name> dhcp-options default-gateway 10.X.0.1
configure vlan <name> dhcp-options dns-server primary 8.8.8.8
configure vlan <name> dhcp-lease-timer 3600
enable dhcp ports 3 vlan <name>                                     ← enable per-port, not per-VLAN

What Each Command Does

CommandWhat It Does
configure vlan Guest ipaddress 10.30.0.1/24Sets the switch's IP on the Guest VLAN interface — this becomes the default gateway that clients will use
enable ipforwarding vlan GuestAllows the switch to route packets that arrive on the Guest VLAN interface — without this, the switch drops any packet not addressed to itself
configure vlan Guest dhcp-address-rangeDefines the pool of IPs the DHCP server can hand out to clients (10.30.0.100 through 10.30.0.200 = 101 addresses)
configure vlan Guest dhcp-options default-gatewayTells clients "your gateway is 10.30.0.1" — the DHCP option 3 sent in every DHCP Offer and Ack
configure vlan Guest dhcp-options dns-server primary 8.8.8.8Tells clients which DNS server to use (Google Public DNS — works for any network that can reach the internet)
configure vlan Guest dhcp-lease-timer 3600IP lease duration in seconds (3600 = 1 hour). Client must renew at 30 min mark or get a different IP after 1 hour of absence
enable dhcp ports 3 vlan GuestActivates the DHCP server on port 3 for the Guest VLAN — the switch will answer DHCP Discover broadcasts arriving on that port/VLAN

Phase 9

Internet Connectivity — Diagnosis & Options

After DHCP was configured, a phone connected to the Guest SSID and successfully received an IP: 10.30.0.100, gateway 10.30.0.1, DNS 8.8.8.8. However, the phone had no internet access.

Root Cause — NAT Boundary

What Is Failing and Why

The phone's traffic leaves the 5320 sourced from 10.30.0.100. It reaches the Quantum Fiber router at 192.168.0.1. The Quantum Fiber router only knows how to NAT traffic from its own 192.168.0.0/24 subnet. It has no route back to 10.30.0.0/24 — so when Google sends a reply, the Quantum Fiber router drops it. The phone never receives the response.

What NAT Does (Plain Language)

Without NAT (current state)With NAT (needed)
Phone sends packet: from 10.30.0.100 to 8.8.8.8
Quantum Fiber router sees source 10.30.0.100
Google replies to 10.30.0.100
Quantum Fiber has no route to 10.30.0.x — reply dropped ✗		 Phone sends packet: from 10.30.0.100
NAT router rewrites source to 192.168.0.50 (its WAN IP)
Google replies to 192.168.0.50
NAT router rewrites back to 10.30.0.100 and forwards
Phone receives response ✓
Why the 5320 Cannot Do This

The 5320 is a layer-2/3 access switch. It can route between VLANs but it does not support configure ipmasquerade (NAT/PAT). This command does not exist on EXOS SwitchEngine. A separate router or firewall is needed at the boundary between the private 10.x.x.x subnets and the home router.

Three Options to Fix Internet Access

OptionWhatEffortPreserves 10.x subnets?
A — Add a router Place a second router (e.g. TP-Link ~$30) between the Quantum Fiber box and the 5320. It handles NAT for all your VLANs. WAN port gets 192.168.0.x; LAN side connects to 5320 uplink. ~20 min + hardware Yes ✓
B — DHCP relay Remove 10.x.x.x config from the VLANs. Configure iphelper on each VLAN to forward DHCP requests to the Quantum Fiber router (192.168.0.1). Clients get 192.168.0.x IPs — the home router already NATs those. Fastest fix, no hardware needed. ~15 min CLI No — clients on 192.168.0.x
C — ISP static routes Call Quantum Fiber and ask them to add static routes for 10.20.0.0/24, 10.30.0.0/24, and 10.40.0.0/24 pointing to 192.168.0.24. Clean enterprise solution — no extra hardware. Call ISP — unknown wait Yes ✓

DHCP Relay Commands (Option B)

If you choose Option B, run the following to remove the switch DHCP server and relay requests to the home router instead:

# Remove switch DHCP server for each VLAN
configure vlan Guest dhcp-address-range none
configure vlan Guest dhcp-options default-gateway none
disable dhcp ports 3 vlan Guest
disable dhcp ports 5 vlan Guest

# Forward DHCP requests to home router
configure vlan Guest iphelper add 192.168.0.1
configure vlan Corporate iphelper add 192.168.0.1
configure vlan IoT iphelper add 192.168.0.1
