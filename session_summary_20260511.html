EXOS → VOSS — The Four Principles

One Config, Four Structural Pillars

Every line in a VOSS running-config belongs to one of four layers. Understanding which line lives where is the mental model that replaces EXOS port-VLAN thinking. The SW2 config (KhKLab-SW-01, FabricEngine 9.3.2.0) was used as the working example throughout this session.

The analogy that holds: IS-IS builds the map. SPB drives the roads. I-SID is the postal code on every frame. Auto-Sense/FA is the on-ramp that assigns new arrivals to the right postal code automatically.

Principle 1

Brain

IS-IS — The Control Plane

"IS-IS builds the map. Every switch knows every other switch."

IS-IS is the routing protocol that runs over the SPB fabric. It learns who is connected to whom, builds a topology database, and tells SPB exactly which path to use. Without IS-IS, SPB is blind.

The identity lines — the two most important in the entire config:

router isis
  sys-name "KhKLab-SW-01"
  ip-source-address 10.159.4.1   ← SW2's permanent fabric identity (Loopback 1)
  is-type l1                      ← single-area SPB (Level 1 only)
router isis enable

ip-source-address 10.159.4.1 is SW2's unique identifier in the fabric — equivalent to a Router ID in OSPF. Every other switch knows SW2 by this address. It must be reachable and stable. 

spbm 1 nick-name 0.00.02   ← 12-bit SPB label for this node in the topology

Redistribution — how IS-IS tells the fabric about SW2's local subnets:

router isis
  redistribute direct          ← advertise VLAN SVIs (153, 154, 155, 156) into SPB
  redistribute direct enable
exit
isis apply redistribute direct  ← applies immediately (no reload needed)

This is why other switches in the fabric can route to 10.153.4.0/24, 10.154.4.0/24, etc. — IS-IS floods those prefixes across the backbone.

Principle 2

Muscle

SPB — The Forwarding Plane

"IS-IS says where. SPB actually moves the frame."

SPB (Shortest Path Bridging, IEEE 802.1aq) takes the topology map from IS-IS and does the actual frame forwarding using MAC-in-MAC encapsulation across B-VLANs. The original customer frame is wrapped inside an outer B-VLAN header for backbone transport. 

spbm
spbm ethertype 0x8100          ← outer VLAN tag ethertype for B-VLAN encapsulation

router isis
  spbm 1 nick-name 0.00.02
  spbm 1 multicast enable       ← enables SPB multicast (DHCP broadcasts cross the fabric)
  spbm 1 ip enable              ← enables IP routing over SPB backbone

B-VLANs 4051 and 4052 are auto-created on boot — you will see them in show vlan members. These are the backbone VLANs SPB uses for its two redundant trees. You never configure them manually; never assign data ports to them.

Key distinction from EXOS: In EXOS, VLANs ARE the transport. In VOSS, VLANs are customer-edge labels. SPB/B-VLAN is the actual backbone transport — invisible to the customer.

Principle 3

Service

I-SID — The Logical Service Identifier

"VLAN = local color. I-SID = global postal code in the fabric."

An I-SID (Instance Service Identifier) is a 24-bit label that binds a local VLAN to a globally unique service within the SPB backbone. When a frame arrives on VLAN 153, SPB wraps it with I-SID 144153 so any switch in the fabric knows exactly which service this frame belongs to.

DHCP Architecture

SW2 is the authoritative DHCP server for all four subnets. The server is bound to the management CLIP (10.158.4.1). Relay paths go from each VLAN SVI → CLIP on the same switch — a standard VOSS self-relay pattern. 

ip dhcp-relay fwd-path 10.153.4.1 10.158.4.1 enable mode bootp_dhcp
ip dhcp-relay fwd-path 10.154.4.1 10.158.4.1 enable mode bootp_dhcp

ip dhcp-server subnet 10.153.4.0/24
  pool 10.153.4.2 10.153.4.250
  router 10.153.4.1
  lease-time 86400                           ← 24h — matches May 8 GTK mitigation finding
ip dhcp-server enable
Principle 4

Edge

Auto-Sense / Fabric Attach — The On-Ramp

"Plug it in. The switch finds its own I-SID."

Auto-Sense replaces manual VLAN-per-port assignment. Every port on SW2 is Auto-Sense enabled. When a device plugs in, the switch reads its LLDP Fabric Attach TLV to determine device type, then automatically assigns the matching I-SID.

Important: Discovery is LLDP-based — NOT MAC-based. The AP announces "I am a WAP-Type1" via an LLDP FA TLV. SW2 reads that TLV. Devices without FA TLVs fall to proxy-no-auth or the onboarding I-SID.

<!-- All 20 ports -->
interface GigabitEthernet 1/1
  auto-sense enable
  no shutdown
  poe fast-poe-enable        ← ports 1/1-1/16 are PoE (AP2 powered here)
  poe perpetual-poe-enable

<!-- The decision rules -->
auto-sense fa wap-type1 i-sid 144153      ← AP announces WAP-Type1 → AP-Mgmt I-SID
auto-sense fa camera i-sid 144155         ← camera FA TLV → Camera I-SID
auto-sense fa proxy-no-auth i-sid 144156  ← non-FA proxy → Data I-SID
auto-sense data i-sid 144156              ← data device → Data I-SID
auto-sense onboarding i-sid 15999999      ← unknown device → Onboarding I-SID

AP2 flow: AP2 plugs into any PoE port → sends LLDP FA TLV "WAP-Type1" → SW2 auto-assigns I-SID 144153 (AP-Management) → AP2 gets DHCP from 10.153.4.0/24 → XIQ reaches AP2 for management.

The 802.1X Clue Hidden in the Banner

banner motd "to disable NAC in the event of internet outage or NAC is unreachable"
banner motd "no eapol enable"

The banner is a dead giveaway: eapol enable is planned on this switch. When your boss activates it, every Auto-Sense port will require 802.1X authentication before assigning a service I-SID. The banner is the emergency override for staff if the NPS server goes unreachable.

Until NPS says Access-Accept → device stays in I-SID 15999999 (Onboarding). After Access-Accept → device moves to I-SID 144154 (WirelessUser, VLAN 154).

Conceptual Corrections

Getting the Labels Right

Three instincts surfaced today that needed calibration:

Swap to fix: "IS-IS is service" → IS-IS is Brain (control plane). I-SID is the Service. IS-IS and SPB work as a pair — IS-IS builds the topology map, SPB drives the forwarding. I-SID is a separate, third layer entirely.

FA discovery is LLDP-based, not MAC-based. Auto-Sense reads the LLDP Fabric Attach TLV the device announces. The MAC address is irrelevant for I-SID assignment. A device with no FA TLV (and no 802.1X) lands on the onboarding I-SID regardless of its MAC.

Correct instinct confirmed: 802.1X packets must be authenticated by NPS before being tagged into the service I-SID. The controlled/uncontrolled port concept — the port stays in onboarding until the RADIUS Access-Accept arrives. After that, SPB moves the client into I-SID 144154.

The Three Actors

ActorDeviceRoleProtocol
SupplicantMacBook / clientWants network accessEAPOL (over 802.11 air)
AuthenticatorAP2Proxies EAP — does NOT validate credentialsEAPOL inbound, RADIUS outbound
Auth ServerNPS (RADIUS)Sole validator — checks AD, assigns VLAN, delivers MSKRADIUS UDP 1812

AP2 has two logical ports per client:

  • Uncontrolled port — always open. Only EAPOL frames pass. Nothing else.
  • Controlled port — locked until NPS Access-Accept. Then data flows on VLAN_154.

Full PEAP-MSCHAPv2 Frame Sequence (verbatim from session dialogue)

MacBook (Supplicant)        AP2 (Authenticator)         NPS (RADIUS)
       |                           |                          |
       |── EAPOL-Start ──────────>|                          |
       |<─ EAP-Request/Identity ──|                          |
       |── EAP-Response/Identity ─|  ← USERNAME IN PLAINTEXT |
       |   (last visible frame)   |── RADIUS Access-Request─>|
       |                          |                          | checks AD
       |                          |<─ Access-Challenge ──────| (TLS ServerHello)
       |<─ EAP-Request/PEAP ──────|                          |
       |── EAP-Response ─────────>|── RADIUS Access-Request─>|
       |   [TLS ClientHello]      |   [TLS ClientHello]      |
       |                          |                          |
       |   ══ PEAP TLS TUNNEL OPEN — all below is encrypted ══
       |                          |                          |
       |── MSCHAPv2-Response ────>|── RADIUS Access-Request─>|
       |   (NT-hash inside tunnel)|                          | validates vs AD
       |                          |<─ RADIUS Access-Accept ──|

Key detail: The username travels in plaintext in EAP-Response/Identity — visible to any OTA capture. The password never leaves the client in plaintext. MSCHAPv2 uses NT-hash challenge-response inside the PEAP TLS tunnel.

How NPS Knows Which VLAN to Assign

The VLAN is a policy decision, not a credential outcome. It is configured in the NPS Network Policy as a RADIUS attribute returned on Access-Accept:

RADIUS AttributeValueMeaning
Attr 64 — Tunnel-Type13 (VLAN)Encapsulation type
Attr 65 — Tunnel-Medium-Type6 (802)Medium type
Attr 81 — Tunnel-Private-Group-ID"154"VLAN number AP2 uses to tag the client

NPS matches the request against a Network Policy (conditions: AD group membership, SSID name). The matched policy specifies these attributes. Different AD groups can map to different VLANs — same RADIUS server, different policies.

MSK Generation — What Goes In

PhaseWhat HappensOutput
PEAP outer (TLS) Client and NPS negotiate TLS tunnel. NPS presents certificate. Client verifies. TLS master secret + session randoms
MSCHAPv2 inner (inside tunnel) NPS sends challenge. Client responds with NT-Hash(password) + challenge. NPS validates against AD. MSCHAPv2 session key material
MSK derivation PRF(TLS master secret + MSCHAPv2 key material + randoms). Both sides compute independently. MSK (512 bits, unique per session)

NPS sends MSK to AP2 inside RADIUS Access-Accept via MS-MPPE-Recv-Key / MS-MPPE-Send-Key attributes — over the wired RADIUS path, never over the air. The client computes the identical MSK locally. AP2 never sees the password.

PMK — The Critical Difference Between Personal and Enterprise

WPA-PersonalWPA-Enterprise (802.1X)
PMK sourcePBKDF2(passphrase + SSID) — fixedFirst 256 bits of MSK — unique per session
PMK scopeSame for ALL clients, ALL sessionsUnique per client, per session
Offline attackCapture M1+M2 → crack passphrase → derive PTKMSK never on air — no offline attack possible
May 8 relevanceGTK rotation broke iPhones (shared PMK)Each client has unique PMK — GTK rotation still applies but PTK is per-session

The 4-Way Handshake — Identical in Both Modes

PMK (from MSK first 256 bits — Enterprise; from PSK — Personal)
    ↓
PTK = PRF-512(PMK + ANonce + SNonce + AP_MAC + STA_MAC)
    ↓
  KCK (128 bits) — key confirmation key  (signs the MIC in M2/M3/M4)
  KEK (128 bits) — key encryption key    (wraps GTK in M3)
  TK  (128 bits) — temporal key          (encrypts all data frames)

The handshake is structurally identical whether PMK came from a passphrase or from NPS. This is why the May 8 GTK rotation findings apply directly to 802.1X deployments.
