STP & Port 5 Note

Port 5 was added to Corporate/Guest/IoT VLANs via manual CLI but was not added to the STP instance. This means port 5 has no STP loop protection for those VLANs. In a simple single-switch topology this is functionally harmless, but in a multi-switch environment it could cause loops. To fix: configure stpd s0 add vlan Corporate ports 5 dot1d (and same for Guest, IoT).

The Routing Table Is Correct — The Problem Is Upstream

The switch knows exactly what to do: send unknown traffic (including 8.8.8.8) to 192.168.0.1 via the default route. The switch's job is done correctly. The failure is that when traffic from 10.30.0.x arrives at the home router, the home router has no return path — it only knows how to send replies back to 192.168.0.0/24 devices.

Root Cause Analysis

Why the APs Got IP Addresses

The APs use the Default VLAN (untagged) for their management traffic — not any of the data VLANs (20/30/40). This is the critical distinction.

The Key Fact

APs Are on Default VLAN — Clients Are on VLANs 20/30/40

The AP itself (its management interface) always lives on the Default VLAN (untagged). It gets a 192.168.0.x address. The APs then tag WiFi client traffic with VLAN 20, 30, or 40 per SSID and forward it tagged to the switch. So the AP is simultaneously on Default VLAN (management) and carrying client traffic for VLANs 20/30/40 (data). Two completely separate flows on the same physical port.

Root Cause Analysis

Why WiFi Clients Cannot Reach the Internet

Everything on the switch side is working correctly. The failure is entirely at the Quantum Fiber home router — it cannot NAT return traffic to the 10.x.x.x subnets.

Root Cause — One Sentence

The Quantum Fiber home router only NATstrafic sourced from 192.168.0.0/24 (its own LAN subnet). Traffic from 10.20.0.0/24 or 10.30.0.0/24 arrives at the home router as a "foreign" subnet — the home router either drops the outbound packet, or forwards it to the internet but cannot route the reply back because it has no path to 10.x.x.x. The switch cannot fix this because it does not support NAT (configure ipmasquerade does not exist on EXOS).

Issues & Next Steps

Internet Connectivity — Fix Options

Option A — Add a Router Between Quantum Fiber and 5320 (Recommended Long-Term)

What: Place a second router (e.g. TP-Link Archer ~$30) between the Quantum Fiber box and the 5320. This router becomes the NAT boundary for all your private subnets.

How it works: Quantum Fiber router → new router WAN port (gets 192.168.0.x) → new router LAN port → 5320 port 1. New router does NAT for all 10.x.x.x traffic. Quantum Fiber only ever talks to the new router's WAN IP — it never sees 10.x.x.x addresses.

Trade-offs: Preserves your 10.x.x.x VLAN subnet design. Clean architecture. Adds a hardware device. ~20 min setup.

Option B — DHCP Relay (Fastest, No Hardware)

What: Remove 10.x.x.x config from the switch VLANs. Configure DHCP relay so that DHCP requests from VLANs 20/30/40 are forwarded to the home router (192.168.0.1). Clients get 192.168.0.x addresses — the home router already NATs those.

Commands:

configure vlan Guest    dhcp-address-range none
configure vlan Corporate dhcp-address-range none
disable dhcp ports 3,5 vlan Guest
disable dhcp ports 3,5 vlan Corporate
configure vlan Guest     iphelper add 192.168.0.1
configure vlan Corporate iphelper add 192.168.0.1
save configuration

Trade-off: Clients get 192.168.0.x IPs — you lose VLAN separation at the IP level. VLAN-based traffic isolation (tagged frames) still works, but clients are in the same IP subnet. Fine for a home/lab setup.

Option C — ISP Static Routes (Clean, No Hardware, But Requires ISP)

What: Call Quantum Fiber and ask them to add static routes on your CPE for 10.20.0.0/24, 10.30.0.0/24, and 10.40.0.0/24, all pointing to 192.168.0.24 (the switch). Quantum Fiber router then knows where to send reply traffic.

Trade-off: Preserves your subnet design. No extra hardware. ISPs often decline or are unaware of how to do this for residential accounts. Unknown wait time.
