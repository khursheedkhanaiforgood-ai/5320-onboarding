RFC / IETF Reference Index

Every RFC below is directly relevant to either the API or SNMP stack used in EP1. Links go to the canonical IETF RFC Editor. Open any entry to read the authoritative specification.

REST / HTTP / API

RFC 7230 RFC

HTTP/1.1 — Message Syntax & Routing

Defines the wire format: request-line, headers, body. Foundation for all REST calls. Governs how clients and servers exchange messages. rfc-editor.org/rfc/rfc7230

RFC 7231 RFC

HTTP/1.1 — Semantics & Content

Defines methods (GET, POST, PUT, DELETE, PATCH), status codes (200, 201, 400, 401, 403, 404, 429, 500), and content negotiation. The primary reference for REST method semantics. rfc-editor.org/rfc/rfc7231

RFC 7235 RFC

HTTP/1.1 — Authentication

Defines the WWW-Authenticate and Authorization header framework. The plumbing that Bearer tokens (RFC 6750) slot into. rfc-editor.org/rfc/rfc7235

RFC 6749 RFC

OAuth 2.0 Authorization Framework

The authorization standard used by EP1 API. Defines four grant types; EP1 uses Client Credentials for machine-to-machine (script ↔ EP1). rfc-editor.org/rfc/rfc6749

RFC 6750 RFC

OAuth 2.0 Bearer Token Usage

Defines how the access token is passed in HTTP: Authorization: Bearer <token> header. Every EP1 API call uses this format. rfc-editor.org/rfc/rfc6750

RFC 8259 RFC

JSON — The JavaScript Object Notation

Defines the JSON data format used in all EP1 API request and response bodies. Canonical reference for parsers and serialisers. rfc-editor.org/rfc/rfc8259

RFC 7807 RFC

Problem Details for HTTP APIs

Standardises error response bodies: type, title, status, detail, instance. EP1 API error payloads follow this pattern. rfc-editor.org/rfc/rfc7807

RFC 8446 RFC

TLS 1.3

Transport security wrapping all EP1 API and SNMP traffic. Required for all cloud-bound communications. Supersedes TLS 1.2 (RFC 5246). rfc-editor.org/rfc/rfc8446

SNMP

RFC 1157 RFC

SNMPv1 — Simple Network Management Protocol

Original SNMP (1990). Defines Manager/Agent/MIB model, Get/GetNext/Set/Trap operations, community string auth (cleartext). Still supported by EP1 devices but not recommended. rfc-editor.org/rfc/rfc1157

RFC 1901 RFC

SNMPv2c — Community-Based SNMPv2

Adds GetBulk (efficient table retrieval) and Inform (acknowledged trap). Still uses cleartext community string. EP1 default config uses v2c. rfc-editor.org/rfc/rfc1901

RFC 1905 RFC

Protocol Operations for SNMPv2

Defines the PDU types for SNMPv2: GetRequest, GetNextRequest, GetBulkRequest, Response, SetRequest, InformRequest, SNMPv2-Trap. rfc-editor.org/rfc/rfc1905

RFC 3410 RFC

Introduction to SNMPv3 Framework

Overview of the SNMPv3 architecture. Entry point to the SNMPv3 RFC cluster (3411–3415). Explains why v1/v2c auth was inadequate. rfc-editor.org/rfc/rfc3410

RFC 3411 RFC

Architecture for SNMP Management Frameworks

Defines the modular SNMPv3 engine architecture: Message Processing, Security Model, Access Control, applications (notification originator, command responder). rfc-editor.org/rfc/rfc3411

RFC 3414 RFC

User-Based Security Model (USM) for SNMPv3

Defines authentication (HMAC-MD5, HMAC-SHA) and privacy (DES, AES) for SNMPv3. Each user has separate auth/priv keys. EP1 SNMPv3 config maps to this RFC. rfc-editor.org/rfc/rfc3414

RFC 3415 RFC

View-Based Access Control Model (VACM)

Defines what MIB objects a given user/group can read or write. Controls the scope of SNMP access per security level. rfc-editor.org/rfc/rfc3415

RFC 2578 RFC

SMIv2 — Structure of Management Information v2

The language (ASN.1 subset) used to define MIBs. Every MIB file — including Aerohive/EP1 proprietary MIBs — is written in SMIv2. rfc-editor.org/rfc/rfc2578

RFC 1213 RFC

MIB-II — Standard MIB for TCP/IP Internets

The baseline MIB every IP device must support: system group (sysDescr, sysUpTime), interfaces group (ifDescr, ifAdminStatus), IP, TCP, UDP groups. Applies to EXOS SW1 and AP1. rfc-editor.org/rfc/rfc1213

RFC 2863 RFC

IF-MIB — Interfaces Group MIB

Extends MIB-II interfaces group. Defines ifTable, ifXTable (high-capacity counters), link state traps (linkUp, linkDown). Essential for port-utilisation polling on SW1. rfc-editor.org/rfc/rfc2863

Part 1 — API Foundation

What Is a REST API?

REST (Representational State Transfer) is an architectural style defined by Roy Fielding in his 2000 doctoral dissertation, later codified via the HTTP RFCs. It is not a protocol — it is a set of constraints applied to HTTP that produce a scalable, stateless, cacheable web service.

The Six REST Constraints

ConstraintWhat it meansEP1 application
Client–ServerUI and data storage are decoupledYour script (client) calls EP1 cloud (server) — no shared state
Stateless RFC 7230Each request contains all context needed to process itEvery API call includes the Bearer token — EP1 does not maintain session state
CacheableResponses must declare cacheabilityGET /devices responses include cache headers; POST/PUT are not cacheable
Uniform Interface RFC 7231Standard HTTP methods on resource URLsGET /xapi/v1/devices, POST /xapi/v1/network-policies
Layered SystemClient cannot tell if it's talking to an origin server or a proxyEP1 load balancers and CDN are invisible to your API client CONFIRMED — white paper p.3
Code on DemandServer can extend client by sending executable code (optional)Not used by EP1 API

OAuth 2.0 + Bearer Token Authentication

How EP1 API Auth Works

EP1 uses the OAuth 2.0 Client Credentials Grant RFC 6749 §4.4 for machine-to-machine access (scripts, automation, integrations). Bearer tokens are passed per RFC 6750.

Token Flow — Step by Step

EP1 Auth Server

access_token

Architecture note CONFIRMED — white paper p.3, p.7

The Auth Server lives in the GDC (Global Data Centre, AWS-only). The API Server (your VIQ) lives in the RDC (Regional Data Centre, AWS/Azure/GCP). Token issuance goes to GDC; all subsequent API calls go to your regional RDC endpoint. This is why the OAuth token URL and the API base URL are different hostnames.

Token Request (curl)

# Step 1 — Get an access token (Client Credentials Grant, RFC 6749 §4.4)
curl -s -X POST "https://extremecloudiq.com/api/oauth2/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET"

# Response
{
  "access_token": "eyJhbGci...",
  "token_type": "Bearer",        # RFC 6750 §2.1
  "expires_in": 3600
}

# Step 2 — Use the token (RFC 6750 §2.1 — Authorization header method)
curl -s -X GET "https://extremecloudiq.com/xapi/v1/devices" \
  -H "Authorization: Bearer eyJhbGci..." \
  -H "Accept: application/json"

Security rules RFC 6750 §5

  • Never pass the token in a URL query parameter — always use the Authorization header
  • Always use TLS 1.2+ (EP1 requires TLS 1.3 RFC 8446) — tokens over plaintext HTTP are invalid
  • Rotate client secrets regularly; treat them like passwords
  • Token lifetime is 3600s (1 hour) — build token refresh into any long-running script

Key takeaway for the Sr Director session

Clients with existing XIQ API integrations (Python scripts, Ansible playbooks, custom dashboards) do not need to change their code when moving to EP1. The API surface is additive — same base URL, same auth, same core endpoints. The only change is new endpoints become available once the EP1 Standard licence is active.

Part 2 — SNMP Foundation

What Is SNMP and Why Does It Exist?

SNMP (Simple Network Management Protocol) was created in 1988 to solve a specific problem: how do you monitor and manage hundreds of network devices (routers, switches, APs) without logging into each one individually? The answer: run a small agent on each device that speaks a common protocol, expose a structured database of device state (the MIB), and allow a central manager (NMS) to query or set values remotely.

The Three-Component Model

ComponentWhat it isIn your lab
Manager (NMS)The system that polls agents and receives traps. Runs your monitoring software.SolarWinds / PRTG / Zabbix / your laptop running snmpwalk
AgentRuns on the managed device. Responds to queries, sends traps.AP1 (IQ Engine OS), SW1 (EXOS 33.5+)
MIBThe structured data model describing what the agent knows. A tree of OIDs.MIB-II RFC 1213 + IF-MIB RFC 2863 + Aerohive enterprise MIBs

Transport and Ports

PortDirectionPurpose
UDP 161NMS → AgentGet/Set polling requests
UDP 162Agent → NMSTraps and Informs (unsolicited alerts)
TCP 161NMS → AgentOptional (rarely used) — TCP SNMP for reliability

Critical architecture fact for EP1

AP1 and SW1 send SNMP traps directly to your NMS on UDP 162. Traps do not go through the EP1 cloud. This means: (1) your NMS must have network reachability to the AP/switch management VLAN, and (2) EP1 going down does not affect SNMP trap delivery. CONFIRMED — User Guide p.240: "SNMP server IP" is the NMS, not EP1 cloud

MIBs & OIDs — The Data Model

Understanding the OID Tree

Every piece of data an SNMP agent exposes has a unique Object Identifier (OID) — a dotted-decimal address in a global tree. The tree is defined by SMIv2 RFC 2578.

iso(1)
  └── org(3)
        └── dod(6)
              └── internet(1)
                    ├── mgmt(2)
                    │     └── mib-2(1)          ← RFC 1213 MIB-II starts here
                    │           ├── system(1)   ← sysDescr, sysUpTime, sysContact
                    │           ├── interfaces(2) ← ifTable, ifAdminStatus (RFC 2863)
                    │           ├── ip(4)
                    │           └── ...
                    └── private(4)
                          └── enterprises(1)
                                └── aerohive(26928)  ← Aerohive/EP1 proprietary MIBs
                                      └── ahProduct(...)
                                            ├── ahSSIDTable
                                            ├── ahClientTable
                                            └── ahTrap...

Aerohive/EP1 proprietary MIBs (ahMIB.mib, ahRadio.mib, ahClient.mib, etc.) are downloadable from GTAC. In your NMS: import the MIB files → browser can resolve OIDs to human-readable names. SNMP Server Config — Step by Step

Reference: EP1 Networking User Guide v25.9.0 p.240 CONFIRMED

  1. Toggle SNMP ON — the toggle is off by default per network policy
  2. Click + Add SNMP Server
  3. Enter the NMS IP address or hostname
  4. Select Version: v2c (lab) or v3 (production)
  5. Select Operation: Get and Trap — enables both polling and unsolicited traps
  6. Enter Community String: hivecommunity (EN default — change this in production)
  7. Click ADD SNMP SERVER → click SAVE SNMP SERVER
  8. Run a Complete Config Update to push to devices — SNMP changes require a full push, not a delta update

Syslog — Quick Companion Reference

Syslog is configured in the same Management Servers screen as SNMP (p.238). CONFIRMED — User Guide p.238

  • Protocol: UDP 514 (default) — APs send directly to syslog server, not through EP1 cloud
  • Severity threshold: set minimum level (Debug / Info / Notice / Warning / Error / Critical)
  • Use case: SNMP = operational alerts (traps), Syslog = verbose event stream — use both together for a complete picture
  • For lab: point both SNMP trap receiver and syslog to your Mac running snmptrapd + a syslog daemon

Sr Director Session — Key Talking Points

  1. Zero disruption: Both API integrations and SNMP NMS configs carry over unchanged to EP1. No rework required at migration.
  2. Additive API surface: EP1 Standard unlocks new endpoints (Contracts, Inventory, Fabric, AI Canvas). Existing scripts keep working; new scripts can do more.
  3. SNMP is device-direct: Traps bypass EP1 cloud — NMS receives alerts even if EP1 cloud has an outage. Network visibility is decoupled from cloud health.
  4. The convergence pattern: Use SNMP for low-latency operational traps (milliseconds), API for enrichment and automation (seconds). They complement, not replace, each other.
  5. SNMPv3 for compliance: If a client is PCI-DSS or HIPAA scoped, push them to SNMPv3 authPriv — community string in cleartext is a finding.
