Part 0 — Foundations: How APIs and SNMP Work

Before reading the RFC index or the EP1-specific sections, build the mental model from first principles. This section walks both protocols from wire to outcome — then places three real scenarios in front of you so the abstractions lock in.

What Is a REST API Call? — E2E Mechanics

A REST API is a convention layered on top of HTTP. Nothing runs persistently on the remote device waiting for you. Each call is self-contained: you send a request, the server handles it, returns a response, and forgets you. That statelessness is both its power and its constraint.

What must reside on the calling host

RequirementWhat it doesExample
HTTP client librarySends HTTPS requests, handles TLS, follows redirectsPython requests, curl, Go net/http
JSON parserSerialises request body, deserialises responsePython json stdlib — included by default
Client credentialsclient_id + client_secret stored securely (env var or vault)Obtained from XIQ / EP1 Settings → API Token
TCP 443 outboundTLS connection to cloud endpointFirewall rule to api.extremecloudiq.com
Token cacheStores bearer token to avoid re-auth every callIn-memory variable; refresh when 401 received

No daemon, no SNMP agent, no background process. One Python script + two lines of credentials is the entirety of the host footprint.

The 8-step request lifecycle

1. Script starts
→
2. TCP+TLS 443
→
3. POST /oauth2/token
→
4. Bearer token returned
5. GET/PATCH/POST + Authorization: Bearer
→
6. EP1 validates token
→
7. 200 / 201 / 202 + JSON body
8. If 202: poll GET /jobs/{id} until COMPLETED

Steps 1–4 are OAuth2 Client Credentials Grant (RFC 6749 §4.4). Steps 5–7 are a standard REST exchange per RFC 7231. Step 8 is the async job pattern — EP1 uses it for bulk operations that touch many devices.

Token lifetime

Bearer tokens from EP1 expire after 3600 s (1 hour). The correct pattern is: cache the token, use it until you receive a 401 Unauthorized, then re-authenticate. Do not call /oauth2/token before every request — that wastes auth round-trips and may hit rate limits.

Example 1 — Daily QoS Config Push to All APs in a Region CONFIRMED

Scenario: Every night at 02:00, a script runs on the lab laptop. It updates the radio QoS profile on every AP3000 in the KhKLab region to enforce DSCP EF (Expedited Forwarding) for voice traffic. There are 12 APs today, but the pattern scales to any number.

Sequence of operations

  1. Authenticate — POST /oauth2/token → cache bearer token
  2. Enumerate APs — GET /devices?pageSize=100&page=1&locationId={region_id} → collect all device IDs. Loop pages until data.length < pageSize
  3. For each AP: PATCH /devices/{id}/network-policy with updated QoS profile JSON body
  4. Deploy — POST /devices/deploy with the full list of AP IDs → receives 202 Accepted + job_id
  5. Poll job — GET /jobs/{job_id} every 10 s until status = "COMPLETED"
  6. Audit — write pass/fail per device to a log file
import os, time, requests

BASE = "https://api.extremecloudiq.com"
CREDS = {"grant_type":"client_credentials","client_id":os.environ["XIQ_ID"],"client_secret":os.environ["XIQ_SECRET"]}

# Step 1 — auth
token = requests.post(f"{BASE}/oauth2/token", json=CREDS).json()["access_token"]
h = {"Authorization": f"Bearer {token}"}

# Step 2 — enumerate APs (one page, lab scale)
aps = requests.get(f"{BASE}/devices?pageSize=100&page=1", headers=h).json()["data"]
ap_ids = [ap["id"] for ap in aps]

# Step 3 — update QoS profile on each
QOS_PAYLOAD = {"qos_profile": {"voice_dscp": "EF", "video_dscp": "AF41"}}
for ap_id in ap_ids:
    requests.patch(f"{BASE}/devices/{ap_id}/network-policy", json=QOS_PAYLOAD, headers=h)

# Step 4+5 — deploy and poll
r = requests.post(f"{BASE}/devices/deploy", json={"ids": ap_ids}, headers=h)
job_id = r.json()["job_id"]
while True:
    status = requests.get(f"{BASE}/jobs/{job_id}", headers=h).json()["status"]
    if status == "COMPLETED": break
    time.sleep(10)
print(f"QoS deployed to {len(ap_ids)} APs")

What this teaches: the API is fully stateless — the script could crash between steps 3 and 4 and resume by re-running the deploy step. Each HTTP request carries everything the server needs. The bearer token is the only "session."

Example 2 — Migrating 1,000 APs via API CONFIRMED

Scenario: A client is moving from their legacy XIQ Classic network policy to a new EP1-native policy as part of a campus refresh. All 1,000 APs must be re-assigned in a single maintenance window. Manual UI work would take 4–6 hours. The API does it in under 10 minutes — but three engineering problems must be solved.

The three problems at scale

Problem 1 — Pagination (100 APs per page = 10 GET requests)

EP1 returns a maximum of 100 devices per call. To collect all 1,000 you must loop: page 1 → page 2 → … → page 10. Stop when data.length < pageSize. If you call only page 1, you silently miss 900 APs. This is the most common migration script bug.

Problem 2 — Rate limiting (HTTP 429 Too Many Requests)

EP1 / XIQ enforces API rate limits. Sending 1,000 PATCH requests back-to-back will return 429 with a Retry-After: 60 header. The correct pattern is exponential backoff: on 429, wait Retry-After seconds, then retry. Do not retry immediately. RFC 6585 §4 defines the 429 status.

Problem 3 — Partial failure (some APs unreachable during deploy)

During the deploy job, 12 of 1,000 APs may be offline (in closets, low on uplink). The job returns COMPLETED_WITH_ERRORS. Your script must parse the errors[] array, log the failed AP IDs, and either retry those only or page the on-call engineer. Never assume COMPLETED = 100% success.

import os, time, requests

BASE = "https://api.extremecloudiq.com"
token = requests.post(f"{BASE}/oauth2/token", json={
    "grant_type":"client_credentials",
    "client_id":os.environ["XIQ_ID"],
    "client_secret":os.environ["XIQ_SECRET"]
}).json()["access_token"]
h = {"Authorization": f"Bearer {token}"}

# Paginated AP collection
all_ids, page = [], 1
while True:
    resp = requests.get(f"{BASE}/devices?pageSize=100&page={page}", headers=h).json()
    batch = resp.get("data", [])
    all_ids += [d["id"] for d in batch]
    if len(batch) < 100: break
    page += 1

NEW_POLICY = {"network_policy_id": "np-ep1-campus-v2"}

# Patching with rate-limit backoff
for ap_id in all_ids:
    while True:
        r = requests.patch(f"{BASE}/devices/{ap_id}/network-policy", json=NEW_POLICY, headers=h)
        if r.status_code == 429:
            time.sleep(int(r.headers.get("Retry-After", 60)))
        else:
            break

# Async deploy — batched to 200 APs per job
failed = []
for i in range(0, len(all_ids), 200):
    batch = all_ids[i:i+200]
    job_id = requests.post(f"{BASE}/devices/deploy", json={"ids":batch}, headers=h).json()["job_id"]
    while True:
        result = requests.get(f"{BASE}/jobs/{job_id}", headers=h).json()
        if result["status"] in ("COMPLETED","COMPLETED_WITH_ERRORS"): break
        time.sleep(15)
    failed += [e["device_id"] for e in result.get("errors",[])]

print(f"Done. {len(all_ids)-len(failed)} success, {len(failed)} failed → {failed}")

What this teaches: at scale, three things collapse — pagination correctness, backpressure handling, and failure accountability. The API surface is unchanged whether you touch 1 AP or 10,000; the engineering challenge is in your wrapper code.

Example 3 — SNMP AP Outage Detection RFC

Scenario: An AP3000 in the KhKLab goes offline — abrupt power loss, no graceful shutdown. The network management system (NMS) must detect this and page an engineer. How does SNMP actually do that, E2E?

Two detection paths

PathHow it worksLatencyCaveat
Trap path AP sends ahAPDown trap (OID 1.3.6.1.4.1.26928.1.1.1.2) to NMS on UDP 162 the moment the radio subsystem detects shutdown < 1 s Requires graceful shutdown. Abrupt power cut = no trap sent.
Poll path NMS sends GetRequest every 60 s to AP's management IP on UDP 161. After 2 consecutive timeouts (no response), marks device as DOWN. 60–75 s Always works, even on hard crash. More conservative.

read HOSTNAME; read IP; read OID_LINE
echo "$(date): AP trap from $IP OID=$OID_LINE" >> /var/log/ap_traps.log
# escalate ahAPDown
echo "$OID_LINE" | grep -q "ahAPDown" && \
  curl -s -X POST https://events.pagerduty.com/v2/enqueue \
    -d "{\"routing_key\":\"$PD_KEY\",\"event_action\":\"trigger\",\"payload\":{\"summary\":\"AP offline: $IP\"}}"

Hard crash caveat — the polling gap

If the AP loses power mid-frame (circuit breaker, PoE switch reboot, kicked power strip), the SNMP agent has zero time to send a trap. The NMS will not know until the next poll cycle fails — up to 75 seconds later. In a campus with active VoIP calls, that 75 s window is the gap between "something's wrong" and "the NMS confirmed it." Design alert thresholds accordingly.

Architectural takeaway

API and SNMP are complementary, not competing. SNMP is the fast-path fault detector — it fires before any API poll would run. The API is the config and automation plane — it does things SNMP was never designed to do (create SSIDs, push QoS profiles, bulk-migrate devices). A mature network operations platform runs both: SNMP for the millisecond-latency operational layer, API for the automation layer above it.

The baseline MIB every IP device must support: system group (sysDescr, sysUpTime), interfaces group (ifDescr, ifAdminStatus), IP, TCP, UDP groups. Applies to EXOS SW1 and AP1. rfc-editor.org/rfc/rfc1213

Extends MIB-II interfaces group. Defines ifTable, ifXTable (high-capacity counters), link state traps (linkUp, linkDown). Essential for port-utilisation polling on SW1. rfc-editor.org/rfc/rfc2863

Part 1 — API Foundation

What Is a REST API?

REST (Representational State Transfer) is an architectural style defined by Roy Fielding in his 2000 doctoral dissertation, later codified via the HTTP RFCs. It is not a protocol — it is a set of constraints applied to HTTP that produce a scalable, stateless, cacheable web service.

OAuth 2.0 + Bearer Token Authentication

EP1 uses the OAuth 2.0 Client Credentials Grant RFC 6749 §4.4 for machine-to-machine access (scripts, automation, integrations). Bearer tokens are passed per RFC 6750.

The Auth Server lives in the GDC (Global Data Centre, AWS-only). The API Server (your VIQ) lives in the RDC (Regional Data Centre, AWS/Azure/GCP). Token issuance goes to GDC; all subsequent API calls go to your regional RDC endpoint. This is why the OAuth token URL and the API base URL are different hostnames.

Token Request (curl)

# Step 1 — Get an access token (Client Credentials Grant, RFC 6749 §4.4)
curl -s -X POST "https://extremecloudiq.com/api/oauth2/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials" \
  -d "client_id=YOUR_CLIENT_ID" \
  -d "client_secret=YOUR_CLIENT_SECRET"

# Response
{
  "access_token": "eyJhbGci...",
  "token_type": "Bearer",        # RFC 6750 §2.1
  "expires_in": 3600
}

# Step 2 — Use the token (RFC 6750 §2.1 — Authorization header method)
curl -s -X GET "https://extremecloudiq.com/xapi/v1/devices" \
  -H "Authorization: Bearer eyJhbGci..." \
  -H "Accept: application/json"

Security rules RFC 6750 §5

  • Never pass the token in a URL query parameter — always use the Authorization header
  • Always use TLS 1.2+ (EP1 requires TLS 1.3 RFC 8446) — tokens over plaintext HTTP are invalid
  • Rotate client secrets regularly; treat them like passwords
  • Token lifetime is 3600s (1 hour) — build token refresh into any long-running script

Part 2 — SNMP Foundation

What Is SNMP and Why Does It Exist?

SNMP (Simple Network Management Protocol) was created in 1988 to solve a specific problem: how do you monitor and manage hundreds of network devices (routers, switches, APs) without logging into each one individually? The answer: run a small agent on each device that speaks a common protocol, expose a structured database of device state (the MIB), and allow a central manager (NMS) to query or set values remotely.

The Three-Component Model

ComponentWhat it isIn your lab
Manager (NMS)The system that polls agents and receives traps. Runs your monitoring software.SolarWinds / PRTG / Zabbix / your laptop running snmpwalk
AgentRuns on the managed device. Responds to queries, sends traps.AP1 (IQ Engine OS), SW1 (EXOS 33.5+)
MIBThe structured data model describing what the agent knows. A tree of OIDs.MIB-II RFC 1213 + IF-MIB RFC 2863 + Aerohive enterprise MIBs

Transport and Ports

PortDirectionPurpose
UDP 161NMS → AgentGet/Set polling requests
UDP 162Agent → NMSTraps and Informs (unsolicited alerts)
TCP 161NMS → AgentOptional (rarely used) — TCP SNMP for reliability

Critical architecture fact for EP1

AP1 and SW1 send SNMP traps directly to your NMS on UDP 162. Traps do not go through the EP1 cloud. This means: (1) your NMS must have network reachability to the AP/switch management VLAN, and (2) EP1 going down does not affect SNMP trap delivery. CONFIRMED — User Guide p.240: "SNMP server IP" is the NMS, not EP1 cloud

Sr Director Session — Key Talking Points

  1. Zero disruption: Both API integrations and SNMP NMS configs carry over unchanged to EP1. No rework required at migration.
  2. Additive API surface: EP1 Standard unlocks new endpoints (Contracts, Inventory, Fabric, AI Canvas). Existing scripts keep working; new scripts can do more.
  3. SNMP is device-direct: Traps bypass EP1 cloud — NMS receives alerts even if EP1 cloud has an outage. Network visibility is decoupled from cloud health.
  4. The convergence pattern: Use SNMP for low-latency operational traps (milliseconds), API for enrichment and automation (seconds). They complement, not replace, each other.
  5. SNMPv3 for compliance: If a client is PCI-DSS or HIPAA scoped, push them to SNMPv3 authPriv — community string in cleartext is a finding.

API vs SNMP — first principles

Two fundamentally different models for managing network infrastructure. Understanding both is essential before reading the protocol-specific tabs. This tab maps the mechanics, walks three real scenarios, and ends with a side-by-side decision table.

🔌
What is a REST API call?
Stateless · TCP 443 · RFC 7230–7235

A REST call is a self-contained HTTP request. Nothing runs persistently on the remote device waiting for you. Each call carries all authentication and context it needs.

TransportTCP 443 / TLS 1.3 (RFC 8446)
AuthOAuth2 Client Credentials → Bearer token (RFC 6749 §4.4 + RFC 6750)
Wire formatJSON (RFC 8259) in request + response body
Host requirementHTTP library + client_id/client_secret + TCP 443 outbound. No daemon.
Async model202 Accepted + job_id → poll GET /jobs/{id} until COMPLETED
8-step request lifecycle
1Script starts, opens TCP+TLS to api.extremecloudiq.com:443
2POST /oauth2/token with client_id + client_secret → receive bearer token (3600 s TTL)
3Script caches token in memory
4Send GET/PATCH/POST with Authorization: Bearer <token> header
5EP1 gateway validates token (audience, expiry, scope)
6EP1 applies operation, returns 200/201/202 + JSON body
7If 202: poll GET /jobs/{job_id} every 10 s until status = "COMPLETED"
8On 401: re-authenticate (step 2). On 429: wait Retry-After seconds then retry.
What is SNMP? — The poll/trap model
📡
SNMP fundamentals
UDP 161/162 · RFC 1157 (v1) → RFC 3410–3418 (v3)

SNMP is an application-layer protocol on top of UDP. The agent (on the device) exposes a MIB tree of named counters and state variables. The manager (NMS) reads them by polling or receives unsolicited notifications via traps.

Poll (GetRequest)NMS → device UDP 161. Response returns OID value. No connection state.
TrapDevice → NMS UDP 162. Fires on event (AP down, auth failure). No ack (v1/v2c).
InformLike trap but NMS sends acknowledgment back. v2c/v3 only.
Host requirement (NMS)snmptrapd + NMS app + Extreme MIB files + UDP 162 inbound
Host requirement (device)snmpd built into EXOS/IQ Engine OS — zero install

Scenario B — 1,000 AP migration
⚡
Bulk policy re-assignment at scale
API · three engineering problems
Problem 1Pagination — 100 APs/page = 10 GET requests. Miss page 2–10 = silent failure.
Problem 2Rate limit — 429 Too Many Requests. Must respect Retry-After header (RFC 6585 §4). Use exponential backoff.
Problem 3Partial failure — job returns COMPLETED_WITH_ERRORS. Parse errors[] array, log failed AP IDs, retry or escalate.
StrategyBatch into 200-AP deploy jobs. Parallel workers. Full audit log per AP ID.
SNMP role hereNone. SNMP cannot create, reassign, or migrate network policy objects.
Scenario C — SNMP AP outage detection
🔴
AP3000 hard power loss
SNMP · ahAPDown trap · OID 1.3.6.1.4.1.26928.1.1.1.2
Trap pathAP sends ahAPDown trap → NMS UDP 162 → snmptrapd → alert. Latency: <1 s. Requires graceful shutdown.
Poll pathNMS polls every 60 s. After 2 timeouts → marks DOWN. Latency: 60–75 s. Works on hard crash.
Hard crash caveatAbrupt power cut = no trap. Only poll detects it. 75 s gap is by design.
API role hereLimited. EP1 API can read device status but requires a script polling on a schedule — same latency as SNMP poll.

Why SNMP is no longer enough for configuration

SNMP was designed for polling and notification. Configuration via SNMP SET is fragile, lacks transactions, and has no rollback. The IETF responded with YANG + NETCONF + RESTCONF, which underpin modern network automation including ExtremeXOS RESTCONF on the 5320.
