4. The "Trojan Horse" Debate — Is EP1 Just a UI Facade?
Assistant — Analysis (reconstructed from session summary)

The "Trojan Horse" framing was validated — but with an important nuance. The device-facing layer is genuinely unchanged: CAPWAP still carries the config, the AP applies it the same way, and EP1 is explicitly not in the data path (p.12). In that specific sense, the facade interpretation is understandable.

However, the Trojan Horse metaphor holds precisely because EP1 installed a data platform — knowledge graph, lakehouse, multi-agent orchestration — while the user was focused on the UI change. The network configuration objects (SSIDs, policies, VLANs, user profiles) are the same objects. The cloud machinery that reads, enriches, reasons over, and learns from those objects is architecturally different.

The UI was the on-ramp. The AI Core + Data Hub is the real installation.

Conclusion framed for engineering review: The question "is it just a UI change?" has two correct answers depending on what layer you look at. Device-facing: yes, identical. Cloud-processing: no — EP1 added an AI Core layer on top of the XIQ base platform. The community post confirmation ("underlying platform unchanged") refers to the device-config objects and VIQ, not to the cloud-side processing architecture.

✎ Engineering annotation space: Does this framing of XIQ base + AI Core overlay match how EP1 was actually architected? Is the AI Core a separate logical layer, or was the base platform itself restructured?

5. AI-Assisted Policy Creation — The NL-to-Network Question
Khursheed — Verbatim

"One thing that I was surprised with, that the AI_Engine did not yet have the ability to create 'policies or workflows' from natural language input? Then push that into the network with the human in the loop?"

Assistant — Analysis (reconstructed from session summary)

The white paper (p.6) confirms EP1 leverages AI in four ways, including: "A conversational front-end interface, enables users to interact with data and knowledge through natural language" and "AI agents execute workflows and tasks based on their deployment and action scope."

The HITL gate is confirmed as the default (p.6). What is HYPOTHESIS is whether the natural language → policy intent → CAPWAP push loop is currently implemented end-to-end or whether the AI Expert in the current release only handles knowledge queries (Talk to Knowledge / Talk to Data / Think it Through) without direct push authority.

From the EP1 User Guide (pp.37-46, AI Expert section): the current GA features are Talk to Knowledge, Talk to Data, and Think it Through (Beta). Service Agent (Beta) handles workflow tasks. Whether "create an SSID with PPSK auth on VLAN 5 and push it to AP1" is a current Service Agent capability vs. a roadmap item is HYPOTHESIS — VERIFY WITH PRODUCT TEAM.

✎ Engineering annotation space: Is NL → policy config → CAPWAP push currently available in any EP1 release, or is this roadmap? Which Service Agent tasks are GA vs Beta?

7. The Three-Flow Analysis — Verbatim Response
The following section is the verbatim analytical response produced in this session, prior to the source-confidence challenge. It is preserved exactly as written. The subsequent section (Section 8) shows how the same analysis was then re-labeled with explicit confidence tiers.
Assistant — Verbatim Response: "XIQ vs EP1: E2E Architecture Across Three Flows"

The Core Architectural Thesis (Before the Flows)

XIQ is a system of record — siloed vertical stacks per function, each with its own store, each optimized for ingest. EP1 is a system of engagement — a shared lakehouse + knowledge graph at the center, with microservices publishing events into it from all directions. The protocols to the devices (CAPWAP, SSH, supplemental CLI) are identical in both platforms — the UI transition post confirmed "underlying platform, configurations, policies, and data remain unchanged." The difference is entirely in what happens inside the cloud boundary.

Flow 1: Policy Push

Trigger: Operator creates/modifies Network Policy → pushes to AP + switch.

XIQ Classic:

  1. Operator edits SSID/User Profile/Switch Template in UI → Config DB (relational — PostgreSQL [Inf])
  2. XIQ Config Compiler generates device-specific config blob → stored back to Config DB
  3. Operator triggers Delta or Complete Config Update → XIQ Orchestration Layer (monolithic [T3])
  4. AP: config delivered over CAPWAP control channel → DTLS-encrypted CAPWAP (RFC 5415) [T1]
  5. Switch: config delivered via SSH / Supplemental CLI over HTTPS tunnel
  6. XIQ polls device for ACK → CAPWAP keepalive / SSH return code
  7. Device state updated → Device State DB (Cassandra [T3])
  8. Operator sees green checkmark → Cache (Redis [T3]) invalidated → UI refresh

What XIQ does NOT do: No event record of why the policy changed. No cross-domain impact assessment before push. No lineage. Config DB is authoritative point-in-time state — no history of intent.

EP1:

  1. Operator edits policy in EP1 UI (same fields, same underlying objects) [T1] → VIQ Config Store (RDC-local)
  2. Optional AI path: Natural language intent → AI Expert → Orchestration Layer → policy suggestion → HITL gate → AI Core: Orchestrator + LLM [T1]
  3. Policy Microservice validates intent + checks Knowledge Graph for conflicts → Knowledge Graph (entity relationships: device ↔ policy ↔ SSID ↔ client) [T1]
  4. Config compiled → pushed to device via same CAPWAP / SSH protocols [T1] → CAPWAP RFC 5415/5416 (unchanged)
  5. Config change published as event to event bus → Kafka [Inf] → Data Ingestion Service [T1]
  6. Metadata Management records: who pushed, what changed, why (if AI-assisted: the intent), which devices affected → Data Hub: Metadata Management [T1]
  7. Auto Compliance validates: does this policy violate any governance rule? → AI Core: Auto Compliance [T1]
  8. Knowledge Graph updated: new policy ↔ device ↔ operator relationships written → Data Hub: Knowledge Graph [T1]
  9. Device ACK path: identical to XIQ → CAPWAP keepalive

What EP1 adds over XIQ: Event sourcing (every push is an immutable event), full intent lineage, pre-push conflict check via Knowledge Graph, governance validation, and optionally AI-assisted policy generation with HITL gate before any device is touched.

Flow 2: Telemetry / Observability

Trigger: Client associates to AP → generates association events, RSSI/SNR samples, roaming events, error counters.

XIQ Classic:

AP
 └─ CAPWAP data channel (DTLS) / HTTPS POST
     └─ XIQ Telemetry Ingest (cloud edge, RDC region [T1])
         ├─ Time-Series DB: RSSI, SNR, channel util, client counts (InfluxDB/TimescaleDB [T3])
         ├─ Relational DB: device inventory, policy mapping, user profiles (PostgreSQL [T3])
         └─ Redis cache: live dashboard counters [T3]
             └─ Dashboard: queries Time-Series DB → renders graphs
                 └─ Alert Engine: static threshold rules → email/webhook
                     └─ ML Insights: batch jobs, run periodically
                         └─ Anomaly surfaced → recommendation card in UI

EP1:

AP
 └─ CAPWAP / HTTPS → RDC
     └─ [ANONYMIZATION LAYER — shift-left, inside RDC] [T1]
         MAC → pseudonymous token
         Hostname stripped
         PII fields masked
         └─ Anonymized telemetry forwarded to GDC [T1]
             └─ Data Hub: Data Ingestion Service [T1]
                 ├─ Structured path: counters/stats → Data Lake (real-time + batch) [T1]
                 │   └─ Apache Iceberg-style lakehouse (append-only, versioned) [Inf]
                 ├─ Unstructured path: log events, error strings → Data Lake [T1]
                 └─ Knowledge Graph writer [T1]
                     └─ Entities: anon_client ↔ AP ↔ SSID ↔ VLAN ↔ policy ↔ time_window
                             └─ Metadata Management: lineage, classification, ownership [T1]
                                 └─ RAG-as-a-Service: telemetry events indexed for AI queries [T1]

Flow 3: AP Parameter Optimization

Scenario: High channel utilization on AP1, 2.4GHz, Ch 6.

XIQ Classic:

  1. Time-Series DB: rolling avg channel util crosses threshold (e.g., >70% for 15 min)
  2. Alert Engine: threshold rule fires → notification
  3. ML Insights (batch): pattern match against historical data → "Ch 11 historically better at this time"
  4. Recommendation card surfaced in UI — operator manually reviews → manual policy edit → delta config push
  5. New telemetry ingested → new time series points
  6. No closed loop: system does not know if the change helped unless operator re-checks manually

EP1:

  1. Data Lake: streaming telemetry → real-time inference detects sustained high utilization [T1]
  2. AI Core: multi-agent workflow triggered [T1]
    • Agent A — Channel Utilization Analyzer: queries Knowledge Graph for AP1 neighbors; queries Data Lake for historical channel utilization
    • Agent B — Client Impact Assessor: KG query for anonymous client tokens associated to AP1; estimates RSSI delta for affected clients
    • Agent C — Policy Conflict Checker: KG query for fixed-channel policies; checks regulatory constraints
  3. Orchestration Layer: assembles recommendation package with full reasoning chain [T1]
  4. HITL Gate: recommendation surfaced to operator with full reasoning. Operator approves, rejects, or modifies. [T1 — HITL is default]
  5. Policy Microservice: generates delta config → CAPWAP push to AP1. Event published: optimization_event.
  6. Outcome tracking [T1 — Metadata Management]: new telemetry linked to optimization_event. Knowledge Graph: before/after nodes.
  7. Feedback loop [T1 — Auto Compliance + RAG-as-a-Service]: 30 min post-change outcome metric written. RAG corpus updated.

Assistant — Re-Analysis with Source Labels

The white paper was re-read in full (all 13 pages, exact page numbers confirmed). Every claim in the three-flow analysis was re-examined against the official text. The key finding:

  • The EP1 data flow path is explicitly diagrammed on p.10 — "Data Flow in the AI Core" section. This is the only official flow diagram in any document read. It shows: Extreme Device → Anonymization (RDC) → Data Hub (GDC) → AI Service → Orchestrator → IAM Service → User. LLM and AI/Data Governance Service branch into AI Service.
  • The Data Hub six components are confirmed on p.8 with exact function descriptions (Data Ingestion Service, Data Lake and Processing, Knowledge Management graph, Metadata Management, RAG as a Service, Auto Compliance).
  • The AI Core three-component structure is confirmed on p.7 (Orchestration + Services + Data Hub).
  • Specific technology names (Kafka, InfluxDB, PostgreSQL, Redis, Cassandra, Apache Iceberg, Neo4j, gRPC) are HYPOTHESIS in every case — not named in any official document.
  • The multi-agent optimization workflow (3 specific agents: Channel Analyzer, Client Impact, Policy Conflict) is HYPOTHESIS — a plausible decomposition of the p.7 Services layer capability, but not described in any official document.
  • XIQ's internal telemetry database architecture is entirely HYPOTHESIS — no official EN document describes it in this corpus.

The flowchart section in the deployment guide (ep1-deployment-guide.html, section "XIQ vs EP1 — E2E Architecture: Three Flows") was then built with explicit solid-border (confirmed) and dashed-border (hypothesis) nodes, and a 7-question engineering verification checklist.

9. Open Questions for R&D Verification

These are the seven questions that the analysis cannot answer from publicly available documentation. They require input from engineers with direct knowledge of the EP1 and XIQ implementations.

  1. What database technologies does the XIQ/EP1 base platform use for telemetry (time-series) and config (relational)? The p.3 diagram shows "DB" per region (Primary + Local Replica + In-memory Cache) but does not name the technology. Are these siloed per function, or is there a unified store?
  2. Does the EP1 Data Ingestion Service ingest network configuration change events (policy pushes), or is the Data Hub limited to telemetry and observability data? The white paper describes the capability ("structured and unstructured data from diverse sources including databases, APIs, and file systems" — p.8) but does not enumerate what event types are ingested.
  3. Does anonymization in the RDC apply to data stored in VIQ, or only to the copy forwarded to the GDC Data Hub? The PII appendix (p.11) shows DevOps/Engineering access to raw MAC, IP, hostname — implying VIQ retains raw data. But the anonymization boundary with the AI Core path is not explicitly drawn in the white paper.
  4. Is the AP optimization path implemented as an actual multi-agent workflow, or as a simpler single-model ML pipeline with threshold rules? The p.7 Services layer says "agentic system for complex workflows, multi-agent interaction, and proactive task management" — but whether this maps to the network optimization use case specifically is unconfirmed.
  5. Are AP ↔ AP neighbor relationships explicitly modeled in the Knowledge Graph schema? The KG "establishes and maintains complex relationships between data entities" (p.8) including "technical" sources — neighbor AP relationships are plausible but not confirmed.
  6. What is the coupling between the AI Core decision and the CAPWAP push — synchronous API call, async event (message bus), or operator-triggered? No official document describes this handoff mechanism.
  7. Does the feedback loop (p.9) apply to network optimization outcomes specifically, or only to LLM accuracy and hallucination correction? The p.9 statement is in the context of AI model reliability, not necessarily the network optimization closed loop.
