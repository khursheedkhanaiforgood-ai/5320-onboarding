Troubleshoot

VLAN_70 — 169.254 APIPA → Root Cause → Fix

  • Symptom: iPhone on VLAN_70 SSID gets 169.254.x.x (APIPA — no DHCP response)
    VLAN_40 SSID working correctly on same AP policy
  • Switch verified clean: port 3 tagged VLAN_70, DHCP enabled port 3, ipforwarding ON, ping 8.8.8.8 OK
    show vlan · show dhcp-server · show ports 3 information detail — all correct
  • FDB diagnosis: show fdb ports 3 showed AP MAC only on Default VLAN — no VLAN_70 entry
    AP connected on port 3 confirmed, but not sending VLAN_70 tagged frames
  • Root cause identified: Both APs shared policy AP_X6680_Apr20 broadcasting both SSIDs. iPhone associated with AP on SW1 (stronger signal). VLAN_70 tagged frames went to SW1 which has no DHCP for VLAN_70.
    XIQ policy correct · SSID VLAN mapping correct · wrong AP selected by phone
  • Fix: Split AP policies — AP_VLAN_40 assigned to SW1 AP only · AP_VLAN_70 assigned to SW2 AP only
    Eliminates roaming ambiguity — each AP broadcasts only the SSID for its switch's VLAN
  • Result: Both SSIDs verified ✅ iPhone gets 10.40.0.x on VLAN_40 SSID · 10.70.0.x on VLAN_70 SSID · internet confirmed on both
Lessons

Key Lessons — April 20, 2026

⚠ Split AP policies when each switch has a different VLAN
If two APs share one policy and both broadcast the same SSID, the phone picks the AP with the strongest signal — not the one on the correct switch. DHCP requests land on the wrong switch and fail. One AP policy per switch eliminates this.
⚠ EXOS config filenames — no dots allowed
save configuration sw1_vlan40_apr20 — underscores only. EXOS auto-appends .cfg. Using a dot gives: %% Name cannot contain . character.
⚠ SCP runs on Mac — not on the switch
scp admin@192.168.0.28:/usr/local/cfg/sw1_vlan40_apr20.cfg ~/Downloads/ — run this in a Mac terminal tab, not the screen console session.
⚠ Factory reset changes SSH host key — clear before SCP/SSH
ssh-keygen -R 192.168.0.28 after every wipe. Otherwise scp fails with REMOTE HOST IDENTIFICATION HAS CHANGED.
Diagnose VLAN tagging issues with show fdb ports <n>
If DHCP fails on a tagged VLAN, run show fdb ports 3. If the AP MAC only appears under Default (0001) and not under the expected VLAN, the AP is not sending tagged frames — investigate AP policy, not the switch.
screen session detached — reattach not restart
screen -ls to see detached sessions · screen -r <id> to reattach. The detached session holds the serial port — opening a new screen session will fail with "resource busy".
